# Impact Analysis — IS461. QuizGameImpl IndexOutOfBoundsException on empty quiz list

## 1. Анализ каждого варианта

### Вариант A: Guard в `nextStep()`

**Side effects:** минимальные. Изменяется только логика перехода `Pending → Started(0)`. Для непустого `quizList` поведение идентично текущему. Для пустого — `hasNextQuestion()` вернёт `false`, `DatasourceEffectHandler` пойдёт в ветку `SessionOver`.

**Контракты:** не нарушает. `QuizGame` interface не меняется. Внешний контракт: `hasNextQuestion() = false` → `nextQuestion()` не вызывается. Это уже соблюдается вызывающим кодом (`DatasourceEffectHandler`).

**Тесты:** существующие тесты не затронуты (они работают с непустым `quizList`). Нужен 1 тест: `hasNextQuestion returns false when quizList is empty`.

**Качество кода:** улучшает — устраняет очевидную ошибку. Код остаётся компактным.

**Сложность:** низкая. 1 строка.

**Риск регрессии:** низкий. Условие `quizList.isNotEmpty()` ортогонально к остальной логике.

### Вариант B: Guard + bounds check в `getQuiz()`

**Side effects:** те же что у A. Bounds check в `getQuiz()` — чисто защитный, не меняет нормальный путь выполнения.

**Контракты:** не нарушает. `getQuiz()` — private метод, вызывается только внутри `QuizGameImpl`.

**Тесты:** как у A + опционально тест на `getQuiz()` с невалидным индексом (но это тестирование private метода — спорно).

**Качество кода:** улучшает. Более информативный exception message при будущих багах. `getOrElse` с контекстным сообщением — стандартная практика defensive programming.

**Сложность:** низкая. 2 изменения, обе тривиальны.

**Риск регрессии:** низкий. `getOrElse` с throw ведёт себя идентично прямому доступу по индексу, только с лучшим сообщением.

### Вариант C: Guard + bounds check + логирование

**Side effects:** как у B + одно обращение к `logger.w` при пустом списке. Logger уже используется в `addQuizData()` — никаких новых зависимостей.

**Контракты:** не нарушает.

**Тесты:** как у B + опционально тест что лог вызывается при пустом списке (mock logger).

**Качество кода:** улучшает, но лог в `loadData()` — граничный случай. Если пустой словарь — нормальная ситуация (пользователь только создал словарь), warning-лог может быть шумным. С другой стороны, это единственный способ узнать о частоте ситуации в продакшне.

**Сложность:** низкая. 3 изменения, все тривиальны.

**Риск регрессии:** низкий. Логирование не влияет на поведение.

## 2. Таблица сравнения

| Критерий | A | B | C |
|---|:---:|:---:|:---:|
| Side effects | нет | нет | нет |
| Нарушает контракты | нет | нет | нет |
| Тесты падают | 0 | 0 | 0 |
| Качество кода | улучшает | улучшает+ | улучшает+ |
| Сложность | низкая | низкая | низкая |
| Риск регрессии | низкий | низкий | низкий |

## 3. Рекомендация

**Рекомендуется: вариант C**

Обоснование: все три варианта имеют одинаковый уровень риска и сложности — разница между ними минимальна. Вариант C даёт максимум: корневой фикс (`nextStep`), страховку (`getQuiz` bounds check с контекстным сообщением), наблюдаемость (warning лог при пустом списке). Для production-бага наблюдаемость особенно важна — позволит понять как часто пользователи попадают в эту ситуацию и нужен ли специальный UI для "нет слов для квиза".

Главный риск: шумность warning-лога если пустой словарь — частый сценарий. Митигация: `logger.w` (не `logger.e`) — не триггерит алерты, но виден при ручном анализе логов.
