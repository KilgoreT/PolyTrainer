# Impact Analysis: IS463 — Room @Relation NULL

## 1. Анализ каждого варианта

### Вариант A: Nullable @Relation + фильтрация

**Side effects:**
- Тип поля `lexemeDbWithWordDbRelation` меняется с NON-NULL на nullable. Все call sites, обращающиеся к этому полю, требуют обновления.
- `toApiEntity()` перестаёт компилироваться — нужен null-check или safe call. Компилятор сам покажет все сломанные места.
- Количество quiz в сессии может оказаться меньше запрошенного `limit` (отфильтрованные orphaned записи). Пользователь просто увидит меньше вопросов — некритично.

**Контракты:**
- Контракт `CoreDbApi.QuizApi.getRandomWriteQuizList()` не меняется — возвращает `List<WriteQuizComplexEntity>`. Фильтрация внутри `CoreDbApiImpl`.
- Контракт entity (`WriteQuizDbEntity`) меняется — но он internal для `core-db-impl`, не экспортируется.

> 📎 guide: docs/guides/data-layer.md — "Три слоя маппинга: DB → API → Domain. Каждый через extension-функцию"

**Тесты:**
- Существующих unit-тестов на entity маппинг нет (проверено — тесты на quiz-layer в `QuizTabReducerTest`, не на DAO).
- Нужен тест на фильтрацию orphaned записей — но это instrumented test (Room), не unit test.

**Качество кода:**
- Ухудшает: nullable поля без семантической причины — это defensive coding, не clean architecture.
- Маскирует проблему данных вместо её решения.

**Сложность:** Низкая. 3 файла, минимальные изменения.

**Риск регрессии:** Низкий. Компилятор поймает все null-safety проблемы на этапе сборки.

---

### Вариант B: Миграция 11→12 с cleanup

**Side effects:**
- Удаляет orphaned `write_quiz` записи — пользователь теряет quiz-прогресс для невалидных записей (score, grade, error_count). Это допустимо: данные и так нефункциональны (крашат приложение).
- Удаляет orphaned `lexemes` — аналогично, нефункциональные данные.
- Версия БД повышается до 12 — влияет на все последующие миграции.

**Контракты:**
- Никакие контракты не меняются. Данные просто становятся консистентными.

**Тесты:**
- Нужен `MigrationFrom10to11.kt` (его нет — пропущен при создании миграции 10→11).
- Нужен `MigrationFrom11to12.kt` для нового cleanup.
- Нужно обновить `AllMigrationTest`.
- Нужны Schemable: `WriteQuizV11`, `WordV11`, `DictionaryV11` (для тестовых данных v11).

> 📎 guide: docs/guides/testing-migrations.md — "Один тест-класс на миграцию. Schemable на каждую версию таблицы. AllMigrationTest обязателен"

**Качество кода:**
- Улучшает: БД становится консистентной. Принцип "правильные данные — правильный код".
- Не добавляет defensive null-checks в entity.

**Сложность:** Средняя. Миграция проста (2 DELETE), но тестирование требует создания нескольких Schemable и тест-классов.

**Риск регрессии:** Низкий для самой миграции (только DELETE orphaned). Средний для тестов — нужно корректно описать схему v11.

---

### Вариант C: Nullable entity + миграция cleanup

**Side effects:**
- Комбинация A + B. Те же side effects, что у обоих.
- Два слоя защиты — redundancy в production, но защита от будущих edge cases.

**Контракты:**
- Те же что у A (entity nullable) и B (версия БД → 12). Контракт `CoreDbApi.QuizApi` не меняется.

**Тесты:**
- Всё от B (миграция + Schemable) + обновление маппинга из A.

**Качество кода:**
- Неоднозначно: миграция чистит данные (хорошо), nullable entity — defensive coding (спорно).
- Defense in depth — аргумент в пользу, если есть опасение что orphaned данные могут появиться из нового источника.

**Сложность:** Средняя (объём A + B, но не O(A*B) — работы аддитивны).

**Риск регрессии:** Низкий. Те же гарантии компилятора что у A, плюс миграция тестируется отдельно.

---

### Вариант D: Raw SQL с JOIN

**Side effects:**
- Полная замена data layer для quiz. Все три метода DAO переписываются.
- `LexemeDbEntity` содержит вложенный `@Relation` на `SampleDb` (samples). При flat JOIN потребуется дополнительный запрос или GROUP_CONCAT для samples — значительное усложнение.
- Потеря Room @Relation magic — ручное управление маппингом.

**Контракты:**
- Контракт `CoreDbApi.QuizApi` не меняется (возвращаемый тип тот же).
- Внутренний контракт DAO полностью меняется.

**Тесты:**
- Нужны тесты для нового flat entity маппинга.
- Instrumented tests для проверки JOIN-запросов.

**Качество кода:**
- В долгосрочной перспективе: лучше (explicit queries, нет N+1).
- В контексте багфикса: over-engineering, непропорциональный объём работы.

**Сложность:** Высокая. Фактически рефакторинг quiz data layer.

**Риск регрессии:** Высокий. Переписывание запросов — богатый источник регрессий, особенно с samples.

## 3. Рекомендация

**Рекомендуется: вариант A (Nullable @Relation + фильтрация)**

Обоснование: анализ экспортированных БД (DB10, DB11) показал что orphaned записей нет — обе базы консистентны. Миграция cleanup (вариант B/C) бессмысленна — чистить нечего.

Наиболее вероятная причина краша — Room баг с RANDOM()+@Relation (known issue: https://issuetracker.google.com/issues/413924560). Nullable @Relation защищает от этого бага и от любых будущих несогласованностей.

Вариант A — минимальный, надёжный, низкий риск. Без миграции, без version bump БД.
