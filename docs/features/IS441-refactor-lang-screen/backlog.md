# IS441. Бэклог фичи

- **[Краш при повторном добавлении языка].**
  `numericCode` имеет UNIQUE constraint в Room. Если пользователь выберет уже добавленный язык — SQLite constraint violation, необработанное исключение, краш.
  Строка `lang_selection_error` ("Language already added") заготовлена в ресурсах, но не подключена.
  Нужно: либо фильтровать уже добавленные языки из списка, либо показывать ошибку, либо дизейблить их визуально.

- **[Добавить AppBar с кнопкой "назад"].**
  Сейчас экран без topBar — нет кнопки навигации назад. Это ок для первого запуска (onboarding, деваться некуда), но при вызове из настроек или dropdown словаря пользователь может вернуться только системным жестом.
  Нужно: добавить AppBar с иконкой "назад". Показывать его во всех случаях кроме первого запуска. Для этого передавать флаг (например, `showBackNavigation: Boolean`) при навигации на экран.

- **[Изучить библиотеку флагов (blongho/country-data)].**
  `FlagProvider` использует `World.getFlagOf(numericCode)` — numericCode берётся из этой библиотеки (ISO 3166-1 numeric).
  Библиотека нужна и останется, но надо согласовать: что именно хранить в БД?
  Сейчас в `languages.name` лежит локализованная строка (зависит от локали устройства) — это рудимент.
  Нужно: определить схему хранения, которая будет работать с кастомными словарями (см. следующий пункт).

- **[Концепция: словари вместо языков].**
  Сейчас экран называется "Создание словаря", но по факту это выбор языка из 6 захардкоженных.
  В будущем "язык" → "словарь". Пользователь сможет создать кастомный словарь:
  - Вбить произвольное название (например, "Биология на английском")
  - Опционально добавить комментарий/описание
  - Выбрать флаг из библиотеки (не привязан к языку — можно испанский язык с мексиканским флагом)
  - Фильтрация флагов по названию страны
  Это отдельная фича на будущее, но архитектура БД и стейта должна закладывать эту возможность уже сейчас.
  Нужно: при рефакторинге IS441 учитывать, что `languages` таблица — это по сути `dictionaries`, а `numericCode` — только один из способов привязки флага, не единственный идентификатор.

## Миграция базы

- **[`getFromDatabase()` — копипаста на каждый Schemable].**
  WriteQuizV5, WriteQuizV10, LexemeV9, LexemeV10 — все вручную парсят cursor. 50-80 строк boilerplate на версию.
  Забытая колонка или ошибка в индексе — тест тихо пропустит данные.
  Нужно: generic cursor→map парсер или хотя бы вынести общую логику.

- **[`checkMatcher` — нет exhaustiveness].**
  Лямбда не заставляет проверять все поля entity. Добавляется новое поле — компилятор молчит.
  В MigrationFrom09to10 `addDate` и `changeDate` закомментированы — уже были проблемы и забили.
  Нужно: рассмотреть `assertEquals` на весь entity вместо поэлементного сравнения, или генерировать matcher из data class.

- **[Schemable.data() использует актуальные entity].**
  Для тестирования версии N-1 используется entity текущей версии (WriteQuizDb, LexemeDb).
  При переименовании полей (langId → dictionaryId) **сломаются все старые Schemable**.
  Отдельные data class'ы на версию есть (WriteQuizDbV5), но не для всех.
  Нужно: каждая версия Schemable должна иметь свой data class, не зависящий от актуального entity.

- **[Дублирование интерфейсов TableName, ColumnId, ContentValue, FromDatabase].**
  Определены и в `Schemable.kt` и в `Schema.kt`. Два набора одинаковых интерфейсов.
  Нужно: убрать дубли, оставить только в `Schemable.kt`.

- **[Schema.kt — god object, 400+ строк].**
  Версионированные схемы (Languages, LanguagesV1, Definition, LexemeV1, Sample, HintV1) в одном файле.
  Часть схем вынесена в `schemable/` (Word, Lexeme, WriteQuiz), часть нет. Непоследовательно.
  Нужно: вынести все версионированные схемы в `schemable/`, Schema.kt оставить пустым или удалить.

- **[afterCreateCheck дублирует afterMigrationCheck].**
  Обе фазы делают hasTable + hasColumns + checkData. Разница только в версии Schemable.
  Нужно: извлечь helper `verifySchemable(db, schemable, origin, checkMatcher)`.

- **[Нет теста на удаление данных].**
  Тесты проверяют что данные сохранились, но не проверяют удаление (если миграция дропает колонку или таблицу).
  Нужно: добавить негативные проверки — дропнутая колонка не существует, удалённые данные отсутствуют.

- **[Date — хак с погрешностью 1000ms].**
  `isEqualTo()` сравнивает даты с точностью до секунды. В `checkMatcher` даты вообще закомментированы.
  Корневая причина: Room и ContentValues по-разному обрабатывают timestamp.
  Нужно: разобраться с точностью хранения дат, починить, раскомментировать проверки.

## Исследования

- **[Исследовать API blongho/country-data].**
  Сейчас используется только `World.getFlagOf(numericCode)`. Нужно выяснить, что ещё даёт библиотека:
  - Список стран по языку (например, все испаноязычные страны → их флаги)
  - Названия стран/языков
  - Связь язык → страны
  Это нужно для будущего UX кастомного словаря: пользователь выбирает язык (например, "Испанский") → библиотека возвращает список стран с этим языком (Испания, Мексика, Аргентина, Уругвай...) → пользователь выбирает флаг из этого подмножества.

- **[UX выбора флага для кастомного словаря].**
  Когда появятся пользовательские названия словарей, нужна цепочка:
  1. Пользователь вводит название ("Биология на испанском")
  2. Выбирает язык из списка (фильтр для флагов)
  3. По выбранному языку показываются флаги стран, где говорят на этом языке
  4. Пользователь выбирает флаг
  Нужно: проверить, поддерживает ли blongho связку язык → список стран. Если нет — искать альтернативу или захардкодить маппинг.
