## F004 [analyst] critical — защита от двойного нажатия FAB / промежуточное состояние пока CreateLexeme в полёте

**Description:** Инвариант 3 («в lexemeList только лексемы с id из БД») противоречит сценарию ошибки CreateLexeme в open question 3 — не описано промежуточное состояние FAB между нажатием и возвратом RefreshLexeme (loading/disable FAB, защита от двойного нажатия).

**Verdict (инквизитор):** State не описывает защиту от двойного нажатия FAB / промежуточного состояния пока `CreateLexeme` в полёте — это область State (нужен флаг/loading), валидный проёб контракта.

## F005 [analyst] critical — empty state lexemeList не описан

**Description:** Не описано состояние пустого lexemeList — UI-сценарий empty state после загрузки слова без лексем не покрыт ни полем, ни инвариантом.

**Verdict (инквизитор):** Computed `hasAnyLexeme` намеренно не введён, но описание пустого состояния как явного семантического случая в контракте State отсутствует — поле/инвариант или хотя бы фиксация что `lexemeList.isEmpty()` = empty state нужна.

## F001 [architect] minor — инвариант 4 тривиально истинен

**Description:** Инвариант 4 написан как `translation?.isEdit == false || edited не пуст` — по тексту описывает противоположное; формула должна быть `isEdit == true || edited не пуст` либо переформулирована.

**Verdict (инквизитор):** Инвариант 4 действительно логически тривиален в текущей формулировке — нужно переформулировать.

## F003 [architect] minor — TextValueState.isEdit дефолт = true не обоснован

**Description:** `TextValueState.isEdit` по дефолту `= true`, тогда как семантика создания (создан, но ещё не редактируется) не различима; нужно зафиксировать why-дефолт или сделать `false`.

**Verdict (инквизитор):** Дефолт `isEdit = true` в `TextValueState` действительно не задокументирован why; комбинация «создаётся при нажатии chip сразу в режиме edit» требует явного обоснования в контракте.

## F007 [analyst] minor — инвариант 4 неоднозначен (или инвариант, или политика)

**Description:** Инвариант 4 сформулирован неоднозначно («мягкий, не блокирующий») — либо инвариант, либо политика; в текущем виде не инвариант.

**Verdict (инквизитор):** «Мягкий, не блокирующий инвариант» — оксюморон; либо это инвариант (жёсткое утверждение), либо политика, в текущем виде это не инвариант.

## F009 [analyst] minor — порядок lexemeList не уточнён для удаления и configuration change

**Description:** Инвариант про порядок lexemeList не уточняет поведение после удаления лексемы и configuration change.

**Verdict (инквизитор):** Инвариант 5 говорит о порядке но не уточняет инвариант после удаления и configuration change — границы инварианта размыты.

## F010 [analyst] minor — комбинация null + isEdit=true противоречит whitelist

**Description:** TextValueState.isEdit дефолт true, но translation/definition nullable со значением null — комбинация «поле существует и сразу в режиме edit» противоречит whitelist-логике «создаётся только при нажатии chip».

**Verdict (инквизитор):** Действительно комбинация «translation: null + дефолт isEdit=true при создании» создаёт семантическое противоречие.
