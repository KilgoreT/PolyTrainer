# Правки для промпта шага `ui_layout`

Накопитель правок к `docs/forgeflow-overlay/steps/ui_layout.md` — что добавить в промпт чтобы агент при заполнении формата `ui_layout_example.md` корректно отражал реальность и Figma.

Каждая правка относится к одному нюансу. Не сливать в общий "make it better" — каждая — отдельный пункт, обсуждаемый изолированно.

---

## Правка #1 — маркеры и отход от Figma в `notes:`

**Нюанс:** маркер 📌 ("кастомный, без изменений в этой фиче") **не означает** "соответствует Figma". Это две независимые оси:
- маркер ⚙️/❇️/🔄/📌 = состояние в текущей фиче;
- ⚠ в `notes:` = отход от Figma.

Без явного требования в промпте агент ставит 📌 по принципу "в фиче не трогаем" и забывает отразить расхождение с Figma (пример: TopBar — в Figma component `Variant=with btn`, у нас ⋮ DropdownMenu; маркер 📌 корректен, но без ⚠ в notes Figma-долг скрыт).

**Что добавить в промпт шага:**

1. **Обязательное правило:** для каждого виджета с маркером ⚙️/🔄/📌 **проверить** соответствие реализации (или планируемой реализации) Figma-источнику. Если есть структурное / типовое / spec-расхождение — обязательно ⚠ в `notes:` с явной формулировкой "Figma X → реализация Y, причина: Z".

2. **Чек-лист расхождений** (что подсвечивать):
   - **kind:** Figma component (например, `tui-input`, `tui-tag`, `tui-button`) → используемый M3 / project widget.
   - **structure:** sibling vs nested, FlowRow vs Column-place в дереве.
   - **numeric tokens:** gap / padding / border radius — если значение отличается от Figma.
   - **variant:** Figma component variant (`Variant=with btn` и т.п.) → выбранный variant или замена.
   - **typography:** style token из Figma → LexemeStyle.* в коде.

3. **Формат `type:`-буллета:** если Figma-источник и реальный код расходятся по типу — писать пару:
   ```
   • type: tui-input (Figma componentId=4620:57849) → LexemeEditableText (project inline-edit pattern)
   ```
   Это правило применяется к любому буллету где есть расхождение (`type`, `padding`, `colors`, `typography` и т.д.).

4. **Маркер 📌 ставить только если:** виджет не трогаем в фиче **И** при этом нет ⚠ в notes. Если ⚠ есть — формат остаётся 📌 (по делу фичи), но **запрещено** трактовать это как "по Figma".

5. **Дисциплина:** агент должен пройти по всем виджетам с 📌 в конце шага и **явно подтвердить** что для каждого либо нет расхождений с Figma, либо ⚠ в notes стоит.

**Куда применить:** `docs/forgeflow-overlay/steps/ui_layout.md` — в секцию про заполнение `## 🔍 Анализ виджетов` (или эквивалент в overlay-промпте).

## Правка #2 — миграция логики удалённых виджетов

**Нюанс:** секция `## ❌ УДАЛЯЕМ` сейчас только перечисляет имена удалённых виджетов, но **не говорит куда переехала их логика** (callbacks, state-обновления, business-rules). При review нельзя проверить "ничего не потеряли".

**Что добавить в промпт шага:**

1. **Обязательное правило bi-directional migration trace.** Для каждого удалённого виджета — указать куда переехала логика. И зеркально: в `notes:` нового / модифицированного виджета, который **принимает** эту логику — указать откуда она пришла.

2. **Формат секции `## ❌ УДАЛЯЕМ` пересматривается:** каждый bullet расширяется до строки миграции.

   Было:
   ```
   ## ❌ УДАЛЯЕМ
   - AddLexemeBottomWidget (bottom-sheet)
   - LexemeMeaningWidget (переключатели Перевод/Definition)
   - ActionsWidget (кнопки Добавить/Отмена внутри bottom-sheet)
   ```

   Стало:
   ```
   ## ❌ УДАЛЯЕМ (с миграцией)
   - AddLexemeBottomWidget.onAdd        → SubentityChip.onActivate + Msg.Create{Translation|Definition}
   - LexemeMeaningWidget.onTypeSelect   → разделено на Msg.CreateTranslation / Msg.CreateDefinition
   - ActionsWidget (commit/cancel)      → trailing-icons LexemeValueField (commit) / out-of-field tap (cancel)
   ```

   Если логика удалена бесследно — явно `удалено, нет аналога` (это тоже важный сигнал).

3. **Зеркальная пометка в `notes:` принимающего виджета.** Любой виджет, который **получил** мигрировавшую логику, обязан в `notes:` упомянуть источник:
   ```
   • notes:
       ⚠ принимает onActivate из удалённого AddLexemeBottomWidget.onAdd (миграция IS479)
   ```

4. **Дисциплина:** агент в конце шага проверяет, что для **каждой** строки в `## ❌ УДАЛЯЕМ` есть зеркальная пометка в notes принимающего виджета (или явное `удалено, нет аналога` без зеркала).

**Эффект:** delta фичи виден полностью — что выкинуто, что мигрировало, что новое. Сейчас "куда уехало" — отсутствует.

**Куда применить:** `docs/forgeflow-overlay/steps/ui_layout.md` — в секцию про заполнение `## ❌ УДАЛЯЕМ`, дополнить требование к buller-list "Анализа виджетов" зеркальной пометкой ⚠ миграции.
