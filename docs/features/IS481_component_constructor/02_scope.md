# Scope analysis: IS481 component_constructor

## Резюме

Фича добавляет user-defined конструктор компонентов словаря (CRUD: create / rename / soft-delete) с двумя независимыми entry-point'ами: глобальный `ComponentsManagerScreen` из `SettingsTab` (aggregated view всех user-defined компонентов) и per-dictionary `PerDictionaryComponentsScreen`, открываемый по icon-button «молоток» в `DictionaryAppBar` (scoped view конкретного словаря). Параллельно — миграция M12→M13 с типизированной перестройкой модели значений (`Primitive` + `Field` + `TemplateValues` sealed вместо M12-iерархии `TextValue/LongTextValue/ImageValue`), composite JSON-rewrite `component_values.value`, новое поле `is_multi`, добавление timestamp-колонок (`created_at`, `updated_at`, `removed_at`) и переименование `remove_date → removed_at` **только в `component_types`** (см. F002 ниже), drop UNIQUE `(lexeme_id, component_type_id)` для поддержки multi-cardinality, drop UNIQUE `(dictionary_id, name)` на `component_types` для поддержки пересоздания после soft-delete (см. aspect `db_migration` и F018/F021).

F028/F034); JSON rewrite формата `component_values.value` (M12 `{"v":1,"text":"foo"}` / `{"v":1,"uri":"u"}` → M13 `{"fields":{"value":{"type":"text","value":"foo"}}}` / `{"fields":{"value":{"type":"image","uri":"u"}}}`; различение M12-формы по `template_key` родителя, не по дискриминатору в JSON); template-key consolidation `long_text → text`. Также: новые/обновлённые `ComponentTypeDao` / `ComponentValueDao` методы (DAO convention `WHERE removed_at IS NULL`, JOIN-фильтр на parent.removed_at для cascade); новый mapper `parseTemplateValues(json, template): TemplateValues?` + симметричный `toJson()` в `core-db-impl/mapper/`; обновление Room entity (`ComponentTypeDb.removeDate → removedAt` + новые поля + drop UNIQUE на `(dictionary_id, name)`, `ComponentValueDb` + новые поля); обновление `LexemeDbEntity` (стратегия фильтрации soft-deleted `component_values` в `@Relation` — см. Полная замена Storage-слоя на typed `TemplateValues` boundary (см. § Затронутые файлы → Data layer для конкретных файлов: `CoreDbApiImpl.kt`, `ComponentValueWithType.kt`, `SeedBuiltIns.kt` и др.).

## Аспекты

- `db_migration` — M12→M13: composite JSON rewrite (`fields` envelope, per-primitive discriminator), DROP UNIQUE на `component_values` (`index_component_values_lexeme_id_component_type_id`), **DROP UNIQUE на `component_types` (`index_component_types_dictionary_id_name`)** — см. F028/F034 (Open question F018/F021 best-guess B перенос uniqueness в UseCase), add `is_multi` с CHECK, rename `remove_date → removed_at` (scope = только `component_types`, см. F002), add timestamps на обеих таблицах, **template-key consolidation `UPDATE component_types SET template_key='text' WHERE template_key='long_text'`** (F046). См. `template_model.md` § Миграция M12→M13.
  - **Реальные M12-формы** (verified в `core-db-impl/mapper/ComponentValueDataJson.kt`): `{"v":1,"text":"..."}` для `text` / `long_text` template, `{"v":1,"uri":"..."}` для `image` template. Дискриминатора `type` в M12 нет — различение по `template_key` родителя.
  - **M13-формы**: `{"fields":{"value":{"type":"text","value":"..."}}}` (для `text` template; `long_text` сливается с `text` — упразднён), `{"fields":{"value":{"type":"image","uri":"..."}}}` (для `image` template).
  - **Strategy для `seedBuiltIns` schema mismatch на upgrade-path** (F044): юзер с `user_version=11` → при первом запуске v13 пройдёт M11→M12 → вызовет `seedBuiltIns(connection)` (`Migration_011_to_012.kt:58`) → если `SeedBuiltIns.kt` обновлён под M13-схему, runtime fail «no such column: is_multi / removed_at». Best-guess: **(A) frozen-копия `seedBuiltIns_v12()` в `Migration_011_to_012.kt`** — заморозить старый SQL под M12-схему; current `SeedBuiltIns.kt` обновляется под M13 для fresh install / M12→M13 path. Альтернативы: (B) schema-aware `seedBuiltIns(connection, schemaVersion)` параметризованный по версии; (C) перенести `seedBuiltIns` вызов в M12→M13 (ломает fresh install path). Финализация — на `data_design_tree`.
- `migration_edge_cases` — стратегия обработки edge cases при JSON rewrite на M12→M13 (см. F008):
  - **Malformed JSON в `value` row** (не парсится через `JSONObject`): log в Crashlytics + skip rewrite (row остаётся с M12-значением). Типизированный парсер на чтении (`parseTemplateValues`) вернёт `null` и тоже залогирует — fail-soft на уровне UI. Приоритет: «лучше data inconsistency чем падение миграции».
  - **Unknown / unsupported `template_key` родителя** (например legacy template не из {text, long_text, image}): fallback на `text` discriminator при rewrite (наследует M12 `ComponentTemplate.fromKey` fallback). После M13 — `fromKey` становится nullable; парсер на чтении вернёт `null` для unknown key.
  - **Null / empty `value`**: skip rewrite, оставить как есть (`NULL` остаётся `NULL`).
  - **Идемпотентность миграции**: перед rewrite проверять — если JSON уже содержит ключ `fields` (M13-формат), skip (не rewrite'ить повторно). Защита для `MigrationTestHelper` runForward от того же `failAfterStep` хука.
- `migration_timestamps_backfill` — стратегия backfill `created_at` / `updated_at` для existing rows на M13 (см. F009):
  - Для existing rows в `component_types` и `component_values`:
    - `created_at = <время миграции>` (момент применения M13), через `UPDATE ... SET created_at = strftime('%s','now') * 1000` в migration SQL. Альтернатива «`created_at = removed_at`» отклонена — даёт неверную семантику (created != removed).
    - `updated_at = <время миграции>` (тот же подход).
    - `removed_at` существующих rows = `NULL` (активные) либо migrate value `remove_date → removed_at` (в `component_types`, через `ALTER TABLE RENAME COLUMN` + сохранение значений).
  - Технически: либо temp DEFAULT-значение с последующим `ALTER ... DROP DEFAULT` (Room), либо двухшаговая миграция (ADD COLUMN nullable → UPDATE backfill → ADD CHECK NOT NULL).
- `public_contract_change` — `CoreDbApi.LexemeApi` (или новый `ComponentApi` — см. Open questions):
  - **Добавление новых методов:** `createUserDefinedComponent`, `renameComponent`, `softDeleteComponent`, `previewDeletionImpact`, `flowUserDefinedTypes(...)`, агрегация `componentsAcrossAllDictionaries`.
  - **Изменение сигнатур существующих методов (BREAKING):** упразднение `ComponentValueData` меняет сигнатуры 5 существующих публичных методов `CoreDbApi.LexemeApi` (F029): `addLexemeWithBuiltInComponent`, `addLexemeWithUserDefinedComponent`, `addLexemeWithComponents`, `addComponentValue`, `updateComponentValue` — все принимают `ComponentValueData`, после M13 принимают `TemplateValues`. Это breaking change для всех callsite'ов (wordcard, quiz/chat, app/mapper, тесты). `addLexemeWithTranslation(wordId, dictionaryId, translation: TranslationApiEntity)` — НЕ в списке (принимает `TranslationApiEntity`, shim сам по себе в M13 не меняется, verified `CoreDbApi.kt`) — см. F036.
  - Domain `ComponentType` — переименование `removeDate → removedAt` + новое поле `isMulti: Boolean`.
  - Domain — новый sealed `TemplateValues` + конкретные `*Values` (`TextValues` для MVP) + новые типы `Primitive`, `Field`, `PrimitiveType`. Упразднение `ComponentValueData` sealed (M12 legacy).
  - **Каскад `ComponentTypeApiEntity.template` nullable** (F019): переход `ComponentTemplate.fromKey: (String) → ComponentTemplate?` каскадно меняет публичный контракт `ComponentTypeApiEntity.template` (сейчас non-nullable). Стратегия — см. Open question ниже.
  - **`previewDeletionImpact` сигнатура** (F024 + F033): `previewDeletionImpact(typeId): DeletionImpact` где `DeletionImpact = (valueCount: Int, dictionariesWithValues: List<String>, quizConfigsAffected: List<Pair<DictionaryId, QuizMode>>, affectedPrefs: List<DictionaryId>)`. `affectedPrefs` — список словарей, у которых `quiz_picker_dict_<id>` pref ссылается на удаляемый ref (см. aspect `prefs_cleanup_on_soft_delete`). Реальная сигнатура — на `business_contract` шаге.
- `new_dependency` — НЕТ. Используется existing `org.json` (Android stdlib) для JSON парсинга (как в M12-mapper).
- `domain_rewrite` — `ComponentTemplate` (drop `LONG_TEXT`, добавление `fields: List<Field>` schema), `ComponentValueData` (упразднение) → `TemplateValues` sealed. Breaking change для всех callsite (wordcard, quiz/chat, app/mapper). Compile-time gate через exhaustive `when`.
- `dao_convention` — все active-queries для `component_types` + `component_values` обязаны иметь `WHERE removed_at IS NULL` либо JOIN на `parent.removed_at IS NULL`. Audit existing DAO methods (`flowTypesForDictionary`, `getTypesForDictionary`, `getBuiltInTypes`, `getBySystemKey` в `ComponentTypeDao`; все методы `ComponentValueDao` — сейчас НЕ фильтруют `removed_at`).
  - **`LexemeDbEntity.componentValueListDb` `@Relation`** (F031): Room `@Relation(parentColumn = "id", entityColumn = "lexeme_id")` не поддерживает WHERE-условия — после M13 в read'е подтягиваются soft-deleted `component_values`. Best-guess стратегия: **(A) post-load filter в `toApiEntity()`** (отфильтровать `removedAt != null` после загрузки) — минимум изменений, единственный callsite mapping. Альтернатива: (B) custom `@Query` с явным JOIN и WHERE вместо `@Relation` — переписать чтение лексемы. Финальное решение — на `data_design_tree`.
- `cardinality_safety` — `insertSingleSafe @Transaction` DAO-метод для `is_multi=false`: «SELECT count активных values (`WHERE component_type_id = ? AND lexeme_id = ? AND removed_at IS NULL`) для этого component_type у этой лексемы + reject (через abort / IllegalStateException) если count > 0 + INSERT нового value» — всё в одной транзакции (F035). Без фильтра `removed_at IS NULL` soft-deleted value на `is_multi=false` будет молча блокировать пересоздание active value для той же лексемы. Сшивается с aspect `dao_convention` (общая convention на active queries). См. `template_model.md` § БД «Защита cardinality на INSERT».
- `multi_to_single_downgrade` — UseCase-уровневая проверка `MAX(count) per lexeme_id` перед изменением `is_multi: true → false`. См. `template_model.md` § Open Q9.
- `soft_delete_unique_collision` — `ComponentTypeDb` имеет `Index(value = ["dictionary_id", "name"], unique = true)`. После soft-delete юзер пробует пересоздать компонент с тем же именем → INSERT падает (soft-deleted row занимает слот). Стратегия — UseCase enforcement uniqueness среди active rows. **SELECT обязан разветвляться по scope** (F032):
  - **per-dict component** (`dictionaryId != null`): `SELECT ... WHERE dictionary_id = ? AND name = ? AND removed_at IS NULL`.
  - **global component** (`dictionaryId == null`, scope = «На все словари» per ui_placement.md): `SELECT ... WHERE dictionary_id IS NULL AND name = ? AND removed_at IS NULL`. SQL `dictionary_id = NULL` всегда возвращает UNKNOWN → без `IS NULL` ветки два global компонента с одинаковым именем создаются молча.
  - UseCase разветвляется по scope (две query-ветки либо один параметризованный @Query с `(dictionary_id = :dictId OR (:dictId IS NULL AND dictionary_id IS NULL))`).
  - Альтернативы (partial unique index, blacklist, auto-rename, «user-defined всегда per-dict») — см. Open question ниже.
- `userdefined_identity_invariant` (F039) — cross-scope name collision invariant. После DROP UNIQUE `(dictionary_id, name)` сами по себе per-dict и global SELECT-ветки (aspect `soft_delete_unique_collision`) могут пройти **одновременно**: active global "Foo" + active per-dict "Foo" в одном словаре. `ComponentTypeRef.UserDefined("Foo")` идентифицируется только по `name` (см. `QuizGameImpl.matchesRef`, F027 cascade `quiz_configs.component_refs`, F033 prefs cleanup) → identity-ambiguity ломает matching, rename cascade и pref cleanup.
  - **Инвариант:** уникальность имени user-defined компонента enforce'ится **глобально + per-dictionary одновременно**:
    - при создании per-dict `"X"` (`dictionary_id = D`) — не должно существовать active global `"X"` (`dictionary_id IS NULL AND removed_at IS NULL`);
    - при создании global `"X"` — не должно существовать active per-dict `"X"` ни в одном словаре (`dictionary_id IS NOT NULL AND name = ? AND removed_at IS NULL`).
  - **Реализация:** UseCase делает **two-prong SELECT** перед INSERT — одна ветка для same-scope коллизии (per aspect `soft_delete_unique_collision`), вторая ветка для cross-scope (этот aspect). Оба должны вернуть пусто.
  - Альтернатива — обогатить `ComponentTypeRef.UserDefined` чтобы он содержал `typeId: ComponentTypeId` (а не только `name`) для устранения identity-ambiguity. Это **breaking change** в domain ref + миграция данных в `quiz_configs.component_refs` (rewrite JSON `user:<name>` → `user:<typeId>`) + миграция prefs (`QuizPickerPrefKey`). Best-guess: two-prong SELECT (минимум изменений; domain ref остаётся stable). Финализация — на `business_design_tree` / `data_design_tree`. См. Open question ниже.
- `quiz_configs_cleanup` — атомарный cleanup при soft-delete `component_type`. **Два эффекта** (F027 + F033):
  - **(1) `quiz_configs.component_refs` cleanup при soft-delete:** удалить ref из `quiz_configs.component_refs` (через `@Transaction` DAO либо UseCase: soft-delete `component_type` + UPDATE `quiz_configs` с `json_remove` либо собрать новый JSON в Kotlin). Без этого квиз будет пытаться загрузить удалённый компонент. Recovery не восстанавливает ref автоматически (см. `deletion_concept.md`).
  - **(2) `quiz_configs.component_refs` cascade при rename** (F027): UserDefined компоненты идентифицируются по `name` в `ComponentTypeRef.UserDefined` + `QuizGameImpl.matchesRef`. При `renameComponent(typeId, newName)` нужен cascade UPDATE `quiz_configs.component_refs` (rename `old_name → new_name` через `json_replace` либо собрать новый JSON). Атомарно в одной транзакции с UPDATE `component_types.name`. Без cascade — silent quiz breakage (config ссылается на старое имя, компонент уже переименован).
  - Реальная реализация атомарности (single transaction vs UseCase-level coordinate) — на `data_design_tree` / `business_contract`.
- `prefs_cleanup_on_soft_delete` — cleanup `QuizPickerPrefKey` при soft-delete `component_type` (F033). `QuizPickerPrefKey` хранит `ComponentTypeRef` per-dictionary (per-dict выбор компонента квиза из IS481_quiz_component_picker). При soft-delete component_type — pref остаётся stale (ссылается на удалённый ref); следующая quiz session попытается восстановить выбор по удалённому ref. Стратегия:
  - При soft-delete component_type — найти все `quiz_picker_dict_<id>` где value соответствует удаляемому типу (`builtin:<key>` для built-in либо `user:<name>` для user-defined) → reset (delete pref entry).
  - Атомарно с soft-delete либо отдельный effect (effect-уровневая координация: soft-delete завершается успехом → reset prefs; финальное решение на `business_contract`).
  - `previewDeletionImpact` обязан учитывать affected prefs (поле `affectedPrefs: List<DictionaryId>` в `DeletionImpact`) — юзер видит «N словарей сбросят выбор компонента в квизе».
- `forward_compat_unknown` / `parser_fail_soft` — fail-soft парсинг unknown / mismatched данных:
  - `ComponentTemplate.fromKey` менять на nullable return (`fromKey(key: String): ComponentTemplate?`), callsite'ы обрабатывают `null` как skip + Crashlytics-лог. См. `template_model.md` § Open Q3.
  - Парсер `parseTemplateValues` обрабатывает unknown primitive type (`{"type": "video", ...}`) → `null` поле + Crashlytics-лог.
  - Парсер `parseTemplateValues` обрабатывает type-mismatch со schema (например schema ждёт `text`, в JSON `image`) → `null` поле + Crashlytics-лог.
  - Покрытие → mapper tests с негативными golden fixtures (детали fixture — на test шаге).
- `mapper_golden_fixtures` — test fixtures `fixtures/component_values/*.json` (golden round-trip per template) в `core-db-impl` для защиты template immutability. См. `template_model.md` § Open Q10.
- `transactional_save_lexeme` — `@Transaction` на repository-методе сохранения лексемы (INSERT lexeme + N INSERT component_values, rollback при FK violation). См. `template_model.md` § Open Q11.

## Open questions

- **Новый `ComponentApi` interface vs расширение существующего `LexemeApi`?** Best-guess: **расширить `LexemeApi`** (F030: `LexemeApi` уже содержит component-методы — `getComponentTypes` (suspend list), `addComponentValue`, `addLexemeWith*` shims — добавление новых CRUD-методов укладывается в существующий интерфейс). Альтернатива: выделить новый `ComponentApi` interface — оправдано если набор component-методов разрастётся (10+ методов и собственная семантика lifecycle). Решение финализировать на `business_walkthrough` / `business_contract` шаге. (См. F004.)
- **Один новый screen-модуль или два?** Best-guess: **два — `modules/screen/components_manager/` (global aggregated) + `modules/screen/per_dictionary_components/` (scoped)**. Альтернатива: один модуль с двумя screen-функциями + parameter. Обоснование: модули в PolyTrainer уже разделены по экранам (dictionary, dictionaryTab, quiz, quiztab, wordcard и т.д.); навигация в `app/navigator/` ожидает один `*NavigatorImpl` per screen-module; меньше merge-conflict-зон. Решение финализировать в `infra_walkthrough` / `business_walkthrough` (минимальное влияние на scope flag — оба варианта остаются `infra_touched=true`).
- **Где живёт `ComponentBlock(name, content)` wrapper?** Best-guess: **`modules/widget/component_widgets/`** (Tier 2 в новом widget-модуле). Альтернатива: `core/ui/` (Tier 1). Обоснование: `ComponentBlock` использует доменные типы (имя компонента → label) + специфичен для domain «component»; `core/ui` — generic primitives. Решение оставить на `ui_design_tree` шаге.
- **Storage migration strategy для JSON rewrite — SQL `json_*` или Kotlin?** Best-guess: **SQL `json_object` / `json_set`** (bundled SQLite ≥ 3.38 — гарантировано). Альтернатива: Kotlin loop с batched UPDATE (chunks по 1k rows в одной transaction). **QA-критерий:** решение должно укладываться в **<5 секунд** на типичном словаре (5k-25k values) и **<30 секунд** на edge-case (100k+ values). 25k rows × парсинг JSON в Kotlin loop на старых девайсах может дать десятки секунд миграции — поэтому SQL предпочтительнее. Финализация — на `data_design_tree` с benchmark.
- **`SettingsNavigator` cross-feature reuse?** Best-guess: **расширить existing `SettingsNavigator`**, добавить `openComponentsManager()`. Альтернатива: создать отдельный `ComponentConstructorNavigator`. Обоснование: navigator уже привязан к `SettingsTabScreen` parent-flow, ComponentsManager — child экран; держать parent → child в одном navigator-контракте — конвенция в PolyTrainer (см. `SettingsNavigator` + `AboutAppScreen`). Решение на `infra_design_tree`.
- **UNIQUE collision при пересоздании компонента с soft-deleted именем?** (F018/F021) Best-guess: **(B) убрать UNIQUE из Room index и enforce uniqueness в UseCase** через SELECT перед INSERT с разветвлением по scope (F032): `WHERE dictionary_id = ? AND name = ? AND removed_at IS NULL` (per-dict) либо `WHERE dictionary_id IS NULL AND name = ? AND removed_at IS NULL` (global). Альтернативы: (A) partial unique index `WHERE removed_at IS NULL` — Room не видит partial index в schema-export (та же проблема, что и для `component_values`); (C) blacklist имени до hard-delete — нельзя пересоздать пока recovery возможен (но recovery вне scope этой фичи); (D) auto-rename при коллизии (`name (2)`) — UX-сюрприз для юзера; (E) «user-defined всегда per-dict, global только built-in» — противоречит ui_placement.md «scope=На все словари». Решение финализировать на `data_design_tree` / `business_contract`.
- **Стратегия `ComponentTypeApiEntity.template` после nullable `fromKey`?** (F019) Best-guess: **(B) `toApiEntity()` возвращает `ComponentTypeApiEntity?` (skip row при unknown template) + Crashlytics-лог**, согласовано с `template_model.md` § Open Q3. Альтернативы: (A) `template: ComponentTemplate?` nullable в API-entity — каскадно nullable во всём UI; (C) DAO filter `WHERE template IN (known_keys)` — skip на чтении (но требует hardcoded list known keys в SQL). Решение на `business_contract` шаге.
- **Cross-scope user-defined name identity?** (F039) Best-guess: **(A) two-prong SELECT в UseCase** (см. aspect `userdefined_identity_invariant`) — при создании per-dict `"X"` SELECT'ом проверить отсутствие active global `"X"`; при создании global `"X"` — отсутствие active per-dict `"X"` в любом словаре. Минимум изменений (domain ref остаётся `UserDefined(name)`, `QuizGameImpl.matchesRef` не трогаем, `quiz_configs.component_refs` JSON-схема не меняется, prefs не мигрируют). Альтернатива: **(B) обогатить `ComponentTypeRef.UserDefined` `typeId: ComponentTypeId`** — устраняет identity-ambiguity на типовом уровне, но breaking change в domain + миграция `quiz_configs.component_refs` (rewrite `user:<name>` → `user:<typeId>`) + миграция `QuizPickerPrefKey` payload. Решение финализировать на `business_design_tree` / `data_design_tree`.
- **`LexemeDbEntity.componentValueListDb` фильтрация soft-deleted в `@Relation`?** (F031) Best-guess: **(A) post-load filter в `toApiEntity()`** — отфильтровать `removedAt != null` из загруженного `componentValueListDb` до маппинга. Минимум изменений (единственный callsite), не требует переписывания query. Альтернатива: (B) custom `@Query` с явным JOIN `lexeme ⋈ component_values WHERE cv.removed_at IS NULL` вместо `@Relation` — корректнее на DB-уровне, но переписывает чтение лексемы. Решение на `data_design_tree`.
- **Prefs cleanup strategy при soft-delete component_type — расширение `PrefsProvider` API или UseCase композиция?** (F049) Best-guess: **(B) UseCase композиция через `DictionaryApi.getDictionaryList()`** (возвращает `List<DictionaryApiEntity>`, UseCase извлекает `id` field; альтернативно `flowDictionaryList()` для реактивного варианта) — iterate all dict IDs, для каждого собрать ключ через `quizPickerPrefKey(dictId)`, прочитать `getStringByRawKey(key)`, проверить match на удаляемый ref (`builtin:<key>` или `user:<name>`), при совпадении сбросить через `setStringByRawKey(key, null)` либо delete entry. Не требует расширения `PrefsProvider` API (остаётся `getStringByRawKey`/`setStringByRawKey`), не требует новых зависимостей в datasource-модуле. Альтернатива: **(A) добавить scan-by-prefix API в `PrefsProvider`** (`findKeysWithPrefix(prefix: String): List<String>`) — корректнее на perf при больших коллекциях словарей (O(1) на DataStore vs O(N) iteration через domain), но расширяет публичный contract datasource-модуля и требует новый метод в DataStore-backing impl. Финализация — на `business_design_tree`.
