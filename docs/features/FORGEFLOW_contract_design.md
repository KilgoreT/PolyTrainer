# ForgeFlow — декомпозиция шага `contract` Business sub-flow

Документ описывает разделение монолитного шага `contract` на последовательность под-шагов, чтобы шаг проектирования контракта TEA-фичи стал управляемым, удобно ревьюимым и опирался на естественный источник входной информации (Figma-макет).

Дополнение к [`FORGEFLOW_design.md`](FORGEFLOW_design.md) §4 (Business sub-flow).

---

## Контекст и мотивация

В TEA-архитектуре проекта контракт фичи (или самостоятельного widget'а со своим reducer'ом) состоит из:

- `State` data class + nested + extension-функции (мутаторы)
- `Msg` sealed interface — события (UI + Datasource + Reactive)
- `Effect` — команды (Datasource / Navigation / Ui)
- `Reducer` — `(State, Msg) → (State, Set<Effect>)`
- `FlowHandler` — реактивные подписки на внешние источники, эмит Msg
- `UseCase` — boundary к Data слою

Это много сущностей. Один шаг `contract` который описывает всё это сразу:

- Невозможно ревьюить целиком — слишком большой
- Не имеет внутренней последовательности — агент пишет «как идёт»
- Размывает границы между слоями TEA

**Тезис документа:** контракт надо разделить на под-шаги по естественной последовательности извлечения информации из источников (макет + бизнес-инварианты).

## Источники входных данных

У контрактного блока **три возможных источника** входных данных. Какой используется — зависит от сценария задачи.

### 1. Figma-макет

- **Откуда:** designer
- **Когда основной:** новая фича / изменения UI / новые экраны
- **Что даёт:**
  - Отображаемые элементы → поля `State`
  - Точки взаимодействия (тапы, инпуты, жесты) → `UI Msg`
  - Альтернативные frames (loading / empty / error / dialog open / action mode) → state machine + инварианты
- **Чего НЕТ:**
  - Effects (что за кадром при действии)
  - Datasource Msg (ответы от data слоя)
  - FlowHandler subscriptions (реактивные источники)
  - UseCase методы
  - Edge cases (запрос упал, data пустая)

### 2. Существующая спека

- **Откуда:** `docs/features-spec/<feature>.md` — результат предыдущего запуска `contract_spec`
- **Когда основная:** subsequent runs для существующей фичи (bugfix, refactor без UI-изменений)
- **Что даёт:**
  - Готовая структура State / UI Messages / IO / UseCase
  - Бизнес-описание + User Stories
  - Инварианты состояния
  - Известные edge cases
- **Авторитет — разделённый по типу содержимого** (см. §«Приоритет источников при конфликтах» ниже):
  - Для **замысла** (бизнес-описание / user stories / инварианты) — спека авторитарна
  - Для **технических деклараций** (имена Msg / Effect / UseCase методы / сигнатуры) — код авторитарен (если код существует)

### 3. Код

- **Откуда:** `State.kt`, `Message.kt`, `Reducer.kt`, `EffectHandler.kt`, `UseCase.kt` модуля фичи
- **Когда основной:** legacy refactor без существующей спеки (первый прогон новой decomposition на старой фиче)
- **Что даёт:**
  - Точное текущее состояние реализации
  - Реальные имена / типы / сигнатуры
- **Особенность:** код может расходиться с замыслом. При использовании как primary — извлекается **«как должно быть»** (с очисткой косметики, рефактором), не **«как есть плохо»**.

### Матрица сценариев

| Сценарий | Макет | Спека | Код | Primary |
|---|---|---|---|---|
| Новая фича | ✓ | — | — | Макет |
| Изменения макета existing | ✓ (новая) | ✓ | ✓ | Макет + спека (diff) |
| Bugfix existing с спекой | — | ✓ | ✓ | Спека |
| Refactor без UI-изменений | — | ✓ | ✓ | Спека + код |
| Legacy refactor (нет спеки) | — | — | ✓ | Код → формирует первую спеку |
| Refactor с устаревшей спекой | — | ✓ (stale) | ✓ (truth) | Код + спека (как должно быть) |

### Приоритет источников при конфликтах

**Принцип — разделённый авторитет:**

- **Замысел** (бизнес-описание, user stories, инварианты, какой фича должна быть) → авторитет за **спекой** (или макет'ом, если он пришёл с изменениями)
- **Технические декларации** (имена `Msg`, `Effect`, методов `UseCase`, сигнатуры, типы payload) → авторитет за **кодом** (если код существует и реализует фичу)

**Почему так:**
- Если в коде `Msg.LoadFailed`, а в спеке записан `Msg.LoadError` (опечатка из предыдущего прогона) — код = truth, спека правится под код. Иначе ошибка реплицируется через прогоны.
- Если в коде `isLoading: Boolean`, а в спеке `loading: Boolean` — то же.
- Но если код имеет поле `temp: String` которое нигде не используется (мёртвый код) — это tech debt в коде, спека его не описывает (замысел не включает мёртвый код).
- **Бизнес-описание / user stories** — это решения людей, не код. Код их не «знает». Здесь спека всегда truth.

**По режимам:**

1. **Макет пришёл → макет главный для UI и замысла.** Новая UI-картина диктует what to build. Спека (если была) обновляется под макет. Код — прошлое состояние, его технические декларации могут потерять актуальность.

2. **Макет нет + спека есть.**
   - **Для замысла** (бизнес-описание / user stories / инварианты) — спека авторитарна. Расхождение «код не соответствует спеке по поведению» = tech debt в коде, чинится в `implement`.
   - **Для технических деклараций** (Msg/Effect/UseCase имена и сигнатуры) — **код авторитарен** при существующей реализации. Расхождение «спека говорит `Msg.LoadFailed`, код имеет `Msg.LoadError`» = **ошибка в спеке**, спека правится под код. Это защита от spec-as-source loop.

3. **Макет нет + спеки нет → код главный во всём.** Из кода извлекается контракт в формате «как должно быть» (с очисткой, рефактором). Получается первая версия спеки.

**Правило для шагов в режиме 2** — операционализация сверки spec↔code:

Технические декларации спеки **сверяются с актуальным кодом конкретными grep'ами**. Не «глазами посмотрел» — реальные tool-операции.

Конкретные сверки для каждого шага:

- **`contract_state`** в режиме 2:
  - Grep `^data class\|^@Stable\|^@Immutable` по `State.kt` модуля фичи → выписать актуальную структуру State + nested data classes
  - Grep `^fun State\.\|^fun .*State\.` по `State.kt` → выписать существующие extension-функции и computed properties
  - Сравнить с разделом `## State` в спеке. Расхождения (имена полей / типы / nested структура / computed property отсутствует или новая) → раздел `## Расхождения spec ↔ code` в output

- **`contract_ui_msg`** в режиме 2:
  - Grep `^\s*data (object|class)` по `Message.kt` → выписать все варианты Msg
  - Grep `is Msg\.` по `Reducer.kt` → перечислить какие Msg реально обрабатываются
  - Сравнить со спекой. Расхождения → output

- **`contract_io`** в режиме 2:
  - Grep `sealed interface .*Effect\|^\s*data (object|class) .* : .*Effect` по `EffectHandler.kt` (или соответствующим файлам) → выписать все Effects
  - Grep `flow.*\(\|\.flow.*\|collectLatest\|\.collect\s*{` по `EffectHandler.kt` → выписать подписки Subscribers
  - Сравнить со спекой
  - Расхождения → output

- **`contract_usecase`** в режиме 2:
  - Read `UseCase.kt` interface → выписать все методы с сигнатурами
  - Сравнить со спекой раздел `## UseCase`. Расхождения (имена методов / сигнатуры / параметры / return types) → output

**Правила:**
- Без явных grep-операций в работе агента (видны в conductor log) — output_criteria шага не пройдёт
- Раздел `## Расхождения spec ↔ code` обязателен в output **даже если расхождений нет** (явный «расхождений не обнаружено: проверено X имён Msg, Y методов UseCase»). Это защита от «hand-waving» без проверки.
- При расхождении — приоритет коду (технические декларации). Спека в этом прогоне правится под код. Это защита от spec-as-source loop.

### Детекция режима

Агент определяет режим по наличию артефактов на старте контрактного блока:

- В input есть макет (или ссылка на Figma в брифе задачи) → **режим 1** (макет-driven)
- Макета нет, `scope_analysis.output` указал существующее имя файла спеки в `spec_dir/` → **режим 2** (spec-driven)
- Нет ни того, ни другого → **режим 3** (code-driven, legacy)

**Контракт от `scope_analysis`:** в свой output (`02_scope.md`) `scope_analysis` обязан включить поле `spec_filename`:
- Имя существующего файла спеки в `spec_dir/` (например `dictionary-list.md`), если фича существует
- `null` или пометка «новая фича», если спеки ещё нет

Контракт-блок читает `parent.scope_analysis.output`, находит `spec_filename`, на его основе определяет режим. Никаких эвристик по `spec_dir/`.

## Принцип порядка

**От видимого к границе с Data.** Снижаемся по слоям TEA:

```
видимое (State + extensions)
   ↓
события от пользователя (UI Msg + reducer-логика)
   ↓
реакции на события (Effects + Datasource Msg + reducer-логика)
   ↓
реактивные подписки (FlowHandler subscriptions)
   ↓
граница к Data (UseCase)
```

Каждый шаг опирается на предыдущий. UseCase **последний** — потому что его методы вытекают из effects + subscribers, не наоборот.

## Последовательность под-шагов

### 1. `contract_state`

**Источник:** в зависимости от режима (см. [Источники входных данных](#источники-входных-данных)):
- Режим 1 (макет-driven): Figma-макет (визуал + альтернативные frames)
- Режим 2 (spec-driven): существующая спека `spec_dir/<feature>.md` — раздел `## State`
- Режим 3 (code-driven, legacy): текущие файлы `State.kt` + nested data classes, существующие extensions — извлекаем «как должно быть» с очисткой

**Что делает — основное:**
- Описать `data class State` + вложенные модели — все отображаемые поля
- Описать **state machine + инварианты** — какие комбинации полей валидны (loading/empty/error/data states из альтернативных frames макета)
- Описать **extension-функции** — мутаторы для переходов между состояниями (выводятся когда видишь как состояние меняется между frames)
- Описать **computed properties** (extension `val`) — для derived полей (см. ниже)
- Описать удаляемые поля и extensions (если переделка существующего)

**Что делает — обязательная проверка алгебры данных:**

Каждый раз когда формулируется поле или группа полей State, **обязательно** прогнать через **правила моделирования** на основе [Моделирование State](../guides/state-modeling.md):

1. **Тип связи — product или sum?** (применяется **только к nested полям и группам**, для top-level State — см. раздел «Внимание» ниже, top-level всегда `data class`)
   - Поля независимы (любая комбинация валидна) → **product** = `data class` (см. [§5](../guides/state-modeling.md#5-product-types--произведение))
   - Поля взаимоисключающие (XOR) → **sum** = `sealed class` (см. [§6](../guides/state-modeling.md#6-sum-types--сложение))
   - Союз «И» → product, союз «ИЛИ» → sum

2. **Подсчёт вариантов.** Для проектируемой группы полей посчитать `count(...)`. Найти:
   - **Лишние варианты** — невалидные комбинации, которые тип допускает (например, `hasSuspension=false, fullSuspension=true` — невалидный «нет амортизации, но двухподвес»). Если есть — переходить к sum type.
   - **Недостающие** — состояния которые есть в макете, но тип не позволяет выразить.
   - Сверка с количеством alternative frames в макете: если 4 frames → должно быть 4 валидных варианта (или объяснить расхождение).

3. **Ловушка ложного «или»** (см. [§7](../guides/state-modeling.md#7-ловушка-ложного-или)). Если возник соблазн расширить sealed class новым вариантом, проверить: это **новое состояние** или **новая ось координат** (например visibility placeholder при disabled)? Если ось — это product, не sum. Решение: вернуться к product + helper-функция для доменной логики.

4. **Computed properties для derived полей** (см. [§12](../guides/state-modeling.md#12-selectors) и [state-and-extensions Принцип 6](../guides/state-and-extensions.md#принципы-стейта)). Если поле выводится из других (`hasNoDictionary` = `currentDict == null`, `canSubmit` = `name.isNotEmpty() && !isLoading`) — **не хранить** в data class, оформлять как extension `val` с getter рядом с data class в State.kt.

5. **Async-данные — простой паттерн по умолчанию.** Для полей которые загружаются с backend:
   - **Дефолт: `data: T? = null` + `isLoading: Boolean`** — покрывает и первую загрузку, и refresh с retain существующих данных. `null` = «ещё не загружали», `emptyList()` = «загрузили, пусто», `isLoading=true` при `data != null` = refresh-overlay.
   - **Sealed `Loadable<T>` применять ТОЛЬКО** для нетривиальных случаев: >2 взаимоисключающих content-состояний (например, Error с typed cause), pagination/streaming со своими промежуточными состояниями, multi-source данные с разной семантикой.
   - **Простой паттерн дешевле и покрывает 95% случаев.** Не плодить sealed обёртки «на всякий случай» (см. [`memory: simple state for loading`](file:///Users/kilg/.claude/projects/-Users-kilg-AndroidStudioProjects-PolyTrainer/memory/feedback_simple_state_loading.md) и [`§9 LCE/Loadable`](../guides/state-modeling.md#9-lce-и-loadable) — раздел про «сжатие» классов).

6. **Нормализация по принципу «State = БД»** (см. [§11](../guides/state-modeling.md#11-state-как-база-данных)). Для коллекций:
   - НЕ дублировать данные между списками («все items» + «viewed items» как два списка — антипаттерн)
   - `Set<Id>` вместо `Map<Id, Boolean>` для «принадлежит или нет» (3 варианта vs 2)
   - Foreign keys через id-ссылки, не вложенные объекты дублированно

7. **Dependency Rule** (см. [§10](../guides/state-modeling.md#10-dependency-rule)). Если в State лежит доменная модель (`Offer`, `Word`, `Lexeme`) и нужно добавить feature-специфичный флаг (`isSelected`, `isViewed`) — **не класть в доменную модель**, а:
   - Обернуть в feature-level wrapper (`OfferListItem(offer: Offer, isViewed: Boolean)`), **или**
   - Хранить рядом как отдельный `Set<Id>` (нормализация, см. п.6)

8. **Editable поля отдельно от domain** (см. [§14](../guides/state-modeling.md#14-редактируемые-поля)). Если экран редактирует доменную сущность, держать:
   - `originalX: X` — нетронутый снимок с backend
   - `editedX: ...` — редактируемая копия / отдельные поля редактирования
   - Не мутировать `originalX` — иначе ломается откат, undo, синхронизация с backend

**Внимание:**
- **State ≠ «то что нарисовано».** Включает управляющие поля (например `selectedTermIds: Set<>` для action mode) которые выводятся из state machine, не из визуала.
- **State = данные, не объекты** (см. [§2](../guides/state-modeling.md#2-state--это-данные)). Никакой бизнес-логики, identity, наследования внутри State. Только immutable факты.
- **Top-level State — БЕЗУСЛОВНО `data class`** даже если правило моделирования №1 указывает на sealed (см. [§16 Q&A](../guides/state-modeling.md#16-qa)). Это override над правилом №1 специально для top-level. Sealed для XOR-семантики — внутри полей (nested groups). Top-level `data class` даёт `copy()`-эргономику для апдейтов и избегает болезненного рефакторинга при росте состава. Правило моделирования №1 применяется к nested структурам, не к корню State.

**Выход:** контракт State, явно содержащий:
1. `data class` с полями + вложенные модели
2. Обоснование выбора product/sum для каждой группы (с подсчётом вариантов где неочевидно)
3. Список computed properties (extension `val`)
4. Список extension-функций мутации
5. Инварианты состояния (что должно быть консистентно)
6. (Если переделка) список удаляемых полей / extensions / computed

### 2. `contract_ui_msg`

**Источник:** в зависимости от режима + контракт State из шага 1:
- Режим 1 (макет-driven): Figma-макет (точки взаимодействия)
- Режим 2 (spec-driven): спека `spec_dir/<feature>.md` — раздел `## UI Messages`
- Режим 3 (code-driven, legacy): `Message.kt` (sealed interface Msg) + `Reducer.kt` (reducer-логика per Msg)

**Что делает — основное:**
- Каждый тап / инпут / жест → один `UI Msg`
- Для каждого Msg описать:
  - **Trigger** — что делает пользователь
  - **State changes** — какие extensions State вызываются
  - **Effects** — **forward reference как гипотеза**: «здесь, вероятно, нужен effect X» (детали + проверка + переписывание на шаге 3)
- Описать удаляемые messages (если переделка)

**Шаг 2 — черновик, шаг 3 — канон.** Forward references на effects в шаге 2 — это **гипотезы агента** в процессе TEA-мышления («пользователь нажал → state меняется → нужен effect»). Они **не являются источником истины** для контракта. На шаге 3 (`contract_io`) агент:
- проверяет каждый forward-ref
- дополняет недостающие effects если обнаружил edge cases
- переписывает forward-ref если оказалось что нужно иначе (например, добавить валидацию перед Save)

Шаг 2 не «фиксирует» Msg ↔ Effect связь — финальная связь живёт в шаге 3. Шаг 5 (`contract_spec`) собирает итог из шага 3, не из шага 2. Расхождение между forward-ref шага 2 и финалом шага 3 — **нормально и ожидаемо**, не fragmentation. На паузе шага 2 пользователь ревьюит UI Msg + state changes, не fixates на effect-цепочках.

**Что делает — обязательная проверка моделирования:**

1. **Msg — это sum type** (sealed interface). Каждый новый Msg = один вариант в sum. **Не плодить Msg-варианты для UI-состояний** — это работа State, не Msg. Если возник соблазн «добавить `Msg.ShowError`, `Msg.ShowLoader`» чтобы выразить loading/error на экране — это **антипаттерн**: error и loading это **состояние** (поле State), не **событие** от пользователя. Msg = «что произошло», State = «как выглядит сейчас».

2. **Computed properties в reducer-логике per Msg.** Если решение reducer ветвится по derived условию (`canSubmit`, `hasData`, `isValid`) — читать computed `val` на State, не повторять условие в каждом Msg:
   ```kotlin
   when (msg) {
       // ✅ Использовать computed
       is Msg.Submit -> if (state.canSubmit) {
           state.startSubmit() to setOf(Effect.Save)
       } else {
           state to emptySet()
       }

       // ❌ Дублирование условия
       is Msg.Submit -> if (state.name.isNotEmpty() && !state.isLoading) { /* ... */ }
   }
   ```
   Computed properties объявляются на шаге `contract_state`. На этом шаге — использовать их в reducer-логике.

3. **Shortcut-игнор Msg когда не имеет смысла.** Если действие пользователя не имеет смысла в текущем state — `state to emptySet()` (явный игнор). Это **штатное поведение** при асинхронном рассинхроне: пользователь успел тапнуть на placeholder пока данные не пришли — обработчик должен спокойно ничего не делать, не падать. Не плодить defensive nulls, casts, error-обработку «на всякий случай».

   Пример:
   ```kotlin
   when (msg) {
       // OpenWordCard имеет смысл только когда данные загружены
       is Msg.OpenWordCard -> {
           val word = state.value?.find { it.id == msg.id }
               ?: return state to emptySet()  // игнор при рассинхроне
           state to setOf(NavigationEffect.OpenWordCard(word.id))
       }
   }
   ```

4. **Editable-сценарии.** Если экран редактирует доменную сущность:
   - Msg на изменение editable значения содержит **только новое значение поля** (`Msg.UpdateName(value: String)`), не всю domain-модель
   - Reducer обновляет только editable копию (`state.copy(editedName = msg.value)`), не original
   - Msg на «откатить» (`Msg.RevertEdit`) — копирует original в editable

   Этот пункт **только для edit-сценариев** (формы редактирования). Для read-only экранов пропустить.

**Выход:** контракт UI Msg, явно содержащий:

**Обязательный header в начале артефакта:**
```
> ⚠ **Этот артефакт — черновик.** Финал Msg-списка и связи Msg ↔ Effect живёт в `contract_io.output`.
> Forward-references на effects ниже — гипотезы. Ревью на этом шаге: UI-триггеры, state changes, категоризация Msg.
> Не фиксировать Msg ↔ Effect связь как окончательную.
```

Далее содержимое:
1. `sealed interface Msg` + варианты (`data object` для безпараметрических событий, `data class` для событий с payload)
2. Категоризация (только UI-стороны): действия пользователя / навигация / UI feedback / переключатели / no-op. **Datasource Msg (результаты данных от effects) на этом шаге НЕ описываются** — они появляются на шаге 3 (`contract_io`)
3. Для каждого Msg — trigger, state changes (через какие extension-функции), forward reference на effects если нужны
4. (Если переделка) список удаляемых / новых messages

### 3. `contract_io`

Объединённый шаг **outer world layer**: effects (команды из reducer наружу) + subscribers (реактивные источники извне). Оба канала эмитят Msg обратно в reducer — общая природа, поэтому в одном шаге.

**Источник:** в зависимости от режима + контракт State (шаг 1) + контракт UI Msg (шаг 2):
- Режим 1 (макет-driven): UI Msg (forward refs на effects) + бизнес-сценарии задачи + бизнес-инварианты («что должно автоматически обновиться когда X меняется снаружи»)
- Режим 2 (spec-driven): спека `spec_dir/<feature>.md` — раздел `## IO` (Effects + Datasource Messages + Subscribers + Edge cases)
- Режим 3 (code-driven, legacy): `EffectHandler.kt` (sealed Effect интерфейсы + onEffect + subscribe блоки) + `Reducer.kt` (reducer-логика для return Msg)

**Что делает — Часть A: Effects:**
- **Взять forward-refs шага 2 как гипотезы**, проверить каждую, дополнить недостающие, переписать неверные. Forward-refs шага 2 — **черновик**, шаг 3 даёт **канон**
- Формализовать все effects по трём категориям:
  - **Datasource Effect** — команды к Data слою (load, save, delete)
  - **Navigation Effect** — навигация (open screen, back, exit). Базовый `Back` + per-screen sealed для конкретных экранов
  - **Ui Effect** — UI-побочки (snackbar, toast, vibration)
- Для каждого effect описать:
  - **Source** — какие Msg порождают (ссылки на шаг 2; если в шаге 2 forward-ref отсутствовал — пометить как «новый effect, добавлен на этапе IO» с обоснованием)
  - **Handler** — какой EffectHandler обрабатывает
  - **Action** — что делает (вызов UseCase метода / навигатор / UI-side)
  - **Return Msg** — какой Msg возвращает (если fire-and-forget — `Msg.Empty`)
- Описать **Datasource Msg + reducer-логика** на каждый return — **здесь же, в одном пункте с Effect**. Цепочка `Effect → Handler → return Msg → state change` живёт целиком, не разрывается.
- **Расхождение с шагом 2 — норма.** Не патчить шаг 2 backwards. Все детали Msg ↔ Effect — здесь.

**Что делает — Часть B: Subscribers (FlowHandlers):**
- Описать **отдельным разделом** в выходе — `## Subscribers`, не смешивать с Effects
- Подписки выводятся **из бизнес-инвариантов**, не из UI Msg:
  - «Когда пользователь сменил словарь в другом экране → текущий должен реактивно обновиться»
  - «Когда настройки изменены в Settings → экран Quiz должен учесть новый режим»
  - «Когда DAO эмитит изменение списка → UI автоматически обновляется без явного запроса»
- Для каждой подписки описать:
  - **Source** — на что подписан (UseCase.Flow / Room / DataStore / другой StateFlow)
  - **Emit Msg** — какой Msg эмитит при каждом обновлении (может уже быть в списке Datasource Msg из части A — не дублировать описание, только сослаться)
  - **First emit special** — если первый emit инициирует зависимую загрузку (`FlagsReady` → инициализация, `FlagsUpdated` → обычное обновление) — описать два варианта
  - **Debounce / filter / map** — если применяются операторы
- Reducer-логика для emit Msg описана в части A (если Msg общий) или отдельно (если Msg уникальный для subscriber'а)

**Что делает — обязательная проверка моделирования:**

1. **Effect = sum type (sealed interface).** Каждый Effect = одна команда наружу. Не плодить Effect-варианты для **результатов** — результат это return Msg, не отдельный Effect. Если возник соблазн «Effect.LoadSucceeded» или «Effect.LoadFailed» — это return Msg, а Effect один (`Effect.Load`).

2. **Payload Effect — только параметры команды.** `data class LoadWord(val wordId: Long)` — id нужен для запроса. `data class SaveWord(val wordId: Long, val value: String)` — оба параметра нужны для записи. **Не класть state-копию в Effect** — handler не нуждается в полном state, только в параметрах действия.

3. **Edge cases — обязательная инвентаризация.** Для каждого Datasource Effect ответить:
   - Что если запрос упал (network / db / parsing)? Какой return Msg, как меняет state?
   - Что если data пустая? (Возвращать `Msg.Empty`? `Msg.NoData`? игнорить?)
   - Что если concurrent calls? (отменять предыдущий через `collectLatest`? игнорировать новые? очередь?)
   - Что если data не найдена по id? (`Msg.NotFound` или null result?)

   Эти кейсы **не видны в макете** — обнаруживаются только здесь. Если не описать — баг в проде.

4. **Navigation = Effect, не State.** Открытие экрана / back / exit — sealed interface `XxxNavigationEffect` с per-screen вариантами. **Не флаг в State** (`closeScreen: Boolean`). `ExitApp` — только в per-screen sealed root-экранов, не базовый `NavigationEffect`.

5. **UiEffect — выбор паттерна:**
   - **Через State (текущий стандарт):** `UiEffect.ShowSnackbar` → handler → `UiMsg.Snackbar(show=true)` → State.snackbarState. Подходит для долгоживущих уведомлений (retain через config change).
   - **Side-channel (Channel/SharedFlow в Composable):** для one-shot fire-and-forget (toast, vibration). Не загромождает State.

   Выбирать по сценарию: если уведомление должно retainнуться через config change — State; если one-shot — side-channel.

6. **Subscriber выводится из бизнес-инвариантов — обязательная инвентаризация.** В output `contract_io` должен присутствовать раздел `## Проверка реактивности` со списком вопросов и **явными ответами Y/N**:

   ```markdown
   ## Проверка реактивности

   1. Может ли state экрана устареть, если другой экран изменит данные в БД? — Y/N + обоснование (одна строка)
   2. Есть ли настройки/пользовательские preferences, изменение которых должно отразиться? — Y/N + обоснование (одна строка)
   3. Активный экран должен реагировать на push-нотификации / system events / lifecycle? — Y/N + обоснование (одна строка)
   4. Есть ли DAO-flow / DataStore-flow / другие реактивные источники эмитящие изменения данных этого экрана? — Y/N + обоснование (одна строка)
   ```

   **Правила:**
   - **Каждый ответ N обязательно с обоснованием** (одна строка — почему именно N). Например: «N — экран не зависит от других модулей, данные приходят только по явному запросу пользователя».
   - Без раздела или с ответами без обоснования шаг `contract_io` не считается завершённым (output_criteria validation).
   - **Хотя бы один Y → обязателен соответствующий subscriber** в разделе `## Subscribers`. Если Y, но subscriber'а нет — рассогласование, шаг возвращается на доработку.
   - Штамповка «N×4» без обоснований — недопустима. Каждый N агент должен явно подумать и записать почему.

   Subscribers легко упустить (нет явного триггера в макете). **Soft prompting «подумать ещё раз» не работает** — нужны явные Y/N в артефакте, чтобы при ревью пользователь видел что вопросы заданы.

7. **Subscriber lifecycle.** Подписка живёт от `subscribe(scope, send)` до `dispose()`. Не нужно вручную отписываться в reducer — это работа Mate framework. Если подписка должна остановиться раньше (например, при специфичном условии) — это сигнал что нужен Effect (управление подпиской), а не subscriber-уровень логика.

8. **Effect → return Msg может быть тем же Msg что Subscriber → emit Msg.** Если `Effect.LoadList` возвращает `Msg.ListLoaded(list)` и `Subscriber.flowList` тоже эмитит `Msg.ListLoaded(list)` — это **корректно** (один Msg, два источника). Reducer не различает откуда пришёл — обрабатывает по типу Msg. Не дублировать описание.

**Выход:** контракт outer world layer, явно содержащий:
1. **`## Effects`** — три раздела. **Каждый Effect описан целостным блоком**: Source / Handler / Action / Return Msg / **Reducer-логика для return Msg (inline)** / Edge cases. Цепочка `Effect → Handler → return Msg → state change` не разрывается между разделами.
   - `### Datasource Effect` — sealed interface + варианты + per-effect целостные блоки
   - `### Navigation Effect` — sealed interface + варианты + связь с навигатором
   - `### Ui Effect` — sealed interface + варианты + выбор паттерна (State/side-channel)
2. **`## Subscribers (FlowHandlers)`** — каждая подписка с {Source, Emit Msg, First emit special, Operators, **Reducer-логика для emit Msg (inline, если уникальна)**, Lifecycle}. Если emit Msg тот же что Return Msg от Effect — ссылка «см. Effect.X», без дублирования.
3. **`## Проверка реактивности`** — **обязательный** раздел с явными Y/N ответами (см. правило 6 выше). Без него шаг не завершён.
4. **`## Бизнес-инварианты`** — список инвариантов которые обоснуют наличие subscribers (полезно для ревью «не упустили ли»)
5. (Если переделка) список удаляемых / новых effects / subscribers

### 4. `contract_usecase`

**Источник:** в зависимости от режима + контракт outer world layer (effects + subscribers из шага 3):
- Режим 1 (макет-driven): только шаг 3
- Режим 2 (spec-driven): шаг 3 + спека `spec_dir/<feature>.md` — раздел `## UseCase`
- Режим 3 (code-driven, legacy): шаг 3 + существующий `UseCase.kt` interface

**Что делает — основное:**
- Описать интерфейс `UseCase` (boundary к Data слою)
- Каждый метод **привязан** к конкретному effect или subscriber из шага 3 (видно `// Effect: LoadWord` / `// Subscriber: flowDictionary` в комментарии)
- Описать сигнатуры:
  - `suspend fun methodName(params): ReturnType` — для одноразовых команд (вызывается из `runEffect`)
  - `fun flowMethodName(params): Flow<T>` — для реактивных источников (используется в `subscribe`)
- Если шаг 3 потребовал чего-то чего **нет** в текущем UseCase — явно отметить как «новый метод» или «изменение существующего» (с описанием изменения)
- Описать удаляемые методы (если переделка существующего)

**Что делает — обязательная проверка моделирования:**

1. **UseCase = публичный контракт Business↔Data.** Изменения требуют ревью со стороны архитектуры. Минимизировать поверхность — не плодить методы «на всякий случай».

2. **Domain типы, не feature.** UseCase методы возвращают и принимают **доменные** типы (`Word`, `Dictionary`, `LexemeInfo`), не feature-обёртки (`WordUiItem`, `LexemeState`). Маппинг feature↔domain происходит в UseCase impl или в feature-уровневом мапере (не в reducer'е).

3. **Параметры по доменным id, не UI-типы.** `fun deleteWord(wordId: Long)` — Long, не `WordUiItem`. UI-типы остаются на стороне feature.

4. **Naming convention:**
   - `flow*()` — Flow-методы (`flowCurrentDict`, `flowDictionaryList`)
   - `get*` — синхронное получение (suspend, одноразовое)
   - `add*` / `update*` / `delete*` — CRUD
   - `search*` / `filter*` — поиск с параметрами
   - `change*` — изменение состояния (например, активный словарь)

5. **Suspend для команд, Flow для подписок.** Не возвращать `Flow<T>` из метода если он используется только как «загрузить один раз» — это suspend. И наоборот, не делать suspend для реактивного источника — это Flow.

6. **Один метод = одна ответственность.** Не комбинировать «load + filter + map» в один UseCase метод — фильтрация может быть на reducer-уровне (через Effect) или внутри импла. UseCase — простой канал к Data.

7. **Не писать реализацию.** Только интерфейс. Реализация (`UseCaseImpl`) — в `app/.../di/module/<feature>/`, появляется на шаге `implement`.

8. **Покрытие 1-к-1.** Каждый метод UseCase должен соответствовать хотя бы одному effect или subscriber из шага 3. Метод без consumer'а — мёртвый код. И наоборот: каждый effect/subscriber должен ссылаться на конкретный метод UseCase (или маркер «нужен новый метод»).

**Выход:** контракт UseCase, явно содержащий:
1. `interface XxxUseCase` со всеми методами
2. Для каждого метода — сигнатура + ссылка на effect/subscriber из шага 3 (комментарий)
3. Раздел «новые методы / изменения существующих» (если переделка)
4. Раздел «удаляемые методы» (если переделка)

### 5. `contract_spec`

Финальный шаг контрактного блока. **Агрегирует 4 предыдущих артефакта в черновик проектной спеки** и кладёт его в **feature dir** (`contract_spec.md` в директории текущего flow run).

**Физическая публикация спеки в `spec_dir/` происходит на шаге `publish_spec`** (после `implement`) — это сохраняет инвариант «спека = срез реальности после реализации», защищает от drift при abort/feedback loop.

**Источник:**
- `contract_state.output` (State)
- `contract_ui_msg.output` (UI Msg + reducer-логика)
- `contract_io.output` (Effects + Subscribers + Datasource Msg)
- `contract_usecase.output` (UseCase)
- **Существующая спека** в `spec_dir/<feature>.md` (если есть для этой фичи) — **читается как reference**, не перезаписывается
- Бриф задачи / `parent.scope_analysis.output` — для бизнес-описания и user stories

**Что делает:**
- Создаёт черновик спеки в формате описанном ниже
- Если в `spec_dir/<feature>.md` уже есть спека — читает её, сохраняет неизменные разделы (бизнес-описание, user stories если они не задеты текущим контрактом), обновляет затронутые разделы под новый контракт
- Спека = **срез финального состояния** «как должно быть» **после реализации этой задачи**. Не история, не было/стало, не план
- Файл записывается **в feature dir**, не в `spec_dir`. В `spec_dir` его перенесёт `publish_spec`

**Структура спеки:** подробно описана в разделе [Формат проектной спеки](#формат-проектной-спеки) в конце документа. Краткий обзор:

1. Заголовок (`# <Feature Name>`)
2. Бизнес-описание
3. User Stories
4. State (структура + computed + инварианты)
5. UI Messages (sealed interface + reducer-логика per Msg)
6. IO (Effects + Datasource Messages + Subscribers)
7. UseCase
8. Тестовые сценарии (опционально)

**Что делает — обязательная проверка:**

1. **Спека = срез финального состояния.** Если в артефактах контракта есть «удалить X» или «было Y, стало Z» — отразить **только Z** в спеке, X и Y не упоминать. Спека — это «что есть после реализации этой задачи», читатель не должен знать что было до.

2. **Минимизация дублирования.** Не копипастить ADT-обоснования из contract_state. Не копировать длинные edge case разборы из contract_io. Спека — это **контракт для разработчиков**, не аудит контрактного процесса.

3. **Чтение существующей спеки — обязательно.** Перед записью прочитать `spec_dir/<feature>.md` (если есть). Если есть разделы которые не затронуты текущим контрактом (например бизнес-описание, user stories которые контракт не менял) — **сохранить как есть**. Контракт обычно меняет State/Msg/Effect/UseCase, но не бизнес-описание.

4. **При bugfix без значимых изменений контракта** — спека может получить минимальные обновления (например, fix flag в computed property). Это норма — спека остаётся актуальным срезом.

5. **Имя файла:** по модулю/экрану/функциональности (`dictionary-form.md`, `word-card.md`). Если новая фича — выбрать имя соответствующее модулю реализации. Если редактируется существующая — использовать существующее имя.

**Выход:** `contract_spec.md` — файл в feature dir текущего flow run, содержит **черновик спеки** в финальном формате (см. ниже). Этот же файл затем читается `publish_spec` и публикуется в `spec_dir`.

**Влияние на остальной business sub-flow:**

- **`update_spec` убирается** — его роль (план изменений «было/стало» в спеке) поглощена `contract_spec`. Контракт сам пишет «как должно быть».
- **`publish_spec` получает новую роль** — физически публикует спеку из `contract_spec.md` (feature dir) в `spec_dir/<feature>.md`:
  - Читает `contract_spec.output` (черновик из feature dir)
  - Если в `implement` вылезли edge cases меняющие контракт — обновляет соответствующие разделы черновика под итог реализации (опционально)
  - Записывает финальную версию в `spec_dir/<feature>.md`
  - Дополнительно — копирует PUML-схемы (`.puml` файлы из feature dir в проектную директорию схем)

**Почему так:**
- Между `contract_spec` и `publish_spec` спека = **черновик в feature dir**, не загрязняет `spec_dir/`
- Если flow прерывается (abort / feedback loop / dropout) после `contract_spec` но до `publish_spec` — `spec_dir/` остаётся **нетронутым** (никакого «обещания из будущего»)
- Feedback loop архивирует `business/` в `business_iter1/`, и `contract_spec.md` уходит вместе с workspace — `spec_dir/` сохраняет последнюю опубликованную версию
- `publish_spec` — естественное место для адаптации спеки под итоги `implement` (если реализация вскрыла edge cases меняющие контракт)

|---|---|---|---|
| 1 | `contract_state` | Макет / спека / код (зависит от режима) | State + nested + extensions + computed + инварианты |
| 2 | `contract_ui_msg` | Макет / спека / код + State | UI Msg + reducer-логика + forward refs на effects |
| 3 | `contract_io` | UI Msg + State + бизнес-сценарии (макет/код) или спека (раздел IO) | Effects (Datasource/Navigation/Ui) + Datasource Msg + reducer-логика + Subscribers + edge cases |
| 4 | `contract_usecase` | Шаг 3 + спека (раздел UseCase) / код (если legacy) | UseCase interface |
| 5 | `contract_spec` | Артефакты шагов 1-4 + существующая спека (если есть, читается как reference) | Черновик спеки `contract_spec.md` в feature dir (физическая публикация в `spec_dir/` — на шаге `publish_spec` после `implement`) |

---

## Типичные ошибки

1. **Попытка вывести Effects из макета.** Effects не видны в макете — нет смысла угадывать на шагах (1)/(2). Дыры всплывут на (3). Решение: явный forward reference на (2).

2. **State = «то что нарисовано».** Не учитывает управляющие поля (action mode, selection, dialog state). Решение: на (1) учитывать визуал + state machine.

3. **Subscribers упущены.** Не видны ни в макете, ни в UI Msg. Обнаруживаются только если задать вопрос «что должно автоматически обновиться когда X изменится снаружи». Решение: на (3) — **отдельный раздел Subscribers в output'е**, не смешивать с Effects. Промпт обязывает агента сделать «бизнес-инвариантную проверку».

4. **UseCase придуман первым.** Соблазн начать «с данных» — какие методы нам нужны. Это инверсия: UseCase вытекает из Effects + Subscribers, не наоборот. Решение: UseCase последний (шаг 4).

5. **Edge cases в макете.** Макет показывает happy path. Сценарии «запрос упал», «data пустая», «network unavailable» обычно не нарисованы. Решение: явно проходиться по edge cases на (3) — обязательная инвентаризация для каждого Datasource Effect.

6. **Effects и Subscribers смешаны.** При объединении в `contract_io` есть риск что агент опишет всё «в кашу». Решение: промпт обязывает разделить на части A (Effects) и B (Subscribers), output разнесён по разделам.

## Альтернативы (рассмотрены, отброшены)

### A. 5 шагов — раздельные `contract_effect` и `contract_subscriber`

Effects и Subscribers — разная природа выведения (Effect из UI Msg, Subscriber из бизнес-инвариантов). Можно разделить для чистоты процесса.

**Почему не выбрано:** оба эмитят Msg в reducer (общая природа результата), физически живут в одном классе `XxxEffectHandler` (subscribe + runEffect), малый объём subscribers (1-2 на экран) делает отдельный шаг overhead. Решение — объединить в `contract_io` с обязательным разделением output'а на части A/B.

### B. 3 шага — `state` / `logic` / `usecase`

Один большой шаг `logic` (Msg + Effect + Subscriber + Reducer всё вместе).

**Почему не выбрано:** монолит, теряется послойная декомпозиция, ревьюить нечем — слишком большой.

### C. 5 шагов — разделить `contract_io` на effects и data-msg

Отдельный шаг для Effect (что делается) и Datasource Msg (что возвращается).

**Почему не выбрано:** Effect и его return Msg связаны цепочкой `Effect → Handler → Msg → state change`. Разделить — придётся склеивать цепочки на третьем шаге.

### D. UseCase первым

Изначально предлагался порядок: UseCase → State → UI Msg → Effects → ...

**Почему не выбрано:** UseCase методы **вытекают** из Effects + Subscribers, не наоборот. Начать с UseCase — значит угадывать что нужно от Data, не зная зачем. Инверсия естественного потока.

### E. UseCase объединить с `contract_io`

Имея effects и subscribers в одном шаге, можно сразу там же сформировать UseCase.

**Почему не выбрано:** UseCase — **синтез** (обобщение методов), не **анализ** (моделирование цепочек). Объединение смешивает разные мыслительные процессы. Отдельная пауза на UseCase даёт ревью именно публичного контракта Business↔Data — критично, так как это поверхность которая видна другим модулям.

## Scope применения

**Базовое правило: одна фича за раз.** Контракт-блок Business sub-flow работает над **одной фичей** (один экран или один виджет с собственным reducer'ом). Изменения, затрагивающие несколько фич — раскладываются на последовательные задачи через срочный бэклог.

### Виджет с собственным reducer'ом — отдельная фича, не часть родительской

Самостоятельные виджеты (`modules/widget/*` с TEA-структурой — например `dictionaryappbar`) являются **отдельными фичами**, не частями родительских экранов.

- Контракт-блок над виджетом — отдельный запуск Business sub-flow.
- Если по ходу работы над фичей экрана обнаружилась необходимость изменить виджет — изменение виджета выносится в **срочный бэклог**, после завершения текущей фичи стартует новая фича с описанием из бэклога.
- Это даёт чёткие границы scope, простые контракты на ревью, и виджеты остаются переиспользуемыми.

### Multi-feature change (несколько фич одной задачей)

Случай когда задача логически затрагивает несколько фич (например: новое поле в общем `Dictionary` → меняет дисплей в `dictionary-list` и `wordcard`; изменение параметра в `PrefsProvider` → меняет subscriber'ы нескольких экранов; общий UseCase меняет API → каскад в feature-обёртках).

**Правило (аналогично виджетам):**

- **scope_analysis выбирает primary feature** — ту фичу, изменения в которой инициировали задачу или несут основную бизнес-нагрузку. Остальные затронутые фичи — **secondary**.
- **Текущий business sub-flow работает только над primary feature.** Контракт-блок пишет один контракт + одну спеку для primary.
- **Secondary features → срочный бэклог** с описанием «follow-up из задачи X: в primary изменилось Y, нужно протянуть Z в фичу N».
- После завершения primary — старт новой фичи под secondary (через срочный бэклог), своя decomposition контракта.

**Что про common changes (UseCase / domain):**

- Изменения в **общих моделях** (`Dictionary`, `Word` — domain entities, не feature-specific) и **общих UseCase интерфейсах** — это не «feature», это часть infra/data слоя. Лежат в соответствующих sub-flow (Infra / Data) и описываются их output'ом, **не в feature-spec**.
- Согласовано с Dependency Rule: domain не знает о feature, feature-spec не описывает domain-уровневые сущности.
- В контракте feature — ссылка на используемый общий UseCase / domain type через `parent.<sub-flow>.output` или явные input. Spec фичи **использует** их, не **определяет**.

### Headless-фичи (без UI)

Контракт-блок Business sub-flow рассчитан на **UI-фичу** — экран или виджет с visible UI, reducer'ом, Msg'ами от пользователя.

Для **headless-фич** (без visible UI — background sync, scheduler, periodic jobs, in-process services) контракт-блок **не применяется**. Такие фичи проходят через **Infra или Data sub-flow** без контракт-декомпозиции — описываются в стандартных артефактах этих sub-flow (`design_tree`, `implement`).

Признак headless-фичи: нет UI Messages, нет State в традиционном смысле (нет reducer'а), нет Composable. Только бизнес-логика реализованная в background-механизмах.

## Применение в Business sub-flow

В `flows/business.yml` контрактный блок:

```yaml
- step: contract_state
  pause: true
  input: [parent.scope_analysis.output]   # макет — часть scope

- step: contract_ui_msg
  pause: true
  input: [parent.scope_analysis.output, contract_state.output]

- step: contract_io
  pause: true
  input: [parent.scope_analysis.output, contract_state.output, contract_ui_msg.output]

- step: contract_usecase
  pause: true
  input: [contract_io.output]

- step: contract_spec
  # pause не объявлена — это шаг агрегации без новых решений;
  # ревью на pause шага contract_usecase покрывает финальный контракт
  input:
    - parent.scope_analysis.output         # для business-описания / user stories
    - contract_state.output
    - contract_ui_msg.output
    - contract_io.output
    - contract_usecase.output
```

`parent.<step>.output` — ссылка на шаг master plan из child sub-flow (см. [`dsl.md → Parent-ref`](../forgeflow/spec/dsl.md#parent-ref--доступ-из-child-sub-flow-к-master-plan)). Используем потому что `scope_analysis` живёт в master plan, а через subflow.input передача внутрь child пока не поддержана.

Каждый шаг `pause: true` — пользователь ревьюит на стыке слоёв.

Дальше идут существующие шаги Business sub-flow:
- ~~`update_spec`~~ — **убран**, его роль поглощена `contract_spec` (теперь `contract_spec` пишет «как должно быть» сразу, план изменений не нужен)
- `design_tree` — без изменений
- `test` — без изменений
- `implement` — без изменений
- `publish_spec` — **физическая публикация спеки** (читает `contract_spec.output`, опционально корректирует под итог `implement` если вылезли edge cases, кладёт в `spec_dir/<feature>.md`) + копирование PUML-схем
- `summary` — без изменений

## Формат проектной спеки

Подробный формат файла который кладёт шаг `contract_spec` в `docs/features-spec/<feature>.md`.

**Принцип.** Спека = **срез финального состояния** фичи. Контракт для разработчиков, читатель не должен знать что было до. Без истории, без обоснований процесса, без деталей реализации.

### Общая структура

```markdown
# <Feature Name>

## Бизнес-описание
## User Stories
## State
## UI Messages
## IO
### Effects
### Subscribers
## UseCase
## Тестовые сценарии  (опционально)
```

---

### Раздел 1. Заголовок

- Формат: `# <Feature Name>` — один заголовок уровня H1 в самом начале файла
- Имя — descriptive, на русском, без префиксов проекта
- Примеры: `# Список словарей`, `# Карточка слова`, `# Чат-квиз`
- **Не должно быть:** технических имён модулей (`# DictionaryList`, `# WordCardScreen`), кодовых префиксов (`# IS476 — ...`)

---

### Раздел 2. Бизнес-описание

- Объём: 1–2 абзаца
- Содержит:
  - Что фича делает с точки зрения пользователя
  - Зачем нужна (бизнес-ценность)
- **Не должно быть:**
  - Технических деталей (TEA, Redux, mate framework, sealed class)
  - Упоминаний слоёв (State / Msg / Effect / UseCase)
  - Истории изменений («раньше было X, теперь Y»)
  - Списка изменений в этой задаче

---

### Раздел 3. User Stories

- Список stories в каноничном формате:
  > Как **[роль]**, я хочу **[действие]**, чтобы **[результат]**
- Покрытие всех значимых сценариев фичи — без искусственного ограничения количества. Если фича сложная и stories много — норма. Если короткая — норма
- **Не должно быть:**
  - Предположений о реализации («чтобы reducer обработал…»)
  - Технических ограничений в роли action/результата

---

### Раздел 4. State

Состоит из трёх подразделов:

#### 4.1. Структура

**Сводный data class** (наглядный артефакт — для быстрого обзора):

```kotlin
data class FeatureState(
    val items: List<Item>? = null,
    val isLoading: Boolean = false,
    val selectedIds: Set<Long> = emptySet(),
    val dialog: DialogState = DialogState(),
)

data class DialogState(
    val isOpen: Boolean = false,
    val title: String = "",
)
```

**Общая структурная пометка** (один раз для всего data class):

> **Структура:** product (data class с независимыми полями). Алгебра нетривиальных полей — в их описаниях ниже (раздел «Почему»).

Это **общая пометка для всех тривиальных полей** — не повторяется per поле. Тривиальный product (boolean флаги, обычные коллекции, nested data class) обходится без обоснования алгебры.

**«Почему» для поля обязательно если** (whitelist триггеров):
- (a) выбран не-default тип — sealed вместо product, Set вместо Map/List, nullable как ось различения, generic с обёрткой
- (b) edit-сценарий — пара `originalX` / `editedX`
- (c) async-данные не дефолтным паттерном — sealed `Loadable<T>` вместо `T? + isLoading: Boolean`
- (d) поле выглядит как product, но семантически sum (или наоборот) — есть ловушка ложного «или»
- (e) изменение существующего поля (rename, тип, default, удаление)

Для тривиальных полей вне whitelist'а — без «Почему».

**Описание полей** — per поле:

##### `items: List<Item>? = null`
- **Что:** основной список данных фичи, отображается на главном экране.
- **Почему `List<Item>?` (nullable):** различает «ещё не загружали» (`null`) и «загружали, пусто» (`emptyList()`). При `null` — loading placeholder; при `emptyList()` — empty state с подсказкой добавить.

##### `isLoading: Boolean = false`
- **Что:** идёт ли в данный момент фоновая загрузка данных.
- **Почему отдельный `Boolean`, а не sealed state `Loading`:** loading **ортогонален** items (refresh-кейс: данные есть + грузим обновление). Sealed `Loading | Loaded` дал бы 2 варианта, но потерял бы кейс «retain previous data во время refresh». Простой Boolean покрывает 2 оси с product-моделью.

##### `selectedIds: Set<Long> = emptySet()`
- **Что:** id выбранных элементов в action mode (множественный выбор для удаления/обработки).
- **Почему `Set<Long>`, а не `Map<Long, Boolean>`:** Set даёт ровно 2 состояния на id (`есть`/`нет`). `Map<Long, Boolean>` дал бы 3 (`true`/`false`/нет ключа) — лишний вариант, провоцирует баги «false vs missing key».
- **Почему не sealed `ActionMode { Inactive; Active(ids) }`:** при `Inactive` ids=emptySet норма. Sealed разделил бы 2 класса для одной семантической оси.

##### `dialog: DialogState = DialogState()`
- **Что:** состояние диалога (открыт ли, заголовок).
- (тривиальный nested data class — без «Почему»)

#### 4.2. Computed properties

**Сводный code block** (наглядный артефакт):

```kotlin
val FeatureState.hasNoData: Boolean
    get() = items.isNullOrEmpty()

val FeatureState.canSubmit: Boolean
    get() = selectedIds.isNotEmpty() && !isLoading
```

**Описание computed** — per свойство, с обоснованием:

##### `hasNoData: Boolean`
- **Что:** удобный флаг «нет данных для отображения».
- **Почему computed, а не поле в State:** значение полностью выводится из `items`. Хранить отдельно — риск рассинхрона (reducer обязан помнить пересчитывать в каждом Msg меняющем items).
- **Где читается:** в Composable для выбора `EmptyState` placeholder.

##### `canSubmit: Boolean`
- **Что:** разрешено ли нажимать кнопку Submit.
- **Почему computed:** выводится из двух полей. Хранение увеличило бы поверхность State без необходимости.
- **Где читается:** в Composable (`enabled = state.canSubmit`) и в reducer (при `Msg.Submit` — guard).

#### 4.3. Инварианты

Список constraints между полями:

- Если `isLoading == true` И `items != null` — отображаются данные + индикатор обновления (refresh-кейс)
- Если `selectedIds.isNotEmpty()` — экран в action mode (action mode выводится как computed по этому полю)
- Не может быть одновременно `dialog.isOpen == true` И `isLoading == true` (если бизнес-правило этого требует)

**Не должно быть в разделе State:**
- Extension-функции мутации (`show*`, `hide*`, `update*`) — это детали реализации reducer'а
- Удалённые поля при переделке (срез финального состояния, не история)
- Развёрнутая ADT-проверка с подсчётом вариантов и анализом ловушек — обоснования per поле/computed достаточно (терсно, не аудит процесса)

---

### Раздел 5. UI Messages

Состоит из двух подразделов:

#### 5.1. Структура

**Сводный sealed interface** (наглядный артефакт):

```kotlin
sealed interface Msg {
    // ─── Действия пользователя ───
    data object Submit : Msg
    data class UpdateName(val value: String) : Msg
    data class SelectItem(val id: Long) : Msg

    // ─── Навигация ───
    data object RequestBack : Msg
    data class OpenDetails(val id: Long) : Msg

    // ─── Результаты данных ───
    data class Loaded(val items: List<Item>) : Msg
    data object NotFound : Msg
    data class LoadFailed(val cause: Throwable) : Msg

    // ─── UI feedback ───
    data class ShowError(val message: String) : Msg

    // ─── Переключатели ───
    data object EarliestOn : Msg
    data object EarliestOff : Msg

    // ─── No-op ───
    data object Empty : Msg
}
```

Категории (применять те которые нужны):
- Действия пользователя
- Навигация
- Результаты данных (из шага 3 `contract_io` — Datasource Msg, return от effects)
- UI feedback
- Переключатели (toggle)
- No-op

**Примечание:** в финальной спеке (этот раздел) присутствуют **все** категории Msg, включая «Результаты данных» из contract_io. Шаг 2 (`contract_ui_msg`) описывает только UI-стороны (actions/navigation/UI feedback/toggles/no-op). Datasource Msg добавляются на шаге 3.

**Общая структурная пометка** (один раз для всего sealed interface Msg):

> **Структура:** sum (sealed interface). Каждый Msg = вариант события. `data object` = безпараметрический вариант, `data class` = с payload. Алгебра нетривиальных Msg — в их «Почему» ниже.

Не повторять «Алгебра: sum» / «Алгебра: product» для каждого Msg — это шум. Тривиальный выбор (data object для безпараметрического, data class с одним полем-id или одним-простым-payload) — без «Почему».

**«Почему» для Msg обязательно если** (whitelist триггеров):
- (a) payload — не одно простое поле и не id (несколько полей, или объект не-primitive)
- (b) одно действие split на несколько Msg (`On`/`Off` вместо `Toggle(Boolean)`)
- (c) RequestX вместо DoX (reducer-controlled паттерн — пользователь сигнализирует intent, reducer решает что делать)
- (d) изменение существующего Msg (rename, изменение payload, удаление)

Для тривиальных Msg вне whitelist'а — без «Почему».

**Описание сообщений** — per Msg:

##### `Msg.Submit` (data object)
- **Что:** пользователь нажал кнопку "Отправить" для финализации.
- (тривиальный data object — без «Почему»)

##### `Msg.UpdateName(value: String)` (data class)
- **Что:** пользователь меняет значение поля имени.
- **Почему `value: String`, а не вся domain-модель:** payload = только изменяемое поле, не вся модель. UI не пересылает то что не меняет.

##### `Msg.SelectItem(id: Long)` (data class)
- **Что:** пользователь выбрал элемент списка (тап / long-press).
- **Почему `id: Long`, а не `Item`:** UI оперирует id, не доменной моделью. State содержит items, reducer находит item по id если нужен.

##### `Msg.RequestBack` (data object)
- **Что:** запрос на закрытие экрана (BackHandler или кнопка Back).
- **Почему отдельный `Msg.RequestBack`, а не сразу `NavigationEffect.Back` из UI:** reducer контролирует логику — может решить **не закрывать** (показать диалог подтверждения, проверить unsaved changes). UI диспатчит intent, не command.

##### `Msg.OpenDetails(id: Long)` (data class)
- **Что:** пользователь хочет открыть детальный экран для элемента.
- (тривиальный data class с id — без «Почему»)

##### `Msg.Loaded(items: List<Item>)` (data class)
- **Что:** результат загрузки от `DatasourceEffect.LoadList`.
- **Почему не отдельный `Msg.LoadingComplete` + State pull из репозитория:** прямой payload проще и atomarno — reducer не делает асинхронные fetches.

##### `Msg.NotFound` (data object)
- **Что:** результат от `DatasourceEffect.LoadItem(id)` когда item не найден.
- **Почему отдельный `Msg.NotFound`, а не `Msg.Loaded(null)`:** «не найдено» — отдельный сценарий с другой reducer-логикой (показать error UI). Различимое event.

##### `Msg.LoadFailed(cause: Throwable)` (data class)
- **Что:** загрузка упала с исключением (network, db, parsing).
- **Почему `cause: Throwable`:** для логирования / диагностики. UI обычно показывает generic «ошибка загрузки», но cause сохраняется для analytics.

##### `Msg.ShowError(message: String)` (data class)
- **Что:** UI feedback — показать ошибку (snackbar/toast). Может быть как Msg для retain через config change.
- (тривиальный data class — без «Почему»)

##### `Msg.EarliestOn` / `Msg.EarliestOff` (data object × 2)
- **Что:** toggle переключателя «сначала редкие».
- **Почему два `data object`, а не один `Msg.EarliestToggle(on: Boolean)`:** явные события легче читать и тестировать. `On/Off` — каждое явно интенциональное действие пользователя.

##### `Msg.Empty` (data object)
- **Что:** no-op для effects без полезного результата (save без return data).
- (тривиальный — без «Почему»)

#### 5.2. Reducer-логика per Msg

Список Msg с описанием reducer-поведения. Формат — bullet-list.

**State changes описываются СЕМАНТИЧЕСКИ** — что меняется в state на уровне смысла, не через имена extension-функций (`state.startSubmit()`) или развёрнутые `state.copy(...)`. Реализация (через extension или inline copy) — деталь шага `implement`, не спеки.

- **`Msg.Submit`**
  - Trigger: пользователь нажал кнопку "Отправить"
  - State changes: начинается submit — устанавливается флаг загрузки, сбрасывается предыдущая ошибка
  - Effects: `DatasourceEffect.Save`
  - Guard: только если `canSubmit == true`, иначе ничего не меняется

- **`Msg.UpdateName(value)`**
  - Trigger: ввод в поле имени
  - State changes: обновляется поле имени значением из payload
  - Effects: —

- **`Msg.Loaded(items)`**
  - Trigger: return от `DatasourceEffect.LoadList`
  - State changes: записываются загруженные items, сбрасывается флаг загрузки
  - Effects: —

- **`Msg.NotFound`**
  - Trigger: return от `DatasourceEffect.LoadItem` если null
  - State changes: устанавливается состояние «не найдено», сбрасывается флаг загрузки
  - Effects: —

(И так далее — для каждого Msg отдельный пункт с Trigger / State changes / Effects / Guard если есть.)

**Принцип «семантика, не реализация»:**
- ✅ «начинается submit — устанавливается флаг загрузки, сбрасывается ошибка» — что
- ❌ `state.startSubmit()` — отсылка к функции которой в спеке нет
- ❌ `state.copy(isLoading=true, error=null, validationErrors=emptyList())` — детали реализации, спека не должна меняться при рефакторинге copy

**Не должно быть в разделе UI Messages:**
- Имена extension-функций мутации (`startSubmit`, `clearForm`) — это детали реализации
- Развёрнутые `state.copy(...)` — нагружают спеку техническими деталями
- Конкретные тестовые данные / магические значения
- Reducer-реализация (`internal class XxxReducer`)

### Раздел 6. IO

Состоит из трёх обязательных подразделов + одного опционального.

#### 6.1. Effects

По трём категориям. Если категория пустая — раздел опускается.

##### Datasource Effects

**Сводный sealed interface** (наглядный артефакт):

```kotlin
sealed interface DatasourceEffect : Effect {
    data class LoadItem(val id: Long) : DatasourceEffect
    data class Save(val item: Item) : DatasourceEffect
    data class Delete(val id: Long) : DatasourceEffect
}
```

**Per effect — описание (целостный блок, цепочка не разрывается):**

- **Source** — какие Msg порождают (`Msg.Submit`, `Msg.Delete(id)`)
- **Action** — что делает (`useCase.saveItem(item)`)
- **Return Msg** — что возвращает (`Msg.Saved` или `Msg.Empty` для fire-and-forget)
- **Reducer-логика для Return Msg** — семантически описать что меняется в state (см. §5.2 принцип: семантика, не реализация). Если Return Msg тот же что уже описан в §5.2 — ссылка «см. §5.2», без дублирования
- **Edge cases** — поведение при failure / timeout / not found / concurrent calls
- **Почему** — **только если выбор нетривиален**. Триггеры: (a) выбран не-default паттерн (например `Msg.Empty` для return, atomarно склеенный load+filter+map); (b) есть >1 разумная альтернатива (inline в reducer vs отдельный effect); (c) изменение существующего контракта. Для тривиальных effects — не писать.

##### Navigation Effects

**Сводный sealed interface:**

```kotlin
sealed interface FeatureNavigationEffect : NavigationEffect {
    data class OpenDetails(val id: Long) : FeatureNavigationEffect
    data object ExitApp : FeatureNavigationEffect  // только в root-экранах
}
```

**Per effect — описание:**

- **Source** — какие Msg порождают
- **Navigator method** — какой метод Navigator'а вызывается
- **Почему** — **только если выбор нетривиален**. Триггеры: (a) per-screen sealed вместо базового `NavigationEffect` — если есть root-only действия (`ExitApp`); (b) cross-graph навигация (tab → root); (c) изменение существующего навигационного контракта.

##### UI Effects

**Сводный sealed interface:**

```kotlin
sealed interface UiEffect : Effect {
    data class ShowSnackbar(val message: String) : UiEffect
}
```

**Per effect — описание:**

- **Source** — какие Msg порождают
- **UI Action** — конкретное UI-действие (snackbar / toast / vibration)
- **Pattern** — через State (`UiMsg.Snackbar` → `snackbarState`) или side-channel (Channel/SharedFlow)
- **Почему** — **только если выбор нетривиален**. Триггеры: (a) выбор side-channel vs State не очевиден из контекста (borderline между long-lived и one-shot); (b) borderline между UiEffect и Datasource (например логирование).

#### 6.2. Subscribers (FlowHandlers)

Per subscriber:

- **`DictionaryFlowHandler`**
  - **Source:** `useCase.flowCurrentDict()`
  - **Emit Msg:** `Msg.DictionarySelected(dict: Dictionary?)`
  - **Operators:** `collectLatest`
  - **First emit special:** запускает `DatasourceEffect.LoadList`
  - **Lifecycle:** подписка от `subscribe()` до `dispose()`
  - **Почему** — **только если выбор нетривиален**. Триггеры: (a) обоснование бизнес-инварианта (зачем реактивная подписка нужна вообще); (b) выбор `collectLatest` vs `collect` (атомарность vs полный поток); (c) разделение first emit vs последующие (Ready/Updated паттерн); (d) изменение существующего subscriber.

Если subscribers нет — раздел `### Subscribers` оставляется пустым с пометкой «нет».

#### 6.3. Edge cases (опционально)

Список known edge cases которые повторяются между несколькими effects:

- **Race condition при двойном тапе на Submit** → reducer игнорит второй Msg.Submit пока `isLoading == true`
- **Concurrent flowCurrentDict + initialLoad** → `collectLatest` отменяет предыдущую загрузку

### Раздел 7. UseCase

**Сводный interface** (наглядный артефакт):

```kotlin
interface FeatureUseCase {

    // ─── Команды (используются в Datasource Effects) ───

    suspend fun loadItem(id: Long): Item?
    suspend fun saveItem(item: Item)
    suspend fun deleteItem(id: Long)

    // ─── Реактивные источники (используются в Subscribers) ───

    fun flowCurrentDict(): Flow<Dictionary?>
    fun flowItems(): Flow<List<Item>>
}
```

**Описание методов** — per метод:
- **Сигнатура** (suspend / Flow)
- **Что делает** — one-line описание
- **Потребитель** — Effect или Subscriber
- **Почему** — **только если выбор нетривиален**. Триггеры: (a) выбран не-default тип возврата (`Result<T>` вместо nullable, throw вместо null); (b) возврат feature-обёртки вместо domain-типа (нестандартно — обычно domain); (c) метод склеен с другим (нарушение single responsibility — должно быть осознанным); (d) изменение существующего метода. Для тривиальных методов (suspend для команд / Flow для подписок, параметры по id, return domain-типа) — не писать.

### Чего в спеке быть НЕ должно

- **Развёрнутый ADT-аудит процесса** (полное прохождение правил моделирования, разбор всех ловушек, шаг-за-шагом подсчёт вариантов) — это **процесс** контракта. Per-поле обоснования в State (Что / Почему / Алгебра) — нормально и нужно, но без углубления в полную хронику решений.
- Детали реализации: классы reducer'а, пакеты, imports, конкретные имена handler'ов
- История изменений (было / стало, удалено / добавлено)
- Артефакты других шагов (`design_tree`, `implement` детали)
- Конкретные ID, payload'ы в качестве примеров «магическими» значениями (если только не часть тест-сценария)
- Reducer-реализация (`internal class XxxReducer`)
- Edge case разборы из `contract_io` с длинным анализом — только итоговое поведение
