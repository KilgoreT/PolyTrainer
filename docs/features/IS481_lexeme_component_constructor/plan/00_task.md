## Задача

Превратить набор компонентов лексемы из **жёстко зашитого** (translation / definition, в планах example / note) в **конструктор**. Пользователь сам:

1. Создаёт «тип компонента» — произвольное имя (например `my-note`, `origin`, `pronunciation`) + выбранный **шаблон** из набора.
2. Использует этот тип при добавлении компонента в любую лексему.

Доступные шаблоны (стартовый минимум):
- `text` — короткий однострочный текст.
- `long-text` — многострочный текст.
- `image` — изображение.

Список шаблонов расширяемый — новые добавляются отдельной задачей.

## Что меняется

1. **Новая сущность** `ComponentType`: имя + шаблон + флаг `built-in` (для translation / definition).
2. **UI управления типами:** создание / переименование / удаление пользовательских типов. Built-in типы защищены от удаления и переименования.
3. **UI добавления компонента в лексему:** выбор из доступных типов вместо хардкода translation / definition.
4. **Миграция данных:** существующие translation и definition мигрируют в built-in типы конструктора (без потери данных).
5. **Возможно** — пересмотр сущности `Lexeme` и связанной DB-схемы: вместо отдельных полей translation / definition — коллекция компонентов с ссылкой на `ComponentType`.

## Что НЕ делаем (пока)

- **Шаринг типов между пользователями / устройствами** — только локально, привязка к локальной DB.
- **Расширение набора шаблонов** — стартовый минимум `text` / `long-text` / `image`. Добавление новых шаблонов (audio / video / link / table) — за рамками.
- **Сложные валидации шаблонов** (regex, длина, etc.) — стартово только тип данных.
- **Совместная редактируемость / sync** — out of scope.

- Возможно отдельный модуль управления типами (или встроенный в настройки словаря).
- `core/ui` — возможно новые примитивы для шаблонов image / long-text если ещё нет.
