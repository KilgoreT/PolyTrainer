## ⚠️ Несоответствия экрана механике редьюсера (найдено по коду)

1. **Строка компонента упрощена в макете.** Текущий `PerDictRowWidget` рисует чипы: **global** (`components_chip_global`), **одно/много** (`components_chip_multi`), **имена словарей** (`dictionaryNames`) + чип шаблона. Макет 5027:1534 рисует ТОЛЬКО чип шаблона («Текст»), название, счётчик и иконку типа — **бейджей global/multiple/scope НЕТ**. Стейт эти данные несёт (`isGlobal`, `isMultiple`) и они смысловые (в per-dict экране виден и global-компонент, и «Несколько»). Вопрос: убираем бейджи совсем (теряем различение global/scope/multiple в списке) или дизайнер их упустил? → Open question, на triage.

2. **Rename-диалог — мёртвый путь.** `Msg.OpenRenameDialog` + `RenameComponentDialog` существуют и в per_dict, и в manager, reducer их обрабатывает, экран монтирует `renameDialog` — но **`OpenRenameDialog` не вызывается ни из одного UI** (edit-кнопка карандаша шлёт `OpenEditDialog` → полный edit-диалог с name+template+multiple). Rename — legacy до phase-2 edit. Макет тоже показывает единый диалог (name+тип+чекбокс), отдельного «переименовать» нет. → Кандидат на удаление (Backlog / отдельная чистка), в редизайне просто не стилизуем.

3. **Общие виджеты — двойной потребитель.** Экран собран из `:modules:widget:component_widgets` (CreateComponentDialog, EditComponentDialog, DeleteComponentConfirmDialog, RenameComponentDialog, PerDictRowWidget, CreateComponentFab, ComponentsEmptyStateWidget, BlueAssistChip). **Эти же виджеты использует второй экран — Manager (`components_manager`).** Рестайл затрагивает ОБА экрана — в отличие от прошлых фич, здесь общие виджеты и есть предмет редизайна (у формы словаря / карточки слова мы, наоборот, общие не трогали). Надо решить: рестайлим общие (оба экрана меняются согласованно — вероятно ОК, оба — «компоненты») либо разводим. → scope_analysis.

4. **Empty-state** в макете не нарисован (список непустой). В коде есть `ComponentsEmptyStateWidget` (headline «В этом словаре только перевод» + CTA). Стилизуем по системе.

5. **Диалог создания в per-dict prod** — `CreateComponentDialog` с `hostVariant=PerDict` скрывает scope-picker (scope hardcoded в reducer). Макет диалога scope-picker и не рисует — совпадает. У Manager-варианта диалог показывает блок «Охват» + чипы словарей (multi-select) — макета на это НЕТ, но Manager не в скоупе этой папки (отдельный экран). → зафиксировать: рестайлим общий диалог, но per-dict-специфику (скрытый scope) не ломаем; Manager-специфику (scope-picker) визуально причёсываем по тем же токенам без макета.

## Решения по открытым вопросам (закрыты пользователем 2026-07-15)

1. **Бейджи global/«Несколько»/словари в строке — УБИРАЕМ** (пока не показываем: терминология названий не финализирована пользователем). Строка = иконка типа + название + «Значений: N» + чип шаблона + edit/delete. Данные (`isGlobal`/`isMultiple`) в стейте остаются, просто не рендерятся.
2. **Rename-диалог — УДАЛЯЕМ В ЭТОЙ ФИЧЕ** (мёртвый путь: Edit его поглотил, из UI не вызывается). Выпилить: `Msg.OpenRenameDialog` + `RenameDialogState` + ветку reducer + `RenameComponentDialog`-виджет + монтирование и связанный стейт в ОБОИХ экранах (per_dict + manager) + затронутые тесты. Не Backlog — часть фичи.
3. **Общие `component_widgets` рестайлим согласованно — ОБА экрана** (per-dict + Manager) получают новый вид. Оба про «компоненты», рассинхрон нелогичен. Manager-специфику (scope-picker, multi-select словарей) причёсываем по тем же токенам (макета на неё нет).
4. **Шрифт — тема приложения** (постоянное правило пользователя).
5. **Фон — `#FCFCFA`** (как форма словаря), не лавандовый. Три зоны приложения — три оттенка, осознанно.
