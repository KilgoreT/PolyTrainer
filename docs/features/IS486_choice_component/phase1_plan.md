## 1. Тест-дизайн

Принцип: ядро логики — **чистые функции** (unit без Room): планировщик каскада и валидаторы получают граф на вход и возвращают решение. Room-слой — тонкое исполнение, instrumented-тесты.

### Зона A — домен (unit, `:modules:domain:lexeme`, JUnit4, синхронные чистые функции)

- **A1. `ComponentTemplate.CHOICE`**: `fromKey("choice")` → CHOICE; `fields` → пустой список; TEXT/IMAGE не изменились.
- **A2.** — схлопнут с D1 (сериализация живёт в мапперах; `ChoiceValues` — голый data class).
- **A3. Валидатор создания/редактирования (§7.1)**, чистая функция → outcome:
  - цель-лексема + core=true → ok;
  - цель-лексема + core=false → ok;
  - цель-компонент + core=false → ok;
  - цель-компонент + core=true → `InvalidTarget`;
  - цель-опция (живая) + core=false → ok;
  - цель-опция + core=true → `InvalidTarget`;
  - цель — removed-компонент / removed-опция → `InvalidTarget`;
  - обе ссылки заполнены → `InvalidTarget`;
  - **кросс-словарная цель** (компонент/опция чужого словаря) → отказ (§7.7);
  - CHOICE + isMultiple=true при создании → `MultiForbiddenForChoice`;
  - **edit-путь**: включение isMultiple на существующем CHOICE → `MultiForbiddenForChoice`;
  - CHOICE с пустым набором опций → **валиден на уровне домена/данных** (решение 2026-07-17); запрет «создать чойс без вариантов» — UI-валидация формы конструктора (фаза 3); следствие для фазы 2 — guard пустого пикера значения в карточке.
- **A4. Ацикличность (§8)** — подъём по ссылкам **независимо от removed-статуса** (спека §8, правка 2026-07-17):
  - цепочка A→B→лексема, назначаем C→A → ok;
  - самоссылка A→A → `CycleDetected`;
  - назначаем A→C, где C в поддереве A, глубина ≥3 → `CycleDetected`;
  - цикл через **опционное звено в середине** цепочки (нормализация на каждом шаге подъёма) → `CycleDetected`;
  - перепривязка к собственной опции (цикл длины 1 через нормализацию) → `CycleDetected`;
  - **кольцо через мёртвое (removed) звено** → `CycleDetected`;
  - мёртвая цепочка без кольца → ok (мёртвость ≠ цикл).
- **A5. Degraded-предикат**: цель-компонент жив / removed; цель-опция жива / removed; цель-лексема всегда жива.
- **A6. Инвариант последнего ядра (§7.8)**, все три пути:
  - два включённых ядра, disable одного → ok;
  - одно включённое ядро: disable → `LastEnabledCore`; soft-delete → `LastEnabledCore`; перепривязка на не-лексему → `LastEnabledCore`;
  - создание нового ядра — всегда ok.
- **A7. Уникальность «(ключ, словарь)»**: тот же ключ + тот же словарь → отказ; тот же ключ + другой словарь → ok.

### Зона B — миграция 12→13 (androidTest, образец `MigrationFrom11to12`)

Решение 2026-07-17: fail-restart-механика (хук `failAfterStep`, тесты сбоя на каждом шаге) — **вырезана** как перебор для одной установки с бэкапом; Room гонит миграцию в транзакции, недокат на диск не попадает.

Фикстура основная (v12): словари D1, D2; глобальная builtin-строка translation; кастомы: «Definition» (значения только в лексемах D1 без перевода), «Synonyms» (в лексемах D1 с переводом), «Mixed» (значения и в лексемах с переводом, и без — в D2), «EmptyType» (без значений, D2); лексемы с переводами в обоих словарях; квиз-конфиг D1 с ref `{"type":"builtin","key":"translation"}`.

- **B1. Схема**: таблица `component_options` с ожидаемыми колонками; 4 новые колонки types + `option_id` values; UNIQUE(`system_key`) отсутствует; новые индексы FK на месте (sqlite_master/PRAGMA — паттерн Case M/P образца); сгенерировать `13.json` в schemas.
- **B2. Размножение builtin**: по строке «Перевод» на D1 и D2 (свой dictionary_id, core=true, enabled=true, цель-лексема); глобальной строки нет.
- **B3. Перевязка значений**: значение перевода лексемы D1 → строка перевода D1 (не D2); сумма значений до == после.
- **B4. Seed «Части речи»**: по строке на словарь (CHOICE, не-ядро, цель-лексема, enabled=true); у каждой 6 опций (существительное, глагол, прилагательное, наречие, предлог, фраза) с устойчивыми position.
- **B5. Backfill кастомов**: «Definition» → core=true, цель-лексема; «Mixed» → core=true (правило «хоть одно значение в лексеме без перевода»); «Synonyms» → core=false, `depends_on_type_id` = перевод **D1** (не D2 — кастом второго словаря проверяет отсутствие кросс-перевязки); «EmptyType» → зависимость от перевода D2; `enabled=true` у всех.
- **B6. Defensive-глобалы** (упрощён): фикстура с кастомом `dictionary_id IS NULL` → миграция бросает исключение с внятным текстом; тест — простой `assertThrows` на прогоне миграции с плохой фикстурой. Ценность: если предположение «глобалов нет» неверно — громкий отказ вместо тихой порчи, восстановление из бэкапа.
- **B7.** — вырезан (fail-restart-механика, см. преамбулу зоны).
- **B8. Сохранность несвязанного**: words/lexemes/hints/quiz-configs не изменились (row-count + spot-check).
- **B9. Вырожденная БД — 0 словарей**: глобальная translation-строка удаляется (размножать некуда; builtin теперь пословарные, сид создаст при первом словаре); миграция проходит.
- **B10. Словарь без лексем**: перевязка — no-op, seed builtin + опций всё равно происходит.

### Зона C — data-слой после миграции (androidTest)

- **C0. Harness (новая работа, отдельный пункт волны):** androidTest-инфраструктуры для Room-стека нет — поднять `RoomApiTestHarness`: `Room.inMemoryDatabaseBuilder` + `setDriver(BundledSQLiteDriver())` (зеркало RoomModule), ручная сборка Api-impl (конструкторы принимают DAO + Database + logger, Dagger не нужен), FakeLogger.
- **C1. `getBySystemKeyForDictionary`**: возвращает строку своего словаря; (key, словарь-без-строки) → null.
- **C2. Создание лексемы с переводом** в D1 и D2 → значение цепляется к переводу своего словаря.
- **C3. Restore-путь (undo) лексемы** в D1 и D2 → восстановленное значение перевода ссылается на строку своего словаря.
- **C4. Резолв builtin quiz-ref**: конфиг словаря D1 с ref `{"type":"builtin","key":"translation"}` резолвится в перевод D1 (не D2).
- **C5. Seed в `addDictionary`**: новый словарь → перевод (ядро) + часть речи (6 опций) в одной транзакции. Разделение (решение 2026-07-17): **состав** builtin-набора — константы домена (`:modules:domain:lexeme`, рядом с `BuiltInComponent`); **исполнение** — атомарно в `DictionaryApiImpl` + `useWriterConnection { immediateTransaction }` (cross-DAO seed в default-методе WordDao не живёт). Повторный вызов → без дублей (`WHERE NOT EXISTS`); единственная точка создания словаря = `addDictionary` (Dump — мёртвое legacy, обходных путей нет).
- **C6. Legacy-create дефолты**: `createUserDefinedComponent` → цель-лексема, core=true, enabled=true.
- **C7. Удаление словаря после миграции**: FK CASCADE — его строки types/options/values исчезли, второй словарь нетронут (прецедент IS476).

### Зона E — каскад-модуль

- **E1. Планировщик (unit, живёт в `:modules:domain:lexeme` — Dependency Rule чист)** — вход: граф типов/опций + значения + событие; выход: план сбросов / dry-run:
  - удаление компонента → значения его и всего поддерева;
  - деактивация опции → только поддерево опции (зависимые от компонента не тронуты);
  - **комбинированный К4+К5**: у опции и выбранные значения, и зависимый компонент с потомками → обе ветви в одном плане;
  - мульти-цель: смерть не-последнего значения → пустой план; последнего → сброс поддерева;
  - цепочка глубины 3: деактивация A каскадит через B к C;
  - dry-run == плану исполнения.
- **E2. Исполнитель (androidTest, на harness C0)** — фаза 1 подключает только softDelete-путь; исполнитель принимает DAO через конструктор:
  - softDelete компонента с детьми: значения детей soft-deleted, дети живы и вычисляемо degraded;
  - транзакционность: DAO-декоратор с инжектированным сбоем в середине (тестовый код, продуктовый хук не нужен) → полный откат;
  - существующий cleanup (quiz-configs + prefs) не сломан;
  - restore цели → дети снова активны без записи.
