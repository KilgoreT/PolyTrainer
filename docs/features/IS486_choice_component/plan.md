# IS486 — План имплементации по фазам

Скоуп: вся [spec.md](spec.md). Процесс каждой фазы — ТДД: тест-дизайн → красные тесты → имплементация до зелёного → девайс-проверка.
Порядок фаз 2/3 изменён после ревью плана (2026-07-17): карточка слова раньше конструктора — окно «конструктор создаёт то, что карточка не умеет» исчезает by construction.

## Фаза 1 — фундамент данных (без видимых изменений на существующих экранах)

Граница честно проходит по data-слою: миграция + обязательные правки CoreDbApiImpl/DAO/seed. «Без изменений UI» = без видимых изменений поведения существующих экранов.

- Миграция 12→13: `component_options`; `core` / `enabled` / `depends_on_type_id` / `depends_on_option_id` в types; `option_id` в values; дроп UNIQUE(`system_key`); индексы на новые FK.
- Размножение builtin по словарям + перевязка значений; seed «Части речи» с опциями (набор из §4 — решён, потребляется здесь); backfill (ядра, зависимости от перевода, legacy-глобалы — defensive-fail).
- **Seed-точка** переезжает в транзакцию `addDictionary` (+ обходные пути restore/import); идемпотентность `WHERE NOT EXISTS (ключ, словарь)`.
- **`getBySystemKey` → `getBySystemKeyForDictionary`** во всех call-site (addLexemeWithBuiltInComponent, addLexemeWithComponents, restore-путь); deprecated-шимы `addLexemeWithTranslation`/`updateLexemeTranslation` — выпилить.
- **Legacy-create дефолты**: `createUserDefinedComponent` пишет цель = лексема, `core = true`.
- Домен: `CHOICE`, `ChoiceValues`, `DependencyTarget`, `ComponentOption`, новые поля `ComponentType`, `PART_OF_SPEECH`, новые ветки outcomes (`InvalidTarget`, `MultiForbiddenForChoice`, `LastEnabledCore`, `EnableOutcome`).
- Каскад-модуль: единый, рекурсивный, транзакционный, dry-run; в фазе 1 подключён **только** к существующему softDelete-пути. Value-триггеры — строго фаза 2.
- Инварианты: создание (§7.1), уникальность «(ключ, словарь)», ацикличность (§8), минимум одно включённое ядро (§7.8 — все три пути).
- Времянки до следующих фаз: WordCard-выдача типов фильтрует `template != CHOICE` (до фазы 2); квиз-пикер компонентов исключает CHOICE (постоянно для v1, §9.6); снапшоты конструктора фильтруют builtin (до фазы 3).
- Manager-код: минимальная адаптация к новым веткам outcomes (компилируется, без новой функциональности).
- **Критерий готовности (чек-лист):** migration-тесты 12→13 зелёные; на девайсе: создать словарь → слово → лексему; квиз работает; undo-restore лексемы работает; существующие экраны без видимых изменений.

## Фаза 2 — карточка слова

- Правило участия в чипсах: в черновике — только ядра; зависимые по мере активации целей (явные поля state, заполняются в reducer).
- CHOICE-пикер значения (builtin «Часть речи» доступна с фазы 1); typed-контент `ComponentValueState` (Text | Choice).
- Подключение value-триггеров каскада: смена/удаление значения → сброс поддерева деактивировавшегося предка; мульти-цель гаснет последним значением.
- Деградация лексемы в черновик вместо авто-удаления (`LexemeCascadeRemoved` → draft).
- Снятие времянки-фильтра CHOICE из фазы 1.
- **Критерий готовности:** end-to-end на девайсе — выбрать часть речи у лексемы; сменить значение → каскад; удалить последнее ядро-значение → лексема в черновике, не удалена.

## Фаза 3 — конструктор

- Создание CHOICE с опциями; пикер цели; CRUD опций (К1–К5, комбинированный impact); builtin-строки в списке с рубильником (`setEnabled`); degraded/disabled-метки; перепривязка (ацикличность, impact + сброс); расширенный `DeletionImpact`.
- Резолв name-based quiz-refs единой функцией (§9.6; миграция refs на id — Backlog).
- **Критерий готовности:** создать кастомный CHOICE с опциями и зависимостью; выключить builtin у словаря (кроме последнего ядра — отказ); перепривязка с impact работает; degraded виден и лечится.
