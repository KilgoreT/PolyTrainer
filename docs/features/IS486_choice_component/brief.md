# IS486 — Шаблон компонента «Выбор» и зависимости между компонентами

Ишус: https://github.com/KilgoreT/PolyTrainer/issues/486
Статус: концепция, UI не обсуждаем. Цель этапа — понять, как шаблон будет работать, и как его назвать.

## Что хочет пользователь

### 1. Новый шаблон компонента: выбор одного значения из списка

- Семантика радиобаттонов: компонент один, возможных значений много, выбирается ровно одно.
- У компонента: **название** + **набор возможных значений**. Набор — часть определения компонента (задаётся при создании), а не значения лексемы.
- Значение лексемы — **одно** выбранное из набора.
- Мотивирующий пример: компонент «Часть речи» со значениями существительное/глагол/…; при выборе в карточке лексемы отображается конкретная часть речи.

### 2. Зависимости между компонентами

- Компонент Б доступен, только когда компонент А имеет конкретное значение.
- Пример (испанский): «Род» работает только для существительных. Выбрал «Часть речи» = существительное → стал доступен выбор «Род».

## Как это ложится на текущую модель

- Сейчас `ComponentTemplate` (modules/domain/lexeme): `TEXT`, `IMAGE` (IMAGE временно задизейблен в UI). Комментарий в enum: «composite — добавляются в будущих фичах» — это первый структурный шаблон.
- Отличие от TEXT/IMAGE: у шаблона появляется **схема на уровне ComponentType** (список опций), а не только тип primitive-значения. Т.е. `fields`-модель придётся расширять: значение — ссылка на опцию, опции — данные типа.
- `isMultiple` для этого шаблона по определению = false (одно значение). Уточнить: запрещаем ли multi для него жёстко.

## Решения по концептуальным вопросам (2026-07-15)

1. **Название шаблона: `CHOICE`** («Выбор»). Решено.
2. **Модель данных опций — РЕШЕНО (2026-07-16):** сборный айдишник не понадобился. Опции — своя таблица `component_options`; зависимость — XOR из двух FK (`depends_on_type_id` | `depends_on_option_id`); значение CHOICE у лексемы — `option_id` в `component_values` (сама опция и есть значение). Схема — в [hierarchy.md](hierarchy.md).
3. **Модель зависимости:** несколько условий у одного компонента — **нет** (ровно одно). Многоуровневые цепочки (А→Б→В) — **да**. Циклы — **запрет**: граф зависимостей — лес (не более одной парент-ссылки у компонента); валидация назначения парента подъёмом по цепочке предков — алгоритм в [hierarchy.md](hierarchy.md), «Модель зависимости».
4. **Каскад при смене значения: полный сброс.** Был глагол с заполненными зависимыми опциями, стало существительное — все зависимые значения обнуляются, никаких «скрытых сохранений».
5. **Эволюция набора опций:** отложено — решается по ходу архитектуры, наблюдаем как остальные решения на это влияют.
6. **Охват:** склоняемся к «глобалов не будет, только пословарные» — но это ограничение **UI**; в коде возможность global-охвата оставить.
7. **Квизы:** в v1 — нет, но задел держать в голове (скорее всего стоит делать позже).

## Договорённости

- **Зависимость — минимальная модель.** Пользователь видит зависимость как простое поле у компонента, задаваемое при создании (ссылка на парент). Уточнение: нужна пара `parentTypeId` + `parentOptionId` («доступен при значении X»), иначе пример «Род только для существительных» не выражается. Если зависимость задаётся только при создании и только на существующий компонент — циклы невозможны by construction (валидация не нужна).
- Остаточная сложность — поведенческая, не storage: каскад при смене значения парента, удаление парента/опции при живых детях (расширение существующего impact-превью), запрет смены зависимости после создания.
- **Мульти запрещён только шаблону CHOICE** (решение 2026-07-16): правило привязано к шаблону, не к зависимости — структура CHOICE подразумевает единственное значение («двух одинаковых вершин не может быть»). Зависимые компоненты других шаблонов мульти могут («Пример» от перевода). Запрет мягкий: валидация/UI, схема хранения single не зашивает — разрешить multiple позже можно без миграции.
- **Общая иерархия — итоговая модель (2026-07-17):** каждый компонент зависит ровно от одного узла — лексема | компонент | опция. Прежний «корневой» заменён флагом **«ядро» (core)** у зависимых от лексемы: ядро оформляет лексему (критерий черновика) и доступно в черновике. Builtin теперь два: Перевод (ядро) и Часть речи (CHOICE, не ядро). Симметричные тренировки — особенность перевода, не ядерности. **Источник истины по механике — [hierarchy.md](hierarchy.md)**; ранние формулировки ниже (parentTypeId, «корневой», isRoot) — история решений, устарели.
