# IS486 — Иерархия компонентов: механика

> **SUPERSEDED (2026-07-17): источник истины — [spec.md](spec.md).** Этот документ — рабочая история концепции; после него приняты решения, которых здесь нет: пословарные builtin (перевод + часть речи seed'ятся на словарь), дизейбл ядра разрешён (запрещён только последнему включённому ядру словаря), глобальный охват убран из продукта (Manager-экран — потенциальная фича, spec §20).

Документ механики иерархии зависимостей компонентов. UI не обсуждаем — только модель и поведение.
Статус: концепция прошла ревью (2 агента, 29 находок) и батч решений 2026-07-16/17. Открытых вопросов — два (в конце).

## Единая модель: узел и зависимость

**Каждый компонент зависит ровно от одного узла.** Узел — одно из трёх:

- **лексема** — супер-корень леса; все вершины дерева — её дети;
- **компонент** — «доступен, когда у парента есть значение» (Пример → Перевод);
- **опция** CHOICE-компонента — «доступен при конкретном значении» (Род → «существительное»).

В домене: `DependencyTarget = Lexeme | Component(typeId) | Option(optionId)` (sealed).

**Закон доступности один:** компонент доступен для заполнения ⇔ его цель активна.

- лексема активна ⇔ оформлена (не черновик);
- компонент активен ⇔ у лексемы есть его значение;
- опция активна ⇔ у лексемы выбрана именно она.

### Ядро (core)

**Термин: «ядро» / `core` (2026-07-17).** Флаг, возможный только у зависимых от лексемы.

- **Ядро** доступно всегда, включая черновик, и **оформляет** лексему: критерий черновика — «лексема оформлена ⇔ заполнено хотя бы одно ядро». Это единственное исключение из закона доступности — по определению: ядра — то, из чего лексема строится, остальное — то, что на неё навешивается.
- **Не-ядро, зависимое от лексемы** (Часть речи) — доступно только у оформленной лексемы, черновик им не закрыть.

## Builtin-набор (2026-07-17)

Официальных builtin два; новый словарь рождается с ними:

- **Перевод** — ядро, зависит от лексемы. Особый для тренировок: тренится симметрично (слово-перевод и перевод-слово). Ядерность других компонентов симметричных тренировок НЕ даёт.
- **Часть речи** — CHOICE, зависит от лексемы, НЕ ядро.

## Состояния компонента

- **Активный** — цель жива, `enabled = true`.
- **Degraded** — **вычисляемое** состояние (решение 2026-07-17): цель указывает на удалённый (soft-delete) узел. Ссылки при деградации **не зануляются** — restore цели автоматически оживляет зависимых. В списке компонентов помечается, в лексемах не участвует.
- **Disabled** — `enabled = false`, ручной рубильник. Значения хранятся, но не показываются; ветка вниз выпадает из участия; в новых лексемах не указать. Включил обратно — всё на месте.
  - **Ядро выключить нельзя** (иначе лексемы скопом теряют опору); всё не-ядро — можно, включая builtin «Часть речи». Защита привязана к core-флагу, не к builtin-ности.

**Правило участия** — компонент участвует в лексеме, когда:

- `enabled` у него и у всех предков;
- цепочка предков доходит до лексемы по живым (не удалённым) узлам;
- для не-ядер дополнительно: лексема оформлена.

Дети degraded/disabled-компонента выпадают автоматически — отдельный флаг им не нужен.

**Инвариант создания** (всё — UseCase-валидация; Room не умеет CHECK — дом-стиль M13):

- при создании компонент обязан иметь цель: лексема | компонент | опция;
- создать сразу degraded нельзя;
- `core = true` валиден только при цели-лексеме;
- обе ссылки заполнены одновременно — нелегально.

## Хранение

Таблица `component_types` (существующая) — 4 новых поля:

- `core` — флаг ядра. Валиден только при цели-лексеме.
- `enabled` — рубильник, дефолт true.
- `depends_on_type_id` — ссылка на компонент-цель.
- `depends_on_option_id` — ссылка на опцию-цель.

Состояния ссылок: обе пустые → цель «лексема»; заполнена одна → зависимый от компонента / от опции; обе → нелегально.

Таблица `component_options` (новая):

- `id` — адрес опции; на него ссылаются зависимости и значения лексем.
- `component_type_id` — какому CHOICE-компоненту принадлежит.
- `label` — текст опции; переименование меняет только его.
- `position` — порядок в списке.
- `created_at` / `updated_at` / `removed_at` — в стиле остальных таблиц.

Таблица `component_values` (существующая) — 1 новое поле:

- `option_id` — выбранная опция. Строка values — факт «компонент у лексемы заполнен»; текст/картинка — JSON `value`, choice — `option_id` (JSON — пустой envelope, колонка NOT NULL). Сама опция и есть значение, копий лейбла нет. Impact К4 — один COUNT по `option_id`.

Перевод и Часть речи — обычные строки `component_types` (`system_key`), зависимость от них — обычные ссылки.

## Каскады

**Закон:** цель деактивировалась → значения зависимых сбрасываются, по всей цепочке вниз. Никаких скрытых сохранений.

- **Сброс = soft delete** (`removed_at` в values, решение М3 2026-07-17) — в стиле остальных таблиц; для приложения значение исчезло.
- Смена значения парента (существительное → глагол) → сброс значений зависимых.
- Удаление значения последнего заполненного ядра → **лексема деградирует в черновик**; значения зависимых от лексемы не-ядер (Часть речи) сбрасываются по общему закону.
- Инвариант значений: пустых значений не существует — значение либо живое и непустое, либо удалено.

## Эволюция набора опций (К1–К5)

- **К1. Добавление** — свободно, опция сразу доступна.
- **К2. Переименование** — свободно; id устойчив, лейбл меняется везде задним числом.
- **К3. Удаление без ссылок** — свободно.
- **К4. Удаление с живыми значениями** — impact-превью + каскадный сброс («выбрана у N лексем...»). Каскад вниз по цепочке.
- **К5. Удаление опции с зависимым компонентом** — опция soft-delete; зависимый компонент становится degraded **вычисляемо** (ссылка живёт и указывает на удалённую опцию). Restore опции — компонент ожил. Перепривязка degraded — свободная, с чистого листа. Деградация никогда не меняет доступность молча (отвергнуты «в корень» и «доступен при любом значении»).

## Перепривязка (парент редактируемый)

- Разрешена всегда, с валидацией ацикличности.
- **Живой компонент:** impact-превью + полный сброс значений (своих и потомков) + перепривязка. Любая смена условия = сброс, без «умных» исключений (решение М2 2026-07-17: сначала настрой иерархию, потом заполняй значения).
- **Degraded:** просто перепривязка, подтверждение не нужно — терять нечего.

### Ацикличность

**Инвариант.** У каждого компонента одна цель, лес растёт из лексемы. Цикл возможен только назначением цели, делающим компонент предком самого себя.

**Алгоритм проверки** (назначаем компоненту C цель P):

1. Нормализация: если P — опция, `P := компонент-владелец опции` (ловит и перепривязку к собственной опции).
2. `X := P`; пока X — компонент: если `X == C` → отказ (цикл); иначе `X :=` цель X (для опции-цели — её владелец).
3. Дошли до лексемы, не встретив C → допустимо.

Сложность O(глубины цепочки). При создании проверка не нужна — у нового компонента нет потомков.

## Квизы и участие

- **Квиз-конфиги чистятся только при удалении компонента** (как сейчас). Disabled/degraded — обратимые состояния: конфиги живут, квиз **скипает** выпавшие компоненты на лету — сборка квиза фильтрует правилом участия, пикер конфигов выпавших не предлагает (решение Б 2026-07-17).
- CHOICE в v1 в квизах не участвует; задел держать.

## Охват и словари

- UI: только пословарные компоненты; возможность global остаётся в коде.
- **Создание — только на один словарь** (решение Д 2026-07-17). Multi-dict путь в коде остаётся, из UI уйдёт; на стыке валидация: «несколько словарей + зависимость» не поддерживается.
- **Удаление словаря** — кастомные компоненты словаря умирают вместе с ним (решение Е 2026-07-17).
- Кросс-словарные и кросс-охватные зависимости запрещены на уровне домена: дерево всегда целиком внутри словаря.

## Миграция 12→13

- Схема: таблица `component_options` + `core`, `enabled`, `depends_on_type_id`, `depends_on_option_id` в types + `option_id` в values.
- **Backfill (решение В 2026-07-17):** Перевод — ядро. Кастомный компонент, имеющий хоть одно значение в лексеме без перевода (фактически используется сам — Definition), — ядро. Остальные кастомные — зависимость от перевода. `enabled = true` всем.
- Seed (fresh install и существующие словари): builtin «Часть речи» (CHOICE, зависит от лексемы, не ядро).
- Побочный эффект принят: пользователь один, иерархию донастроит руками перепривязкой.

## Влияние на существующую реализацию (из ревью совместимости, 2026-07-17)

- **Каскад-модуль** — новый, единый, рекурсивный, транзакционный; триггеры: смена/удаление значения парента, удаление опции, удаление компонента, перепривязка. Dry-run режим = impact-превью.
- **WordCard:** «удалён последний компонент → лексема удаляется» заменяется деградацией в черновик; `ComponentValueState` текстоцентричен — нужен typed-контент (Text | Choice) и `ChoiceValues` в `TemplateValues`; фильтрация чипсов по правилу участия — явными полями state (дом-правило: явные флаги, не вычисления в composable).
- **Impact-превью** расширяется: деградирующие потомки, счётчики значений по цепочке.
- **Квизы:** фильтр участия при сборке quiz item + в пикере; name-based `ComponentTypeRef` → резолв единой функцией (миграция на id — в Backlog).
- **Спеки:** component-constructor (CHOICE, цель-пикер, CRUD опций, расширенный impact, ветки EditOutcome, запрет мульти для CHOICE) и lexeme-domain (владелец lifecycle лексемы) — правки при написании единой спеки.

## Правила мульти

**Мульти запрещён только шаблону CHOICE** (2026-07-16): структура выбора одной вершины из набора подразумевает единственность. Зависимость мульти не запрещает («Пример» — мульти, зависит от перевода). Запрет мягкий: валидация/UI, схема хранения single не зашивает — разрешить multiple позже можно без миграции.
