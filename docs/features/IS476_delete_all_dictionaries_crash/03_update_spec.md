## 1. `dictionary-appbar.md`

### 1.1 Контракт `flowCurrentDict()` — non-null → nullable

#### Было

> ### Текущий словарь
>
> `flowCurrentDict()` — реактивный Flow, который эмитит при:
> 1. Смене ID текущего словаря (переключение между словарями через prefs)
> 2. Изменении данных самого текущего словаря в БД (например, смена флага, переименование)
>
> Реализация: комбинация Flow из SharedPreferences (ID текущего словаря) и Flow из Room (данные словаря).

Тип возврата (в зависимостях ниже не прописан явно, но в коде — `Flow<DictUiEntity>` non-null). Реализация (`DictionaryAppBarUseCaseImpl.flowCurrentDict`) бросает `DictionaryNotFoundException` при пустом списке словарей.

#### Стало

`flowCurrentDict()` возвращает `Flow<DictUiEntity?>`. Пустой список словарей — **валидное** доменное состояние, моделируется эмиссией `null`, а не исключением.

Контракт эмиссий:
1. Смене ID текущего словаря в prefs.
2. Изменении данных текущего словаря в БД (флаг, имя).
3. **Появлении / исчезновении словарей** в БД:
   - Список непустой, текущий ID найден → эмит `DictUiEntity` найденного словаря.
   - Список непустой, текущий ID не найден (orphaned pref) → эмит `DictUiEntity` первого словаря в списке (fallback).
   - **Список пустой** → эмит `null`.

Реализация — комбинация Flow из SharedPreferences (nullable `Long?`) и Flow из Room (`List<DictionaryItem>`). Внутри `combine`-лямбды **запрещено** бросать исключение для моделирования «нет словаря» — это антипаттерн в reactive-stream'е (см. `docs/guides/effect-handlers.md`: «ошибки реактивного источника обязан обрабатывать UseCase/FlowHandler, не пропуская исключения наружу»; `docs/guides/prefs-datastore.md`: «ЗАПРЕЩЕНО throw на null»).

Класс `DictionaryNotFoundException` (из `dictionarytab.deps`) больше **не используется** для этого сценария.

### 1.2 Поведение виджета при `currentDict == null`

#### Было

Спека не описывает поведение AppBar при отсутствии словарей. State уже допускает `currentDict: DictUiEntity?`, но что показывает виджет — не зафиксировано.

#### Стало

Новая секция «Пустой список словарей»:

- `state.currentDict == null` и `state.availableDictList.isEmpty()` — валидное состояние виджета.
- Поведение в этом состоянии: виджет **не рендерится** хост-экраном (см. `dictionary-list.md` — «AppBar не показывается на пустом списке»). На остальных табах (vocabulary/quiz/stats) при пустом списке хост-таб сам отвечает за пустой/редирект-стейт; AppBar в составе таба корректно отрабатывает `null` без креша.
- Если AppBar всё же рендерится с `currentDict == null` (например, гонка во время удаления):
  - Иконка флага — `FlagPlaceholderWidget` с пустой буквой (или скрыта).
  - Dropdown-меню — содержит только «Создать словарь» (список словарей пуст).
  - Никаких throw / no-op на тапы — `ChangeDict` недоступен, потому что список пуст.

### 1.3 Сообщения

#### Было

> `CurrentDict(current)` — Обновить текущий словарь

Параметр `current: DictUiEntity` non-null.

#### Стало

`CurrentDict(current: DictUiEntity?)` — пропускает `null` при пустом списке. Reducer кладёт `null` в `state.currentDict`.

Альтернатива (если решим разделять — см. effect-handlers guide «Паттерн: разделение первого emit»): два сообщения — `CurrentDict(current: DictUiEntity)` и `CurrentDictAbsent` (data object). Выбор оставить за `implement` шагом; в спеке зафиксировать оба варианта как допустимые, с предпочтением nullable-параметра ради меньшей поверхности изменений.

### 1.4 Затронутые секции спеки

- `## Состояние (State)` — уточнить, что `currentDict: DictUiEntity?` корректно представляет «нет словаря».
- `## Реактивность данных` → подсекция `### Текущий словарь` — переписать контракт целиком (см. 1.1).
- `## Dropdown-меню` — добавить пункт про поведение при пустом списке (только «Создать словарь»).
- `## Messages` — обновить тип параметра `CurrentDict` (см. 1.3).
- Новая подсекция `## Пустой список словарей` (или абзац в «Реактивность данных») — см. 1.2.

## 2. `dictionary-list.md`

### 2.1 Поведение `flowCurrentDict()` при удалении последнего словаря

#### Было

> 3. Если удалённый был текущим → useCase переключает на первый оставшийся
> 4. Если словарей не осталось → пустое состояние

Текст оставляет неопределённым, что происходит с `CURRENT_DICTIONARY_ID_LONG` и с подписками `flowCurrentDict()` на других экранах/виджетах. По коду — pref остаётся со старым (несуществующим) id, AppBar `flowCurrentDict()` бросает `DictionaryNotFoundException` (см. триаж/research).

#### Стало

Расширить шаги удаления:

> 3. Если удалённый был текущим:
>    - есть оставшиеся словари → `useCase` записывает в prefs id первого оставшегося (`setCurrentDictionary(remaining.id)`);
>    - оставшихся нет → `useCase` **очищает** pref `CURRENT_DICTIONARY_ID_LONG` (или явно фиксирует «оставить как orphaned»; реализация решает в шаге `implement`, спека требует чтобы все читатели pref'а были устойчивы к обоим вариантам).
> 4. Если словарей не осталось → пустое состояние (на этом экране) + все реактивные подписки `flowCurrentDict()` (AppBar / DictionaryTab) эмитят `null` вместо исключения. Никаких крашей.

Добавить инвариант (раздел «Удаление» или новый «Инварианты»):

> **Инвариант:** удаление любого словаря, включая последний, **не должно** приводить к необработанному исключению ни в одной активной подписке (`DictionaryAppBarUseCase.flowCurrentDict`, `DictionaryTabUseCase.flowCurrentDict/getCurrentDict`, `QuizChatUseCase.getCurrentDictionaryId`). Все читатели pref'а `CURRENT_DICTIONARY_ID_LONG` обязаны корректно обрабатывать ситуацию, когда pref указывает на несуществующий id или отсутствует вовсе. См. `docs/guides/prefs-datastore.md`.

### 2.2 User-journey: удаление всех словарей → exit → онбординг

#### Было

> ## Навигация назад
>
> Conditional в reducer:
> ```kotlin
> is Msg.RequestBack -> if (state.dictionaries.isEmpty()) {
>     state to setOf(ListNavigationEffect.ExitApp)
> } else {
>     state to setOf(NavigationEffect.Back)
> }
> ```
>
> - Пустой список → `ListNavigationEffect.ExitApp` → `ListNavigator.exit()` → `ListNavigatorImpl.onExit()` → `activity.finish()`
> - Есть словари → `NavigationEffect.Back` → `Navigator.back()` → `navController.popBackStack()`

Спека фиксирует механизм, но не связывает его с онбордингом (`DICTIONARY_SETUP`) и не объясняет, почему `ExitApp` — правильное поведение продукта, а не баг.

#### Стало

Переписать «Навигация назад» в раздел `## Навигация назад и user-journey пустого списка` (или добавить отдельный подраздел «User-journey: удаление всех словарей»):

> **Правило (бизнес-логика).** Conditional `RequestBack` — это не просто UX-удобство, а часть продуктового user-journey:
>
> 1. Пользователь удалил все словари → список пуст → AppBar скрыт, экран в пустом состоянии («Нет словарей / Создайте первый!»).
> 2. Системная back / тап «назад» в AppBar (пока он не скрыт) → `Msg.RequestBack` → reducer → `ListNavigationEffect.ExitApp` → `ListNavigator.exit()` → `activity.finish()`. Приложение закрыто.
> 3. **Следующий запуск** приложения — это эффективно онбординг создания словаря: `RootPoint.SPLASH` → `SplashScreen` → видит пустой список → `openDictionarySetup()` → форма создания первого словаря (см. `dictionary-create.md`, режим `DICTIONARY_SETUP`).
>
> Таким образом удаление последнего словаря не «закрывает дверь» пользователя из приложения навсегда — оно возвращает его в стартовую точку онбординга при следующем запуске. Это намеренная связка двух механизмов (`ExitApp` при пустом списке + `SplashScreen → openDictionarySetup()` при отсутствии словарей).
>
> См. также: `dictionary-create.md` — режим `DICTIONARY_SETUP` (форма без AppBar, после успешного создания → openMainScreen).

### 2.3 Orphaned `CURRENT_DICTIONARY_ID_LONG` — общий инвариант

#### Было

В спеке нет упоминания pref'а `CURRENT_DICTIONARY_ID_LONG` и его жизненного цикла.

#### Стало

Добавить (можно секцией «Инварианты pref'а текущего словаря» или абзацем в «Удаление»):

> **`CURRENT_DICTIONARY_ID_LONG` (pref).** Pref может быть:
> - не записан (первый запуск, очистка данных) — `null`;
> - содержать id существующего словаря — happy path;
> - содержать id **удалённого** словаря (orphaned reference) — допустимо как переходное состояние.
>
> Все читатели pref'а обязаны trapсформировать orphaned id в fallback:
> - `DictionaryAppBarUseCase.flowCurrentDict()` — fallback на первый словарь, при пустом списке → эмит `null`;
> - `DictionaryTabUseCase.flowCurrentDict() / getCurrentDict()` — то же поведение; при отсутствующем словаре таб переходит в пустое/редирект-состояние, **не бросая исключение**;
> - `QuizChatUseCase.getCurrentDictionaryId()` — не должен бросать `IllegalStateException`; ответственность за невозможность открыть квиз-чат без словаря лежит выше (gating на уровне навигации), либо метод возвращает nullable и вызывающая сторона корректно обрабатывает.
>
> Это часть единой системы — нарушение в любом из читателей приводит к крашу типа IS476.

### 2.4 Третий близнец дефекта — QuizChat

#### Было

Никакого упоминания.

#### Стало

Добавить (вместе с инвариантом из 2.3 или отдельной заметкой):

> **Системное ограничение (QuizChat).** `QuizChatUseCaseImpl.getCurrentDictionaryId()` исторически бросает `IllegalStateException("Dictionary not found")` при отсутствии словарей. Спека `dictionary-list.md` фиксирует это как **дефект архитектуры**: open-screen квиз-чата без словарей должен быть невозможен на уровне навигации (gating), либо контракт `getCurrentDictionaryId()` должен быть nullable и UI квиз-чата обязан показывать пустое/редирект-состояние. До отдельной спеки QuizChat поведение фиксируется здесь как общий инвариант: «удаление всех словарей не должно приводить к крашу ни в одном модуле, читающем `CURRENT_DICTIONARY_ID_LONG`».

## 6. Сводка по бизнес-правилам, которые фиксируем впервые

1. **Пустой список словарей — валидное доменное состояние**, не исключение. (`dictionary-appbar.md` + `dictionary-list.md`)
2. **`flowCurrentDict()` эмитит `null` при пустом списке**, а не throw. (`dictionary-appbar.md`)
3. **Orphaned `CURRENT_DICTIONARY_ID_LONG` — допустимое переходное состояние**, все читатели обязаны иметь fallback. (`dictionary-list.md`)
4. **Удаление всех словарей → exit → новый запуск = онбординг `DICTIONARY_SETUP`** — намеренная продуктовая связка двух механизмов. (`dictionary-list.md` + `dictionary-create.md`)
5. **QuizChat не должен крашиться при отсутствии словарей** — либо gating на навигации, либо nullable контракт; временно зафиксировано как общесистемное ограничение. (`dictionary-list.md`)
