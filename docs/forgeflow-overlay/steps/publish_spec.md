Финальная публикация проектной спеки и PUML-схем фичи.

## 1. Источники

- `contract_spec.output` — черновик спеки в feature dir (создан шагом `contract_spec`)
- `implement.output` — артефакт реализации (для опциональной корректировки спеки под итог реализации)
- Опционально — `.puml` файлы в feature dir (схемы, созданные на шагах `contract_state` / `design_tree`)

## 2. Публикация спеки в `spec_dir`

`spec_dir` определяется через `plan.context.workspace.spec_dir` (например `docs/features-spec/`).

### 2.1. Чтение черновика

Прочитай `contract_spec.output` (`contract_spec.md` в feature dir). Это **черновик спеки** в финальном формате (Бизнес-описание / User Stories / State / UI Messages / IO / UseCase / Тестовые сценарии — см. `FORGEFLOW_contract_design.md` § «Формат проектной спеки»).

### 2.2. Опциональная корректировка под `implement`

Прочитай `implement.output`. Если в реализации вылезли edge cases которые **меняют контракт** (новый Msg / Effect / поле State которого не было в контракте) — отрази их в спеке:

- Только **значимые** изменения контракта. Косметику (имя локальной переменной, форматирование) — игнорировать.
- **Раздел `## Корректировки от implement` обязателен** в `publish_spec.md` — даже если изменений нет (явно пишется «без изменений: спека опубликована as-is»). Это даёт ревьюеру явную ревью-точку: посмотреть на изменения целиком.
- Если контракт изменился — перечислить **каждое** изменение конкретно (новый Msg, переименованный метод UseCase, удалённое поле State) с указанием где в `implement.output` это обнаружено.
- Если изменения **значимые** (новая сущность, изменение публичного контракта) — это сигнал что контракт-блок недоработал. Reviewer проверит, рассмотрит на pause.

### 2.3. Имя файла спеки

Берётся из `contract_spec.output`. Если в черновике явное имя не указано — выбирается по правилу:
- Существующая спека для этой фичи → имя сохраняется
- Новая фича → имя соответствует модулю / экрану / функциональности (`dictionary-form.md`, `word-card.md`, snake-case)

### 2.4. Запись

Создай или перезапиши `spec_dir/<имя>.md` финальной версией спеки. Перезапись полная (спека = срез финального состояния «как есть после реализации этой задачи»).

**Что НЕ должно попасть в спеку** (см. `FORGEFLOW_contract_design.md` § «Чего в спеке быть НЕ должно»):
- Раздел «Удаляемые поля» / «Удаляемые messages» из артефактов контракт-блока
- Раздел «Корректировки от implement» — этот раздел живёт **только** в `publish_spec.md` (артефакт шага), не в проектной спеке
- Развёрнутый ADT-аудит
- Детали реализации (классы, пакеты)

## 3. PUML-схемы

Если в feature dir есть `.puml` файлы:

- Найди директорию хранения PUML-схем целевого модуля проекта (по конвенциям — найди существующие `.puml` в проекте, положи рядом)
- Имя файла — snake_case по экрану / модулю
- Если PUML с таким именем уже есть — перезапиши

Если в проекте PUML-схем не найдено вовсе — пропусти эту часть и зафиксируй это в `publish_spec.md`.

## 4. Output `publish_spec.md` — что записать

В feature dir создаётся `publish_spec.md` со следующим содержанием:

```markdown
# publish_spec

## Опубликовано
- `<spec_dir>/<имя>.md` — <создан / обновлён>
- Имя файла: `<имя>.md`
- Размер изменений (в строках): `~N`

## Корректировки от implement
- **Обязательный раздел**, никогда не опускается
- Если изменений нет — пишется «без изменений: спека опубликована as-is относительно `contract_spec.output`»
- Если есть — список конкретных изменений с указанием где в `implement.output` обнаружено

## PUML
- Скопировано: `<list of .puml files>`
- Целевая директория: `<путь>`
- Если PUML в проекте нет — отметить «PUML-схем в проекте не найдено, шаг пропущен»
```

## 5. Что НЕ делать

- Не писать новый код, ничего не править в `app/` / `modules/`
- Не менять контракт значимо без отметки в «Корректировки от implement» — если реализация раскрыла что контракт неполон, это **должно** быть видно в артефакте шага для ревью
- Не плодить спеки с разными именами для одной фичи — переиспользовать существующее имя файла
