Шаг Discovery infra sub-flow. Собрать факты о реальной инфраструктуре проекта (DI / build / manifest / proguard) — релевантные будущим изменениям инфры фичи.

Цель — чтобы следующий шаг (`infra_design_tree`) проектировал изменения **не теоретически, а на фактах**: какие DI-модули уже есть, как устроен build, какие правила ProGuard, какие версии плагинов/библиотек.

## Что делать

1. Прочитай `scope_analysis.output` — пойми что за фича (какой инфра-аспект затрагивается).
2. Грепай реальную инфраструктуру проекта. Какие именно сведения нужны — твоё решение. Кандидаты:
   - **DI-граф** — `app/di/module/*`: какие модули уже есть, как зарегистрированы зависимости, какие scope'ы используются (Dagger / Hilt / другое — что в проекте).
   - **Build-конфигурация** — `build.gradle.kts` (root + модульные): плагины, версии, BuildConfig-поля, флагованные сборки.
   - **AndroidManifest** — `AndroidManifest.xml`: permissions, services, providers, declared activities, intent filters.
   - **ProGuard / R8** — правила `*.pro`-файлов: какие классы keep-аются, какие правила для рефлексии.
   - **CI/CD** — `.github/workflows/`: какие jobs, какие триггеры, какие шаги.
   - **Mate framework / logger / navigation infra** — `modules/core/mate`, `modules/core/log`, `modules/core/navigation` (если такие есть): структура, контракты.
3. Если аналогичный модуль / правило / job существует — найди и сошлись на конкретные файлы и строки. Если не существует — зафиксируй явно «аналогов нет в проекте».
4. Запиши факты в `infra_walkthrough.md`.

## Принципы

- **Факты — не выводы.** Walkthrough не принимает дизайн-решений и не пишет конфигурацию. Только что есть в коде сейчас.
- **Ссылки `file:line` обязательны.** Не пересказывай — цитируй или ссылайся на конкретное место.
- **Не дублируй гайды.** Правила DI / ProGuard / build применяются на `infra_design_tree`, не здесь.
- **Если ничего подходящего нет — это тоже факт.** Не притягивай несуществующий аналог.

## Формат `infra_walkthrough.md`

Структура свободная — выбирай разделы по факту того что нашёл. В конце артефакта **обязательная** секция:

```markdown
## Вердикт

Аналог: <найден / не найден>
<краткое обоснование>
```

## Что НЕ делать

- Не писать код, не править файлы.
- Не принимать архитектурных решений.
- Не дублировать конвенции из гайдов проекта.
- Не задавать вопросов пользователю — если фактов мало, фиксируй что нашёл и переходи к вердикту.
