output_criteria:

Финальный шаг любого sub-flow. Создай `summary.md` с frontmatter (для master) и body (для downstream и аудита).

Прочитай **все** input-артефакты — это output'ы предыдущих шагов sub-flow. По ним пойми что было сделано и реши финальный статус.

## Решение по `status`

### `done`

- Все шаги sub-flow прошли успешно
- Реализация уложилась в scope без необходимости трогать другие слои
- Контракт sub-flow выполнен полностью

### `feedback_required`

Только если:
- В одном из шагов sub-flow обнаружено что **scope_analysis был неточен** — например design_tree наткнулся на необходимость менять контракт в другом слое (не отмеченном scope'ом)
- Невозможно завершить работу sub-flow в рамках текущего scope без нарушения архитектуры или дублирования с другим слоем

Если есть сомнения — выбирай `done`. Feedback loop запускает дорогой перезапуск, нельзя триггерить «на всякий случай».

## Формат файла

```markdown
---
status: done | feedback_required
reason: <строка>           # обязательно если status == feedback_required
suggested_changes:         # опционально, подсказки для нового scope_analysis
  <ключ>: <значение>
---

# Summary — <subflow_name>

## Что сделано

**Глубокое раскрытие** работы sub-flow (не просто список — детальное описание):
- Какие файлы созданы / изменены / удалены, с конкретными именами и краткой сутью изменения
- Какие контракты приняты (с сигнатурами)
- Какие тесты добавлены / прошли / падают и почему
- Какие лог-точки добавлены и в каком контексте срабатывают
- Какие миграции введены (для Data)

Этот раздел читается downstream'ом (`checklist_run`, sub-flow следующих итераций, master review). Скудное описание ломает цепочку — будь подробным.

## Ключевые решения

- Контракты, которые приняты (со ссылками на конкретные файлы артефактов)
- Архитектурные решения с обоснованием
- Что осталось вне scope этого sub-flow

## Артефакты

Список со ссылками на конкретные файлы артефактов sub-flow — для downstream и аудита.

## (Если status == feedback_required) Что упущено в scope

Конкретно: какой слой/контракт/аспект не покрыт scope_analysis, и почему без него sub-flow не закроется.
```

## Что НЕ делать (повторение для важности)

- Не пиши код, ничего не правь — твоя задача описать что было
- Не сокращай описание ради «лаконичности» — это рабочий artifact для самопроверки и cross-iteration audit, скудность вредит downstream'у

## Правила frontmatter

- `status: done` — нет дополнительных требований
- `status: feedback_required` — обязательно `reason` (строка), желательно `suggested_changes` (map)
- Если `status` отсутствует — runner трактует как `done` (см. dsl.md → Статусы шага)

## Что НЕ делать

- Не дублируй содержимое предыдущих артефактов целиком — только резюме + ссылки
- Не пиши новый код, ничего не правь
- Не принимай решений за master flow — твоя задача только описать что было и просигналить статус
