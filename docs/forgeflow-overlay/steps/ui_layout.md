Финальная UI-спека фичи. Готовится перед `design_tree` UI sub-flow и публикуется через `publish_ui` в раздел `## UI Layout` общей спеки фичи (`docs/features-spec/<feature>.md`).

Артефакт = **полное финальное состояние UI после реализации фичи** (snapshot), не diff. Источник истины для `design_tree` и `implement` шагов UI.

## Формат артефакта

Формат документа описан в отдельной спеке: **`docs/forgeflow-overlay/steps/ui_layout_format.md`**.

Спека описывает структуру секций, маркеры, шаблон буллетов, договорённости подачи (Карта экрана, Анализ виджетов, Удаляем с миграцией) и чек-лист дисциплины. Артефакт обязан ей соответствовать.

**Перед началом шага** — прочитать спеку формата целиком. При несоответствии формата артефакта спеке — переделывать, а не публиковать.

Если рядом со step-промптом лежат другие файлы с суффиксом `_format.md` — это спеки других форматов, для этого шага не применимы (читать только `ui_layout_format.md`).

## Контекст из plan.context

Из `plan.context` обязательно прочитать (если есть):

- **`figma_dump_path`** — относительный путь к `figma_dump.json` (например, `../figma_dump.json`). Это **единственный источник Figma-данных**. Парсить локально без MCP.
- **`out_of_scope`** — список виджетов вне скоупа фичи. Каждый помечается маркером **📌** в Карте и в Анализе с явной пометкой "OUT OF SCOPE". Любые отходы от Figma по ним — 🚨 в notes как DRIFT, но **не закрываются** в этой фиче.
- **`project_decisions`** — список зафиксированных решений-отходов от Figma. Каждый отход обязан попасть в 🚨 в notes соответствующего виджета с обоснованием из text.
- **`drift_rule`** — правило применения DRIFT. Распространяется на out_of_scope виджеты.

## ПРИОРИТЕТ ИСТОЧНИКОВ (жёсткое правило)

При конфликте между источниками **строго следовать порядку**:

1. **`plan.context.project_decisions` / `out_of_scope`** — **высший приоритет**. Если decision говорит "AddLexeme = FAB в Scaffold-слот" — артефакт фиксирует именно это, независимо от того что в Figma и что в коде сейчас.
2. **Figma dump** — источник целевого UI для всего что не покрыто project_decisions.
3. **Текущий код виджетов** — используется ТОЛЬКО для:
   - REUSE-данных виджетов которые НЕ меняются в фиче (размеры/цвета/типографика — забираются из кода).
   - Заполнения поля `type:` при расхождении Figma ↔ код (формат "X (Figma) → Y (project)").

**ЗАПРЕТ:** фиксировать в Карте / Анализе **текущее состояние кода** как "финал", если это противоречит project_decision из контекста. Артефакт = **target snapshot после реализации фичи**, не **as-is код**. Если код сейчас inline-button, а decision говорит FAB → пишем FAB.

**ЗАПРЕТ:** придумывать собственные DRIFT-обоснования ("кнопка не должна перекрывать контент при scroll" и т.п.). Все 🚨 DRIFT обязаны соответствовать конкретному project_decision из контекста (по `id`) или out_of_scope. Свободные обоснования "от себя" — нарушение протокола.

## ЗАПРЕТЫ при сборке артефакта

- НЕ читать никаких файлов `ui_layout_*.md` / `*_example.md` в `plan.dir` или соседних директориях. Артефакт собирается **с нуля** по format spec + figma_dump + context.
- НЕ копировать структуру и тексты из соседних артефактов — формат единый, диктуется спекой.
- Источники данных: figma_dump (через figma_dump_path), scope_analysis.output, business.output, текущий код виджетов (по 02_scope → затронутые файлы UI), plan.context. **Больше ничего.**

## Скоуп

Покрывать **только** то, что входит в фичу по `scope_analysis.output`. Если фича редактирует кнопку — раздел про эту кнопку + минимальный родительский блок если его layout меняется. **НЕ описывать весь экран целиком если фича трогает один блок.**

## Три источника одновременно

Все три читать обязательно (если присутствуют), не один из них:

### Источник 1 — Существующая спека

`docs/features-spec/<feature>.md` — если существует, прочитать раздел `## UI Layout` (если есть). Это baseline. Финальный артефакт = baseline + дельта от Figma/scope.

Если спеки нет — фича без публичной спеки, baseline пустой.

### Источник 2 — Существующий код виджетов

Все Compose-композиты в скоупе фичи (по `02_scope` → «затронутые файлы UI»). Извлечь:
- Размеры (`Modifier.size/height/width`).
- Отступы (`Modifier.padding`, `Arrangement.spacedBy`, `verticalArrangement`).
- Цвета (`MaterialTheme.colorScheme.X`, `me.apomazkin.theme.*Color`, hex-литералы).
- Типографика (`LexemeStyle.X`, `TextStyle(...)`).
- Иконки (`R.drawable.X`).

Это **текущее реальное состояние UI** — основа для всех виджетов которые остаются `[REUSE]`.

### Источник 3 — Figma dump (через шаг `figma_dump`)

**MCP-вызовы к Figma на этом шаге запрещены.** Единственный источник Figma-данных — артефакт `figma_dump.json` в корне фичи (`plan.dir/figma_dump.json`), создаваемый отдельным шагом `figma_dump`.

Контракт:
- Если `feature_has_figma=true` **и** `figma_dump.json` присутствует → парсить его локально. Извлекать только данные target frame'ов из скоупа фичи. Соседние frames / родителей / черновики — игнорировать.
- Если `feature_has_figma=true` **и** `figma_dump.json` отсутствует → **fail** с явной ошибкой "ожидался figma_dump.json в plan.dir, но не найден; проверь шаг figma_dump". Не пытаться восполнить через MCP.
- Если `feature_has_figma=false` → Figma не источник, опираться на код + старую спеку.

**ЖЕЛЕЗНОЕ ПРАВИЛО — читать ТОЛЬКО target node-id'ы из скоупа фичи.** В дампе есть весь файл Figma — соседние frames / черновики / варианты дизайна тоже там. Правильные — те на которые явная ссылка через `node-id` в `02_scope.md` / `00_task.md`. Не подниматься вверх по иерархии, не "догадываться".

### Действия

1. Загрузить `plan.dir/figma_dump.json` в память (через Read / jq / стримовый парсер — без MCP).
2. Извлечь данные для каждого target `node-id`: layout, размеры (px / dp), цвета (hex), типографика (font, size, weight, line-height), иерархия дочерних элементов, references на иконки.
3. Цвета из Figma маппить на существующие токены `MaterialTheme.colorScheme.X` или `me.apomazkin.theme.*Color`. Если точного токена нет — указать hex и пометить как кандидат на добавление в theme.
4. Типографику маппить на `me.apomazkin.theme.LexemeStyle.*` или `MaterialTheme.typography.*`. Расхождение — явные параметры (size/weight/font/lineHeight).
5. Импорт недостающих иконок — НЕ на этом шаге. Список недостающих фиксировать в артефакте как `TODO: импорт <iconName> из node <nodeId>` для отдельного шага импорта (или отдельного прогона `figma_dump` с `download_figma_images`).

## Логика сборки финального snapshot

1. Если есть **Figma** — это источник нового состояния UI (что должно быть после фичи).
2. Если есть **старая спека** — что было раньше; вычислить дельту с Figma; виджеты не затронутые фичей — переноси as-is из старой спеки.
3. Если есть только **код** (Figma и спеки нет) — финальный артефакт = `[REUSE]` для существующих виджетов + `[NEW]` designer-decisions с обоснованием на основе соседних виджетов / theme tokens.

**Дельты не отражать.** Артефакт описывает результат, не процесс изменения.

## Что НЕ делать

- НЕ описывать виджеты вне скоупа фичи.
- НЕ выдумывать размеры/цвета без источника. Каждый параметр — Figma node либо код либо старая спека.
- НЕ пропускать ассеты — иконка либо существует, либо импортирована.
- НЕ дублировать описание `[REUSE]` виджета — достаточно пути + параметров + изменений (если есть).
- НЕ писать pseudocode реализации — это работа `design_tree`/`implement`.
- НЕ отражать diff (что было → что станет) — только финальное состояние.
