Шаг Discovery data sub-flow. Собрать факты о реальной структуре data-слоя проекта — релевантные будущим изменениям схемы / DAO / миграций.

Цель — чтобы следующий шаг (`data_design_tree`) проектировал изменения **не теоретически, а на фактах**: какие entity / DAO / миграции уже есть, какая текущая версия схемы, какие конвенции FK / индексов используются.

## Что делать

1. Прочитай `scope_analysis.output` — пойми что за фича (какие сущности затрагиваются).
2. Грепай реальный data-слой проекта. Какие именно сведения нужны — твоё решение. Кандидаты:
   - **Entity / DAO** — `core-db-impl/`: какие @Entity-классы и @Dao-интерфейсы уже есть, какие методы у DAO релевантных сущностей.
   - **Схема и версия** — текущий `RoomDatabase.version`, файлы schemas (`schemas/<dbName>/<version>.json`).
   - **Существующие миграции** — `Migration` объекты: как написаны (drop/recreate vs alter), какие паттерны используются.
   - **Индексы и FK** — `@Index`, `@ForeignKey` декларации в Entity, конвенции (cascade / restrict / set_null).
   - **CoreDbApi** — публичный интерфейс data-слоя: сигнатуры методов, типы возврата (sealed result / nullable / list), atomicity-методы.
   - **Prefs / Library-обёртки** — релевантные части `prefs-datastore`, `core-db-api`, оболочки над сторонними библиотеками.
3. Если аналогичная сущность / миграция / DAO существует — найди и сошлись на конкретные файлы и строки. Если не существует — зафиксируй явно «аналогов нет в проекте».
4. Запиши факты в `data_walkthrough.md`.

## Принципы

- **Факты — не выводы.** Walkthrough не принимает дизайн-решений и не пишет миграции. Только что есть в коде сейчас.
- **Ссылки `file:line` обязательны.** Не пересказывай — цитируй или ссылайся на конкретное место.
- **Не дублируй гайды.** Правила миграций / DAO-конвенций применяются на `data_design_tree`, не здесь.
- **Если ничего подходящего нет — это тоже факт.** Не притягивай несуществующий аналог.

## Формат `data_walkthrough.md`

Структура свободная — выбирай разделы по факту того что нашёл. В конце артефакта **обязательная** секция:

```markdown
## Вердикт

Аналог: <найден / не найден>
<краткое обоснование>
```

## Что НЕ делать

- Не писать код, не править файлы.
- Не принимать архитектурных решений.
- Не дублировать конвенции из гайдов проекта.
- Не задавать вопросов пользователю — если фактов мало, фиксируй что нашёл и переходи к вердикту.
