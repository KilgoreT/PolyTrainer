Шаг 1 контрактного блока Business sub-flow. Описать State фичи: data class + nested + extensions + computed properties + инварианты.

См. полные правила в `docs/features/FORGEFLOW_contract_design.md` § 1 `contract_state` и § «Формат проектной спеки → Раздел 4. State».

## Проверка input

Перед работой проверь что `parent.scope_analysis.output` содержит:

1. **Поле `spec_filename`** — имя существующего файла спеки в `spec_dir/` (например `dictionary-list.md`) ИЛИ `null`. **Null валиден** — означает что спеки ещё нет (новая фича / legacy без миграции). Не error.
2. **Описание задачи** — бриф / краткое описание что нужно сделать. Без этого нет источника замысла — **error**: «scope_analysis не содержит описания задачи».
3. **Список затронутых модулей** — путь(и) к модулю фичи (`modules/screen/<feature>`, `modules/widget/<feature>` и т.п.). Без этого режим 3 (code-driven) не может найти `State.kt`.

**Минимальный source требуется**: хотя бы один из (спека / макет / код существующего модуля) должен быть доступен. Если **ничего нет** — спеки `null`, макета в брифе нет, и `State.kt` в указанном модуле отсутствует — **error со stacktrace**: «нет ни спеки, ни макета, ни кода фичи — невозможно построить контракт. Невозможный кейс для существующего проекта».

## Определение режима работы

Прочитай `parent.scope_analysis.output`. По наличию **источников** определи режим (приоритет: spec > mockup > code).

Доступные источники (по факту):
- **Спека:** `spec_filename != null` И файл существует в `spec_dir/<spec_filename>`
- **Макет:** в брифе задачи есть Figma URL / приложенный визуал / описание визуала
- **Код:** `State.kt` существует в модуле фичи (по `Список затронутых модулей`)

Режимы (комбинированно — может быть несколько источников):

- **Режим 1 (макет-driven):** макет есть (с спекой или без). Макет — приоритетный источник истины для замысла. Спека если есть — обновляется под макет.
- **Режим 2 (spec-driven):** макета нет, спека есть. Источник — спека `spec_dir/<spec_filename>` раздел `## State`. Дополнительно — сверка с кодом (см. ниже).
- **Режим 3 (code-driven):** макета нет, спеки нет, есть только код. Источник — текущий `State.kt` модуля фичи. Извлекаем «как должно быть» с очисткой косметики.

В output укажи выбранный режим в начале документа: `## Режим работы` с обоснованием (какие источники доступны и какой выбран primary).

## Что делать

1. **Извлечь structure State** в зависимости от режима (см. выше).
2. Описать в коде data class State + nested модели (для наглядности).
3. **Per поле** написать описание:
   - **Что** — semantic описание (что отражает)
   - **Почему** — обязательно при whitelist триггерах:
     - (a) не-default тип (sealed вместо product, Set вместо Map/List, nullable как ось)
     - (b) edit-сценарий (origin + edited)
     - (c) async-данные не дефолтным паттерном
     - (d) поле выглядит как product, но семантически sum (или наоборот)
     - (e) изменение существующего поля (rename / тип / default / удаление)
   - Для тривиальных полей вне whitelist — без «Почему».
4. Описать **computed properties** (extension `val` с getter) — derived поля. Per computed: Что выводит / Где читается / Почему computed (если выбор нетривиален).
5. **Подсчёт вариантов** — обязательная секция `## Подсчёт вариантов` в output. Два уровня (см. ниже раздел «Подсчёт вариантов — два уровня»).
6. Описать **синхронные структурные инварианты** — constraints между полями state (формат «если X то Y», «не может одновременно X и Y»). Большая часть инвариантов выводится из подсчёта вариантов (п.5). См. ниже раздел «Правила инвариантов» — обязательная проверка перед фиксацией.
7. Если переделка — раздел «Удаляемые поля / extensions / computed».

## Подсчёт вариантов — два уровня (обязательная секция `## Подсчёт вариантов`)

Цель: найти **невалидные комбинации** полей, которые тип допускает. Каждая невалидная комбинация — повод выбрать между:
- **Sum (sealed):** перепроектировать тип так чтобы невалидная комбинация была физически невозможна. Компилятор гарантирует.
- **Product + инвариант:** оставить product, закрыть инвариантом (см. раздел «Инварианты»). Reducer-дисциплина.

Sum строже но требует рефакторинга; product + инвариант дешевле но фрагильнее. Выбор — **явный и обоснованный** в артефакте.

### Уровень 1 — внутри каждого nested (data class / sealed) c >1 атрибутом

Для **каждого** nested-класса в State (включая top-level data class):
1. Перечисли все поля (тип, набор «осей» — для Boolean это 2 значения, для nullable T? — 2 категории `null / non-null`, для String — это «пустая» vs «непустая» как минимум, для enum — все варианты).
2. Перебери комбинации **категорий** (не реальных значений) — для каждого Boolean × Boolean = 4 комбинации, для Boolean × String = 4 (Boolean × {empty, non-empty}).
3. По каждой комбинации напиши вердикт: ✅ валидно / ❌ невалидно (с обоснованием почему). 
4. Для невалидных — выбери подход: sum (показать вариант рефакторинга) или инвариант (зафиксировать формулу в разделе «Инварианты»).

Пример для гипотетического `TextValueState(isEdit: Boolean, edited: String)`:

```
Уровень 1 — TextValueState:
  оси: isEdit ∈ {true, false}, edited ∈ {"", non-empty}
  
  (true, "")        — открыл edit-mode, ничего не ввёл              ✅
  (true, non-empty) — пользователь печатает                          ✅
  (false, "")       — view mode                                      ✅
  (false, non-empty) — закрыл edit-mode, но буфер не очищен         ❌ невалидно
  
  Выбор закрытия: 
  - Sum (sealed View/Edit) — корректно по типу, но затрагивает Reducer + Composable.
  - Product + инвариант 5 «isEdit==false ⇒ edited==""» — дешевле, оставляем.
  Решение: product + инвариант (обоснование: ...).
```

### Уровень 2 — top-level State (cross-field)

Перебор пар/троек **семантически связанных** полей разных nested. Полный декартов перебор не нужен — фокус на пары которые могут конфликтовать.

Группы для проверки:
- **Флаг + payload связанного поля** — `isCreatingX == true ∧ X != null` валидно? и обратно.
- **Несколько флагов** — `isLoading ∧ isCreatingX` — могут ли быть оба true одновременно?
- **Id + список** — `parentId == NOT_IN_DB ∧ childList.isNotEmpty()` — валидно?
- **Edit mode на разных уровнях** — может ли быть несколько вложенных edit-mode одновременно?
- **Async флаг + контент** — `isLoading == true ∧ data.isNotEmpty()` — валидно?

Per пара — вердикт ✅ / ❌ с обоснованием. Невалидные → инвариант на уровне top-level State.

Пример для гипотетического `WordCardState`:

```
Уровень 2 — top-level State:
  (wordState.id == NOT_IN_DB, isLoading):
    (true, true)   — слово не загружено, идёт загрузка     ✅
    (true, false)  — нет id, не грузится — состояние init  ✅ (default)
    (false, true)  — загружено + грузится                   ❌ невалидно
    (false, false) — стандартное after-load                 ✅
  → Инвариант 1.

  (wordState.id == NOT_IN_DB, lexemeList.isEmpty()):
    (true, false) — нет слова, но есть лексемы — невозможно ❌
  → Часть инварианта 3 (либо отдельный инвариант).

  (isLoading, isCreatingLexeme):
    могут ли быть оба true? — нет, isCreatingLexeme требует загруженного слова.
  → Опциональный инвариант (если другие гарантии слабые).
```

### Что не нужно перебирать

- Поля типа `Int` / `Long` / `String` сами по себе — бесконечно. Используй **категории осей** (пустой/непустой, NOT_IN_DB/положительный, и т.п.).
- Тривиальные пары без семантической связи (`topBarState.isMenuOpen × snackbarState.show`) — пропускай.

## Правила моделирования (обязательная проверка)

Для каждого поля прогнать:

1. **Product или sum?** (применяется только к nested) — независимые поля → product (`data class`); взаимоисключающие → sum (`sealed class`).
2. **Подсчёт вариантов** — найти лишние комбинации (невалидные, которые тип допускает) → переход к sum; недостающие → расширить тип.
3. **Ловушка ложного "или"** — соблазн расширить sealed новым вариантом? Проверь: это новое состояние или новая ось координат (placeholder при disabled)?
4. **Computed для derived** — если значение выводится из других полей — не хранить, оформлять extension `val`. Не плодить хранимые computed.
5. **Async-данные** — дефолт `data: T? + isLoading: Boolean`. Sealed `Loadable<T>` только для нетривиальных случаев (>2 content-состояний, pagination/streaming).
6. **Нормализация** (State как БД) — `Set<Id>` вместо `Map<Id, Boolean>`; не дублировать данные между списками.
7. **Dependency Rule** — feature-флаги не в domain model; оборачивать в feature-level wrapper.
8. **Editable отдельно от domain** — `originalX` (snapshot) + `editedX` (рабочая копия).

## Правила инвариантов

State-инвариант — синхронное структурное утверждение о форме state. НЕ описание reducer-поведения.

### Self-check (snapshot test) — обязателен перед фиксацией каждого инварианта

> Могу ли я проверить истинность этого инварианта **имея только snapshot state**, не зная истории reduce и какой Msg был последним?

- Проверяется по одному снимку state → **структурный инвариант** ✅
- Требует знать «какой Msg только что обработан», «что было до», «какой эффект в полёте» → **поведение reducer'а**, не state-инвариант. **Удалить.**

### Self-check (симметрия) — после фиксации основного списка инвариантов

> Есть ли в state другие поля / nested-структуры с похожей семантикой / похожим шейпом? Если да — какие у них инварианты? **Применимы ли симметричные?**

Особенно для пар-маркеров с типовой ролью:
- `origin / edited` (паттерн editable-text) — где есть, инвариант «вне edit-mode `edited` == ""» применяется ко всем уровням одинаково.
- `flag / payload` (паттерн «есть/нет данных») — где есть `is*Active: Boolean`, payload-поле должно быть консистентно с флагом.
- `isLoading / data` (паттерн async-загрузки) — «`isLoading=false ⇒ data != null` (если данные не nullable по семантике)» или зеркальное.
- `selected / list` — `selected != null ⇒ selected ∈ list` (referential integrity).

**Правило:** если один и тот же шейп встречается **в state ≥ 2 раз** (например `WordState.value/edited` и `TextValueState.origin/edited`), инварианты должны быть **зеркальные**. Если один уровень имеет правило, а другой — нет, это **пропущенный инвариант**.

Не делай отдельных гайдов под каждый локальный шейп — просто переноси правила симметрично.

### Запрещённые формулировки инвариантов

- «после `Msg.X` поле Y равно Z»
- «поле Y сбрасывается при `Msg.X` / при ошибке / при NavigateBack»
- «не имеет права писать в Y кроме как из reducer'а X»
- «после `RefreshA` поле A_origin синхронизируется с A_edited»
- «при коммите Y становится Z»

Все они описывают **что делает reducer**, а не **какая форма state**. Это работа `contract_ui_msg` / `contract_io`.

### Запрет раздела «Политики» / «Soft-rules» / «Reducer-правила»

Не создавай такого раздела в артефакте. Если в процессе работы ловишь себя на формулировке «при X происходит Y» — стоп: это reducer-поведение, оно живёт в `contract_ui_msg` или `contract_io`.

В contract_state — только синхронные структурные утверждения о форме state. Никаких политик, никаких soft-rules.

### Transient-окна между reduce()-шагами — норма, не нарушение

Между двумя последовательными `reduce()`-шагами state может проходить через transient-состояния — следствие async-эффектов (DB-ответ ещё не пришёл, эффект в полёте, ViewModel ждёт результат). **Это часть жизни state, не нарушение шейпа.**

Инвариант **не должен** закрывать такое окно — он описывает **каждый снимок**, а не «момент между».

Если хочется явно зафиксировать «определённая комбинация полей запрещена между двумя Msg» — это сигнал ввести **новое поле State** (например `*_pending: Boolean`, `*_inFlight: Long?`), а не добавлять инвариант. Поле делает transient-состояние **наблюдаемым**, и тогда уже про него можно писать структурные инварианты.

### Запрещённые формулировки в Per-field описании

Per field — только:
- **Что** отражает поле (semantic)
- **Почему** именно эта форма (whitelist триггеры)

**Запрещено** в описании поля:
- «сбрасывается при Msg X»
- «обновляется в reducer ветке Y»
- «принимает значение Z после Msg W»
- «должно быть N в момент перехода из A в B»

Связь поля с Msg-ами — работа `contract_ui_msg`, не contract_state.

### Запрет раздела «Кто пишет в поле»

Не создавай такого раздела. Связь поля и его «писателей» (в какой Msg-ветке reducer присваивает значение) — работа `contract_ui_msg` / `contract_io`.

В contract_state поле описывается **независимо от того кто и когда туда пишет** — только что это семантически и почему такая форма.

## Top-level State

**БЕЗУСЛОВНО `data class`**, даже если правило (1) указывает на sealed. Sealed внутри — для XOR-вложенностей. Это override над правилом моделирования №1 — для эргономики `copy()`.

## В режиме 2 (spec-driven) — обязательная сверка spec ↔ code

Технические декларации спеки сверяются с актуальным кодом:
- Grep `^data class\|^@Stable\|^@Immutable` по `State.kt` → выписать актуальную структуру State + nested data classes
- Grep `^fun State\.\|^fun .*State\.` по `State.kt` → выписать существующие extension-функции и computed properties
- Сравнить с разделом `## State` в спеке. Расхождения → раздел `## Расхождения spec ↔ code` в output

Раздел `## Расхождения spec ↔ code` обязателен **даже если расхождений нет** (явное «расхождений не обнаружено: проверено X полей State, Y computed»).

При расхождении — приоритет коду (технические декларации). Спека в этом прогоне правится под код. Бизнес-смысл — авторитет за спекой.

## Что НЕ делать

- Не описывать UI Msg / Effects / UseCase / Reducer-логику — это следующие шаги
- Не плодить «Алгебра: product» per поле — общая структурная пометка одна на data class
- Не углубляться в полную хронику ADT-решений — это процесс контракта, не output
- Не создавать раздел «Политики» / «Soft-rules» / «Reducer-правила» / «Кто пишет в поле» — см. раздел «Правила инвариантов»
- Не описывать «что происходит при Msg X» — это reducer-поведение, не state
- Не описывать «момент между двумя Msg» — инвариант описывает каждый снимок, не транзит

## Источник истины

- Режим 1 — макет (Figma) — точка истины для замысла
- Режим 2 — спека (для замысла) + код (для технических деклараций)
- Режим 3 — код (с очисткой косметики, «как должно быть»)
