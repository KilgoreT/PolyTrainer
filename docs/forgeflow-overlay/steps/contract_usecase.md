Шаг 4 контрактного блока. **Синтез** UseCase interface — boundary к Data слою. Методы вытекают из Effects + Subscribers (шаг 3).

См. полные правила в `docs/features/FORGEFLOW_contract_design.md` § 4 `contract_usecase` и § «Формат проектной спеки → Раздел 7. UseCase».

## Проверка input

1. **`contract_io.output` содержит `## Effects`** с тремя категориями (или хотя бы одной из Datasource/Navigation/Ui). Без неё нет команд для UseCase — **error**: «contract_io без Effects, UseCase не из чего синтезировать».
2. **`contract_io.output` содержит `## Subscribers (FlowHandlers)`** (может быть пустой с пометкой «нет»). Без раздела вообще — **error**: «contract_io не объявил Subscribers раздел, неполный артефакт».
3. **`contract_io.output` содержит `## Проверка реактивности`** с Y/N + обоснованиями. Без неё — **error**: «contract_io не прошёл инвентаризацию реактивности».
4. **Каждый Effect (где Action — вызов UseCase) и каждый Subscriber имеют ссылку на UseCase метод** (по имени). Если нет — **warning**: «UseCase метод не указан, придётся выводить из контекста».

## Источник

Берётся из режима:

- **Режим 1:** только шаг 3 (`contract_io.output`)
- **Режим 2:** шаг 3 + спека `## UseCase` раздел
- **Режим 3:** шаг 3 + существующий `UseCase.kt` interface

## Что делать

1. Описать `interface FeatureUseCase` в code-block.
2. Разбить методы по группам через секции-комментарии:
   - `// ─── Команды (используются в Datasource Effects) ───` — `suspend fun` методы
   - `// ─── Реактивные источники (используются в Subscribers) ───` — `fun ...: Flow<T>` методы
3. **Per метод** — описание:
   - **Сигнатура** — suspend / Flow, параметры, return type
   - **Что делает** — one-line описание
   - **Потребитель** — Effect или Subscriber (комментарий в code block: `// Effect: LoadWord`)
   - **Почему** — обязательно при whitelist триггерах:
     - (a) выбран не-default тип возврата (`Result<T>` вместо nullable, throw вместо null)
     - (b) возврат feature-обёртки вместо domain-типа (нестандартно)
     - (c) метод склеен с другим (нарушение SRP — должно быть осознанным)
     - (d) изменение существующего метода
   - Для тривиальных методов — без «Почему».
4. Если переделка — разделы:
   - `## Новые методы / изменения существующих` — что добавлено / изменено и почему
   - `## Удаляемые методы` — что убрано и почему

## Покрытие 1-к-1

**Каждый метод UseCase** должен соответствовать хотя бы одному Effect или Subscriber из шага 3 (комментарий `// Effect: X` или `// Subscriber: Y`). Метод без consumer'а — мёртвый код, удалить или объяснить.

**Каждый Effect / Subscriber** должен ссылаться на конкретный метод UseCase. Если метода ещё нет — явно отметить «нужен новый метод» с описанием в `## Новые методы`.

## Правила моделирования

1. **UseCase = публичный контракт Business ↔ Data.** Изменения требуют ревью. Минимизировать поверхность — не плодить методы «на всякий случай».
2. **Domain типы, не feature.** Методы возвращают/принимают domain типы (`Word`, `Dictionary`), не feature-обёртки (`WordUiItem`). Маппинг feature↔domain — в impl или feature-уровневом мапере.
3. **Параметры по доменным id, не UI-типы.** `deleteWord(wordId: Long)` — Long, не `WordUiItem`. UI-типы — feature сторона.
4. **Naming:**
   - `flow*()` — Flow-методы (`flowCurrentDict`, `flowDictionaryList`)
   - `get*` — синхронное получение (suspend, одноразовое)
   - `add*` / `update*` / `delete*` — CRUD
   - `search*` / `filter*` — поиск
   - `change*` — изменение состояния (например активный словарь)
5. **Suspend для команд, Flow для подписок.** Не возвращать `Flow<T>` из «загрузить один раз». Не делать suspend для реактивного источника.
6. **Single responsibility.** Не комбинировать «load + filter + map» в один метод — простой канал к Data.

## В режиме 2 — обязательная сверка spec ↔ code

- Read `UseCase.kt` interface → выписать все методы с сигнатурами
- Сравнить со спекой `## UseCase`. Расхождения (имена методов / сигнатуры / параметры / return types) → `## Расхождения spec ↔ code` (обязателен даже если расхождений нет).
- При расхождении в технических декларациях — приоритет коду.

## Что НЕ делать

- Не писать реализацию. Только интерфейс. Реализация — на шаге `implement`.
- Не плодить «Алгебра» / «Сигнатура» для каждого тривиального метода. «Почему» — только нетривиальное.
- Не привязывать UseCase методы к UI-типам.
