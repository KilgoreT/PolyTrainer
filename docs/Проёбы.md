## IS479 — wordcard lexeme inline

### Computed properties в composable вместо State

**Где:** `modules/screen/wordcard/src/main/java/me/apomazkin/wordcard/widget/lexeme/LexemeItemWidget.kt`

**Что:** sub-agent на impl написал прямо в composable:
```kotlin
val hasTranslationPlaceholder = state.translation == null
val hasDefinitionPlaceholder = state.definition == null
```

Это вычисляемые значения, derived from `LexemeState` — должны быть computed properties на самом `LexemeState`, не локальные `val` внутри composable.

**Почему:** sub-agent проигнорировал гайд `docs/guides/state-and-extensions.md` ("computed properties для derived полей"). Senior review это **отметил** на ит.1 (F005 minor `@Suppress("UNUSED_PARAMETER") order` — там тоже про code style), но конкретно про `has*Placeholder` упустил. Architect тоже не поймал. Conductor (я) — тем более.

**Как закрыто:** добавлены computed на `LexemeState`:
```kotlin
val hasTranslationPlaceholder: Boolean get() = translation == null
val hasDefinitionPlaceholder: Boolean get() = definition == null
```

Локальные `val` в composable заменены на `state.hasTranslationPlaceholder` / `state.hasDefinitionPlaceholder`.

**Что не повторять:** при ревью UI-композаблов искать `val xxx = state.* == null` / `state.* > 0` / любые тривиальные derived — это computed property, не локальная переменная.

### LexemeValueField view-mode как чёрный chip вместо обычного текста

**Где:** `modules/screen/wordcard/.../LexemeValueFieldWidget.kt`

**Что:** view-mode реализован как `M3 InputChip(selected=true, ...) + trailing ic_circle_delete`. Значение перевода/определения отображалось **обёрнутым в брендовый chip**, а не как обычный текст под chip-заголовком.

**Почему:** в `ui_layout.md` для `LexemeValueField` стоял тег `⟦chip+input⟧`. sub-agent на impl интерпретировал буквально — обернул value в chip. На самом деле тег означал двухсоставной виджет (chip-заголовок + input-поле), но **chip — это заголовок выше в MeaningField**, а не обёртка вокруг значения. По UX задумке value = обычный Text.

В design_tree это закрепилось как `view: M3 InputChip с trailing ic_circle_delete` — sub-agent зафиксировал собственную интерпретацию. Architect + senior не поймали потому что код был синтаксически корректен и тег `⟦chip+input⟧` для них не противоречил.

**Как закрыто:** view-mode заменён на обычный `Text(modifier.clickable(onClick = onOpenEditMode))`. Trailing remove icon убран — удаление субсущности через ✕ chip-заголовка (`SubentityChip` Active в `LexemeMeaningField`).

**Что не повторять:** теги в `ui_layout` (`⟦chip+input⟧`, `⟦container+text⟧` и т.п.) — это **семантическое описание содержимого**, не Compose-структура. Не интерпретировать буквально. Проверять реальную UX-задумку (chip = заголовок vs обёртка) до реализации.

### AddLexeme inline в Column вместо FAB в Scaffold-слоте

**Где:** `modules/screen/wordcard/.../AddLexemeWidget.kt`, `WordCardScreen.kt`

**Что:** sub-agent на impl ит.1 зафиксировал AddLexeme **как есть в текущем коде** (inline `M3 Button` в основном Column), не **как должно быть после фичи** (`M3 FloatingActionButton` icon-only в `Scaffold.floatingActionButton`).

Дополнительно: придумал ложное обоснование DRIFT'у "кнопка не должна перекрывать контент при scroll".

**Почему:** sub-agent смешал два источника — "текущий код" (REUSE) и "project_decisions" (цель). При конфликте взял текущий код и **рационализировал** через выдуманный DRIFT. Architect не сверил с `project_decisions` из `plan.context` — изолированно проверил артефакт.

**Как закрыто:** в `docs/forgeflow-overlay/steps/ui_layout.md` добавлен раздел "ПРИОРИТЕТ ИСТОЧНИКОВ" — при конфликте текущий код vs Figma vs project_decision побеждает **project_decision**. В `docs/forgeflow-overlay/agents/custom/architect.md` добавлен пункт 7 — сверка `🚨 DRIFT` в notes с `project_decisions` из контекста. Затем sub-agent ит.3 закрыл F006 (FAB в Scaffold-слот).

**Что не повторять:** при impl реализуем **финальное состояние после фичи**, не текущий код. Текущий код = источник REUSE-данных для неменяющихся виджетов, не для определения целевой структуры. Любой DRIFT в notes обязан соответствовать конкретному project_decision из контекста — выдумывать обоснования запрещено.

### `enableLexemeTranslationEdit` не копирует origin в edited

**Где:** `modules/screen/wordcard/.../mate/State.kt` — `enableLexemeTranslationEdit` + `enableLexemeDefinitionEdit`.

**Что:** при тапе на сохранённый перевод/определение → `Msg.EnterTranslationEditMode` → reducer вызывал `it.copy(translation = it.translation?.copy(isEdit = true))`. Поле `edited` оставалось пустым (или со старым value). В edit-mode `BasicTextField` показывал пустое поле вместо сохранённого значения.

**Почему:** business sub-flow когда писал extension — забыл скопировать `origin → edited` при входе в edit-mode. Никакие тесты этого не покрывали (тестировали enabled state, не реальное содержимое поля). Архитектурно `TextValueState(origin, edited, isEdit)` подразумевает что edited заполняется при `isEdit=true`, но контракт не был зафиксирован формально.

**Как закрыто:** добавлено `edited = it.translation.origin` (и аналог для definition) в `enableLexemeTranslationEdit` / `enableLexemeDefinitionEdit`.

**Что не повторять:** при работе с `TextValueState` и edit-mode паттерном — invariant: переход `isEdit=false → true` ОБЯЗАН копировать `origin` в `edited`. Желательно зафиксировать инвариант на уровне State (например, через factory-метод `TextValueState.toEdit()`).
