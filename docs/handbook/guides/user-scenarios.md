# Пользовательские сценарии (план сценариев)

Формат документа `NN_user_scenarios.md` внутри папки фичи (`docs/features/<FEATURE>/`).

**Назначение.** Человекочитаемый мост между UX и кодом: что видит и делает пользователь → какой `Msg`/`Эффект`/дельта `Стейт` за этим стоит. Дополняет, а **не дублирует** тест-контракт (`NN_test_design.md`): полные снимки состояния «все поля до→после» живут в тестах, здесь — только дельта значимых полей + ссылка на эталонный тест.

Reference-пример: `docs/features/IS481_wordcard_components/09_user_scenarios.md`.

---

## Шапка документа

В начале файла — короткая легенда формата и список разделов:

```markdown
# <FEATURE> — Пользовательские сценарии

Формат каждого сценария: **Было** → **Шаги** → **Стало** → **🔧 Тех**.
- **Было / Шаги / Стало** — человеческим языком (что видит и делает юзер).
- **🔧 Тех** — мост к коду: `Msg` · `Эффект` · `Стейт` (дельта) · `Тесты` (эталон в `NN_test_design.md`).

Полные снимки стейта «все поля до→после» — в `NN_test_design.md`, здесь только дельта + ссылка.

Разделы: **L** — <группа1> · **V** — <группа2> · **A** — <группа3>.
```

---

## Формат одного сценария

Четыре блока, всегда в одном порядке:

```markdown
## L1.1 — Короткое имя сценария

**Было**
- Предусловие: состояние экрана глазами пользователя.
- Доступность ключевых контролов.

**Шаги**
1. Действие пользователя (тап / ввод / уход).

**Стало**
- Что изменилось на экране (человеческим языком).

**🔧 Тех**
- Msg: `CreateLexeme`
- Эффект: — (или какой DatasourceEffect, что пишется в БД)
- Стейт: дельта значимых полей (`canAddLexeme=false`, `id=NOT_IN_DB`)
- Тесты: `S21`, `PendingGuardTest`  ← эталон точных значений
```

Правила:
- **Было / Шаги / Стало** — без жаргона и без имён классов. Это читает не только разработчик.
- **🔧 Тех** — единственное место, где появляются `Msg`/`Effect`/имена полей/тестов.
- `Эффект: —` если ветка чисто reducer-side (без обращения к БД).
- `Стейт:` — только изменившиеся поля, не полный снимок.
- `Тесты:` — ссылка на тест-кейс в тест-контракте (`S<N>`, имя теста/файла). Эта строка связывает сценарий с источником истины по точным значениям.

## Overlay-элементы (диалоги, снеки, меню, лоадеры)

Временные элементы — главный источник проскальзывающих багов: сценарий описывает целевой результат («лексема убрана»), а судьба самого overlay остаётся неописанной — и непокрытой тестом. Пример: IS481 BUG-2 — подтверждение удаления не закрывало модалку; L3.1 упоминал диалог в шаге, но «Стало» о нём молчало (`docs/features/IS481_bugs/bugs.md`).

### Судьба элемента в «Стало»

Если элемент упомянут в «Шагах» («подтверждаю **в диалоге**», «жму **в снеке**») — в «Стало» обязана быть строка о его судьбе: закрылся / остался / сменил содержимое. Упомянуть в действии и промолчать в результате нельзя. Строка из «Стало» попадает в `Стейт:`-дельту (например `lexemeIdPendingDelete=null`), а оттуда — в предусловие и ассерт теста.

### Матрица overlay × пути выхода

В конце файла сценариев — таблица: каждое поле стейта, управляющее показом overlay, × все пути выхода. Клетка — ссылка на сценарий либо явное «N/A — почему».

```markdown
| Поле стейта             | Открытие | Подтвердить | Отменить | Back | Внешняя причина        |
|-------------------------|----------|-------------|----------|------|------------------------|
| `lexemeIdPendingDelete` | L3.1     | L3.1        | L3.x     | A-N  | N/A — word удалить нельзя, пока диалог открыт |
```

Пустая клетка = отсутствующий сценарий (и, как правило, отсутствующий тест). Правило «судьба элемента» ловит дыру **внутри** написанного сценария, матрица — **ненаписанный** сценарий; работают только вместе.

## Секция «Дано в БД» (предусловия данных)

Сценарии стартуют с Msg-уровня — всё, что ниже (seed built-in типов, инварианты данных), редьюсер-тесты не видят в принципе. В начале файла сценариев — короткая секция с инвариантами данных, на которых стоят сценарии, и пометкой, чем каждый покрыт (instrumented-smoke / миграционные тесты). Слепая зона должна быть названа, а не подразумеваться.

```markdown
## Дано в БД
- Built-in тип `translation` существует (seed fresh-install `onCreate` + миграция M11→12) — покрытие: instrumented (`MigrationFrom11to12`).
```

Инвариант без указанного покрытия — сигнал: либо дописать instrumented-тест, либо честно пометить «не покрыто».

## Нумерация и разделы

- Сценарии группируются по доменным префиксам (для WordCard: **L** — лексема, **V** — значения, **A** — загрузка/фон/ошибки). Префиксы выбираются под фичу.
- Номер: `<Префикс><группа>.<номер>` — `L1.1`, `L1.2`, `V3`, `A17`. Группа отражает подтему (создание, удаление, undo…).
- Спец-сценарии-ловушки (негативные/граничные) допустимо помечать отдельно (`S-trap-N`), если на них есть тесты-ловушки в контракте.

## Раздел «Решения по спорным вопросам»

В конце файла — нумерованный список закрытых развилок: каждое решение одной строкой, с кодом, на который ссылаются другие файлы плана (`A12`, `P1`, `P2`…):

```markdown
# Решения по спорным вопросам (все закрыты)

1. **Rename типа (A12):** лейбл следует за переименованием…
8. **Пустое значение (P1):** пустое удаляется автоматически… Brief §2 отменён этим решением.
```

- Если решение **отменяет** пункт брифа — указать это явно («Brief §N отменён»).
- Out-of-scope/архитектурные корни выносить в `docs/Backlog.md` (раздел `## ВекторныйПиздеж`) и сослаться отсюда, **не** раздувать сценарий.

---

## Чего НЕ делать

- **Не** вставлять полные снимки состояния — это работа тест-контракта (`NN_test_design.md`); здесь дельта + ссылка.
- **Не** описывать теоретические гонки / «пользователь быстро тыкает» как сценарии — в backlog.
- **Не** плодить висячие коды решений: каждый `A/P/B/#`-код, упомянутый в сценарии, должен иметь определение (здесь в «Решениях» или в бизнес-контракте).
- **Не** смешивать человеческий язык и код в блоках Было/Шаги/Стало.

## Rules — машинно-проверяемые правила

### R-US-001 — элемент из «Шагов» имеет судьбу в «Стало»

- **Severity:** major
- **Applies to:** каждый сценарий, в «Шагах» которого фигурирует временный UI-элемент (диалог, снек, меню, лоадер).
- **Check:** для каждого упомянутого в «Шагах» overlay-элемента в «Стало» есть строка о его состоянии после шага, а соответствующее поле — в `Стейт:`-дельте. Упомянут в шаге, отсутствует в «Стало» — **нарушение**.
- **Почему:** IS481 BUG-2 — «подтверждаю в диалоге» без «диалог закрылся» → тест не проверял сброс `lexemeIdPendingDelete` → модалка висела над удалённой лексемой.
- **Fix:** дописать судьбу элемента в «Стало» и поле в `Стейт:`, тест обновить под полную дельту.

### R-US-002 — у каждого overlay-поля заполнена строка матрицы выходов

- **Severity:** major
- **Applies to:** файл сценариев экрана, стейт которого содержит overlay-поля (поля, управляющие показом диалога/снека/меню).
- **Check:** в конце файла есть матрица overlay × пути выхода; для каждого overlay-поля каждая клетка (открытие / подтвердить / отменить / back / внешняя причина) — ссылка на сценарий или явное «N/A — причина». Пустая клетка — **нарушение**.
- **Почему:** пустая клетка = ненаписанный сценарий = непокрытый путь; правило R-US-001 такие дыры не видит (оно проверяет только существующие сценарии).
- **Fix:** написать сценарий на пустую клетку либо обосновать N/A.
